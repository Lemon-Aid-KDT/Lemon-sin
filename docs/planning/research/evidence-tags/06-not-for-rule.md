# NOT_FOR_RULE: 사용자별 건강 판단 규칙화 금지

> 확인일: 2026-05-14  
> 성격: 오용 방지 목록  
> 사용 범위: 자료를 읽을 때 무엇을 구현 규칙으로 바꾸면 안 되는지 명확히 하는 안전 문서

## 핵심 판단

Lemon Aid는 의료 판단 시스템이 아니라 건강관리 보조 서비스입니다. 아래 항목은 자료에 존재하더라도 사용자별 자동 판단, 추천, 경고, 처방, 복용량 변경 지시로 바꾸면 안 됩니다.

## 규칙화 금지 항목

| 금지 대상 | 이유 | 허용되는 대체 방식 |
|-----------|------|--------------------|
| 논문 속 질환-영양소 상관관계 | 상관관계가 개인별 인과·처방을 의미하지 않음 | 배경 설명, 멘토 질문, 전문가 검토 후보 |
| 진료지침의 치료 목표 | 의료진 판단과 환자 상태가 필요함 | 팀 학습, 상담 권장 기준 검토 |
| 특정 약물의 복용·중단·변경 | 처방 영역이며 위험도가 높음 | “복용 중인 약이 있다면 전문가 상담 권장” |
| 특정 영양제 제품 추천 | 광고·약사법·건기식법 리스크 | 성분·함량·주의사항 단순 표시 |
| 질환별 영양제 복용량 추천 | 개인 상태, 복약, 검사값 검토 필요 | 공식 섭취량과 상한섭취량 대비 확인 |
| 국외 권장량의 직접 적용 | 한국 기준·표시·식생활과 다를 수 있음 | 국내 공식 기준 우선, 국외 자료는 배경 참고 |
| OCR/LLM 추론 결과의 확정 저장 | 입력 오류와 hallucination 가능성 | 사용자 미리보기, 수정, 승인 후 저장 |
| 음식 사진만으로 섭취량 확정 | 분량·조리법·국물 섭취량 오차 큼 | 추정값 표시, 사용자 입력 보완 |
| 검사값 기반 질환 판정 | 진단 행위로 오해될 수 있음 | 입력된 검사값 요약과 상담 권장 |

## 금지 표현 예시

| 피해야 할 표현 | 대체 표현 |
|----------------|-----------|
| “당뇨 위험군입니다.” | “혈당 관련 영양 관리에 주의가 필요할 수 있습니다.” |
| “이 영양제를 드세요.” | “해당 성분의 섭취량과 주의사항을 확인해 볼 수 있습니다.” |
| “고혈압을 개선합니다.” | “나트륨 섭취 상태를 확인해 볼 수 있습니다.” |
| “빈혈 위험입니다.” | “철분 섭취량이 권장량보다 낮게 계산되었습니다.” |
| “복용 중인 약과 함께 먹어도 됩니다.” | “복용 중인 약이 있다면 전문가와 상담을 권장합니다.” |
| “이 식단은 뇌졸중을 예방합니다.” | “위험요인 관리 관점에서 식단과 생활습관을 함께 확인할 수 있습니다.” |

## 자료별 주의

- `KR_OFFICIAL`: 구현 기준으로 쓸 수 있지만, 질환 치료 효과로 확장하지 않는다.
- `KR_CLINICAL_GUIDE`: 팀 학습과 전문가 검토용이며, 앱 로직에 직접 넣지 않는다.
- `KR_RESEARCH`: 한국 사용자 맥락 설명용이며, 상관관계를 개인별 규칙으로 만들지 않는다.
- `GLOBAL_REVIEW`: 최신 동향과 한계 이해용이며, 국내 기준값을 대체하지 않는다.
- `GLOBAL_TECH`: 파이프라인과 검증 설계 참고용이며, 건강 판단 생성 근거가 아니다.

## 구현 체크리스트

- [ ] 사용자 화면에 진단·치료·처방처럼 보이는 문구가 없는가?
- [ ] 알고리즘 기준값이 국내 공식 기준 또는 명시된 프로젝트 가정에서 온 것인가?
- [ ] 논문 결론이 개인별 건강 판단으로 저장되지 않는가?
- [ ] OCR/LLM 결과가 사용자 승인 전 확정 기록으로 저장되지 않는가?
- [ ] 복약·영양제 주의는 최종 판정이 아니라 상담 권장으로 표현되는가?
- [ ] 원문 미확인 자료가 DB, 알고리즘, 프롬프트에 반영되지 않았는가?

## MVP 반영

- 모든 건강 관련 출력은 “입력된 정보 기준”과 “전문가 상담 권장” 경계를 유지한다.
- 추천보다 확인, 확정보다 후보, 판단보다 설명을 우선한다.
- 의료자문위 검토 전 질환별 보정 로직은 도입하지 않는다.
