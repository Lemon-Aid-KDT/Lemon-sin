# KR_RESEARCH: 한국 사용자 맥락/배경 근거

> 확인일: 2026-05-14  
> 성격: 국내 연구보고서, 국내 논문, 학위논문, 특허, UI/UX 연구  
> 사용 범위: 한국 사용자·한식·고령자·만성질환 맥락 설명, 구현 구조 참고

## 핵심 판단

`KR_RESEARCH`는 Lemon Aid의 문제 정의와 UX 설계에 유용하지만, 연구 속 상관관계를 사용자별 건강 판단 규칙으로 바로 쓰면 안 됩니다. 국내 연구는 한국 식생활과 한식 기반 데이터 구조를 이해하는 데 주로 사용합니다.

## 사용할 수 있는 방식

- 기획: 한국 고령자, 만성질환자, 한식 식단 관리의 필요성 설명
- DB 설계: 음식명, 식품군, 영양소, 식사 유형, 후보 식단 구조 참고
- OCR/이미지: 음식명 후보를 공식 식품 DB와 매칭하는 흐름 참고
- UI/UX: 분석 결과 미리보기, 사용자 수정, 후보 교체, 대시보드 정보구조 참고
- 멘토 질문: “이 연구의 질환별 기준을 MVP에 넣어도 되는가?” 같은 검토 질문 정리

## 사용하면 안 되는 방식

- 학위논문 속 질환-영양소 상관관계를 개인별 권고 룰로 저장하지 않는다.
- 특허의 청구항 또는 상세 알고리즘을 그대로 구현하지 않는다.
- 원문 미확인 자료를 기획 근거, DB, 알고리즘, 프롬프트에 반영하지 않는다.
- “한국 노인에서 관련성이 있었다”를 “사용자에게 이 영양소가 필요하다”로 바꾸지 않는다.
- 연구보고서의 질환별 식단 추천 기준을 의료자문 없이 자동 추천으로 만들지 않는다.

## 반영 위치

| 영역 | 반영 방식 |
|------|-----------|
| DB | 식품군·식사 구성·문헌 메타데이터 참고 |
| 알고리즘 | 공식 기준과 분리된 구현 참고, 직접 규칙화 금지 |
| Agent | 배경 설명 후보, 사용자 건강 판단 근거로 사용 금지 |
| UI/UX | 후보/교체/수정/승인 흐름 참고 |
| 안전/검증 | 상관관계 오용 방지, 특허 복제 방지 |

## MVP 반영

- 음식명과 영양소 매칭은 공식 DB 기준으로 구현하고, 연구는 구조 참고만 한다.
- 사용자가 OCR 결과를 수정·승인하는 흐름을 강화한다.
- 한식의 국물, 반찬, 김치, 조리법 차이를 “확정값”이 아니라 “사용자 확인 필요”로 다룬다.

## v2 이후 검토

- 식단 후보 추천, 음식 교체 추천, 질환위험인자별 고도화는 특허·의료자문 검토 후 진행
- 학위논문 원문 확인 후 팀 학습 문서의 근거 수준 보강
- 국내 사용자 테스트 자료와 연결해 UX 근거 업데이트
