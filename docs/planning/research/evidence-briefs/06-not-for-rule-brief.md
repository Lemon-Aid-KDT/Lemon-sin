# NOT_FOR_RULE Brief

> 확인일: 2026-05-14  
> 기준 자료: 앞 5개 brief에서 실제 확인한 공식 페이지, PDF, 연구보고서, 특허, 논문 원문/초록

## 핵심 요약

자료를 실제로 읽어보면 Lemon Aid가 조심해야 할 지점이 분명하다. KDRIs와 식품 DB는 구현 기준으로 쓸 수 있지만 질환 치료 기준은 아니다. 진료지침은 의료진용 문서라서 팀 학습에는 필요하지만 앱 로직으로 자동화하면 안 된다. 국내 식단 추천 연구와 특허는 한식 식단 구조와 사용자 승인 UX에는 유용하지만 그대로 복제하면 안 된다. AI·LLM 논문은 기술 가능성과 동시에 오류, hallucination, 누적 오차, 사람 검토 필요성을 강조한다.

## 규칙화 금지 항목

- 논문 속 질환-영양소 상관관계를 개인별 권고로 바꾸지 않는다.
- 진료지침의 혈당, 혈압, 지질, 골밀도 기준으로 앱이 질환 상태를 판정하지 않는다.
- DUR 자료로 앱이 병용 가능/불가능을 최종 판정하지 않는다.
- 건강기능식품 기능성 문구를 질병 예방·치료 효과처럼 표현하지 않는다.
- 음식 사진, OCR, LLM 추정 결과를 사용자 확인 없이 최종 기록으로 저장하지 않는다.
- 국외 논문·벤치마크 기준을 한국 사용자 권장량으로 직접 적용하지 않는다.
- LLM이 영양 기준값, 질환별 보정, 복약 주의, 영양제 복용량을 생성하게 하지 않는다.

## 대체 표현

- “당뇨 위험입니다” → “혈당 관련 영양 관리에 주의가 필요할 수 있습니다.”
- “고혈압을 개선합니다” → “나트륨 섭취 상태를 확인해 볼 수 있습니다.”
- “복용해도 됩니다” → “복용 중인 약이 있다면 전문가와 상담을 권장합니다.”
- “이 식단은 치료에 좋습니다” → “입력된 식단 기준으로 영양소 구성을 확인했습니다.”

## 구현 체크포인트

모든 건강 출력은 “입력된 정보 기준”을 전제로 한다. 낮은 confidence, 복약 정보, 임신·수유, 노인, 신장·간질환 가능성은 상담 권장 또는 전문가 검토로 넘긴다. 사용자 승인 전 기록 상태는 `candidate` 또는 `draft`로 유지한다.
