> 즉석 판단 / ad-hoc 결정 금지. 모든 새 데이터는 같은 파이프라인을 거친다.

## 1. 단일 진실 (Single Source of Truth)

**파일**: `nutrition_map_enriched.json`

**키**: AI Hub 8자리 코드 (`01011001`, `04019001` 등)
- 이 코드 = 분류기가 출력하는 클래스 ID
- 모든 데이터셋의 클래스는 결국 이 코드로 통합됨

**값 스키마**:
```json
{
  "01011001": {
    "name_ko": "쌀밥",              // 한글 정식명 (canonical)
    "aliases": ["흰밥", "쌀밥"],     // 다른 데이터셋의 표기 (auto-collected)
    "kcal": 335, ...                 // 기본 영양 (AI Hub xlsx)
    "fiber_g": 1.2, ...              // MFDS 보강 필드
    "mfds_match": "쌀밥",
    "mfds_match_score": 1.0,
    "variants": [                    // MFDS 변형 (사용자 선택용)
      {"label": "기본", "name": "쌀밥", "kcal": 335, ...},
      ...
    ],
    "variant_count": 3,
    "sources": ["aihub_74", "aihub_242"]  // 어느 데이터셋들이 이 코드 학습에 기여했나
  }
}
```

## 2. 13가지 케이스 처리 규칙 (잠금)

새 (코드, 이름, 사진) 가 들어오면 아래 순서로 검사. **첫 번째 일치하는 규칙으로 처리.**

### 정확 일치류

**Case 1: 정규화 후 정확 일치**
- "김치찌개" ≈ "김치_찌개" ≈ "(김치) 찌개" ≈ "김치 찌개"
- → **기존 코드 폴더로 사진 합침** (같은 클래스로 취급)
- → aliases 에 새 표기 추가

**Case 2: 코드도 같고 이름도 같음**
- 두 데이터셋이 같은 코드 체계 사용 + 같은 음식
- → 단순 사진 추가, alias 추가 안 함

### 변형류

**Case 3: 변형 (prefix + descriptor)**
- 기존 "김치찌개" + 새 "김치찌개_꽁치"
- → 새 데이터셋에서 분류기 클래스로 들어와 있어도 **변형으로 처리**
- → 사진은 기존 코드에 합침 + variants 에 등록
- 예외: 시각적으로 명백히 다른 경우 (사람 검토)

**Case 4: 변형 (suffix)**
- 기존 "김치찌개" + 새 "참치_김치찌개"
- → Case 3 와 동일

**Case 5: 어순 다른 동일 음식**
- "김치찌개" vs "찌개김치" / "찌개_김치"
- 문자 정렬 일치하면 → **같은 음식으로 처리** (정규화 비교)

### 별칭류

**Case 6: 동의어 / 다른 표기**
- "비빔밥" ≈ "Bibimbap" ≈ "韩国拌饭"
- "Kimchi" ≈ "Kim Chi" ≈ "Kimchee"
- → aliases 에 추가, 기존 코드로 매핑

**Case 7: 한자/영문 표기**
- 한국어 정규화 후 영문 / 한자 alias 로 등록

### 충돌류 (위험!)

**Case 8: 같은 이름 다른 음식**
- "라면" (한국 인스턴트) vs "라멘" (일본) — 1글자 차이
- "마늘" (재료) vs "마늘빵" (음식)
- → **자동 매칭 안 함**. 사용자 검토 필수
- → 이름 유사도 0.85+ 인데 이미지 시각적 차이 크면 경고

**Case 9: 코드 충돌 (다른 음식, 같은 코드)**
- AI Hub #74 의 04019001 ≠ #242 의 04019001 (만약 다른 의미)
- → **새 prefix 부여** (예: #242 코드 앞에 9 붙임 → 904019001)
- → 통합 시 자동 검사

### 새 클래스류

**Case 10: 완전히 새로운 음식**
- 기존 어떤 코드에도 매칭 안 됨
- → **새 코드 부여**:
  - 한식 관련 → AI Hub #74 prefix (01~14) 안에 사용 안 된 번호
  - 디저트·서양식 → 새 prefix (예: 90xxxxxx)

**Case 11: AI Hub 영양 DB 에 있지만 코드 미매핑**
- xlsx 의 146 미매핑 이름 같은 것
- → 시각적 식별 (Florence-2) + xlsx 영양과 매칭 후 코드 할당

### 라벨 없는 경우류

**Case 12: 이름 없는 학습 데이터**
- Roboflow / Florence-2 / Gemini detection 데이터
- → **detector 만 학습용** (1-class food). 분류기 영향 없음
- → nutrition_map 에 영향 X

**Case 13: 학습 사진 없는 영양 데이터**
- MFDS 19,495 같은 영양 DB
- → 분류기 학습에 안 쓰임, **변형 라이브러리로만 사용**
- → 시각적 인식 못 함 (사진 없으니까), 사용자가 변형 선택할 때만 등장

## 3. 일관된 처리 순서 (잠금 흐름)

```
[새 데이터셋 도착]
        │
        ▼
[1] 코드·이름 추출
    - 이미지가 있나? (분류기 학습용?)
    - 클래스 라벨이 있나? (학습용?)
    - 영양 정보가 있나? (보강용?)
        │
        ▼
[2] consolidate_new_dataset.py 실행
    - 정규화 → 13 케이스 분류 → 통합 계획 생성
    - 출력: integration_plan.json
        │
        ▼
[3] 사용자 검토 (5~30분)
    - 자동 처리 (Case 1, 2, 6, 12, 13): 그대로 통과
    - 경계 (Case 3, 4, 5): 변형으로 표시, 검토
    - 충돌 (Case 8, 9): 반드시 사용자 결정
    - 새 코드 (Case 10, 11): 코드 할당 확인
        │
        ▼
[4] 승인 후 자동 적용
    - crops_aihub 폴더 통합
    - nutrition_map_enriched.json 업데이트
    - audit log 기록 (어떤 변경이 일어났는지)
        │
        ▼
[5] 클래스 audit 자동 재실행
    - 코드 갯수, 매핑 갯수, 변형 갯수 검증
    - 회귀 없는지 확인
        │
        ▼
[6] 분류기 fine-tune (선택)
    - 새 클래스 추가됐으면 → 헤드 확장 + 재학습
    - 기존 클래스 데이터만 추가됐으면 → 가벼운 fine-tune
```

## 5. 절대 위반하지 말 것 (제약)

❌ **단일 클래스를 두 코드로 쪼개기** (variants 로 처리할 것)
❌ **시각적으로 같은 음식을 다른 클래스로 학습** (분류기 혼란)
❌ **MFDS 식품코드(D101-...)를 분류기 클래스로 사용** (AI Hub 8자리만 사용)
❌ **이름 검증 없이 자동 병합** (Case 8 충돌 위험)
❌ **수동으로 nutrition_map 직접 수정** (consolidate 스크립트 통해서만)

## 6. 비상 상황 대응

| 문제 | 대처 |
|---|---|
| 통합 후 분류기 정확도 폭락 | rollback (audit log 기반) — 백업 nutrition_map 복원 |
| 같은 음식이 두 코드로 분리됨 | manual merge — `merge_codes.py 코드A 코드B` |
| 잘못된 변형 매핑 | nutrition_map 의 variants[] 에서 해당 항목 제거 |
| MFDS 매칭 안 되는 신메뉴 | 수동 영양 입력 (xlsx 양식) → enrich 재실행 |

## 8. 핵심 원칙 (절대 변경 X)

1. **AI Hub 8자리 코드 = 분류기 클래스 ID** (모든 다른 식별자는 alias)
2. **시각적 구분 가능한 것만 클래스** (구분 안 되면 변형으로)
3. **nutrition_map_enriched.json 이 단일 진실** (다른 곳 안 봄)
4. **새 데이터는 protocol 통해서만 진입** (직접 폴더 조작 금지)
5. **모든 변경은 audit log 에 기록** (rollback 가능하게)
