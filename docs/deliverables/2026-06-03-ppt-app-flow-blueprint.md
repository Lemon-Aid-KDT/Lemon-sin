# 발표 PPT 대본 — 앱 흐름(폰 중앙) + 좌우 스토리

> **레이아웃 규칙(전 슬라이드 공통)**
> - **중앙**: 휴대폰 목업 1개 = 앱 사용 흐름의 한 화면
> - **왼쪽**: "왜" — 사용자 관점 / 우리가 한 판단(고민)
> - **오른쪽**: "어떻게·얼마나" — 택한 기술 + 성능 수치(굵게)
> - 읽기 흐름: **왼쪽(왜) → 폰(결과) → 오른쪽(기술/성과)**
> - 미리캔버스 팁: 폰 목업을 정중앙 고정, 좌/우를 같은 폭(예 30%·40%·30%) 칼럼으로. 좌=부드러운 톤(아이콘+짧은 문장), 우=데이터 톤(숫자·화살표·작은 그래프).

---

## 슬라이드 0 — 표지
- **중앙(폰)**: 앱 홈 화면 / 로고
- **제목**: "사진 한 장으로 끝나는 만성질환 식단 관리"
- **부제**: 음식 인식 AI — 우리가 내린 판단과 성능 개선 여정
- (좌우 비움, 폰만 크게)

---

## 슬라이드 1 — 음식 촬영 (앱 흐름 ①)
- **중앙(폰)**: 카메라로 음식 촬영하는 화면
- **왼쪽(왜)**
  - 만성질환자는 식단 기록이 번거롭다 → **"입력은 사진 1장으로"**
  - 음식이 아니면? → 헷갈리는 결과 대신 안내가 필요
- **오른쪽(기술·성과)**
  - 객체 탐지 모델 도입 (YOLO 계열)
  - **음식 미탐지 시 "음식이 없습니다, 다시 찍어주세요"**
  - 분리 임계값 0.08 → test에서 **음식 16/16 인식 · 비음식 3/3 거부**

---

## 슬라이드 2 — 음식 인식 결과 (앱 흐름 ②) ★핵심
- **중앙(폰)**: 사진 위 **박스 + 음식 이름** 표시된 결과 화면
- **왼쪽(왜)**
  - 처음엔 "모델 크기·증강·복제" 같은 **표준 튜닝**부터 시도
  - 그런데 성능이 **0.79~0.85에서 횡보** (증강·복제는 효과 없음을 검증)
  - → **"데이터·클래스 설계가 진짜 레버"** 라고 판단, 방향 전환
- **오른쪽(기술·성과)**
  - **택소노미 재설계**(424코드 → 의미단위, 혼동군 분할)
    - 탕 0.56→**0.84**, 면 0.55→**0.92**
  - **라벨노이즈 감사**(엉뚱하게 섞인 클래스 정리)
    - 후라이드치킨 0.67→**0.90 (+0.226)**
  - 전체 mAP **0.82 → 0.89**

---

## 슬라이드 3 — Top-5 후보 / 정답 선택 (앱 흐름 ③)
- **중앙(폰)**: Top-5 후보 음식 + 사용자가 정답 고르는 화면
- **왼쪽(왜)**
  - AI도 틀릴 수 있다 → **"사용자가 바로잡게"**
  - 비슷한 음식(닭류·면류)은 사람 확인이 더 정확
- **오른쪽(기술·성과)**
  - 최신 **YOLO26s**(end-to-end) 채택 — 이전 모델 대비 **+0.019**
  - 한 음식에 박스 2개 뜨는 문제 → **중복 박스 병합(NMS 후처리)**

## 슬라이드 4 — 영양소 정보 (앱 흐름 ④) ★차별화
- **중앙(폰)**: 열량·탄단지·나트륨 + 1인분/2인분 카드
- **왼쪽(왜)**
  - 음식 이름만으론 부족 → **만성질환자에겐 "영양소"가 본질**
  - 100g만? 실제로 먹은 양 기준이 필요
- **오른쪽(기술·성과)**
  - **AIHub 영양 데이터 → 59클래스 매핑**(100g 정규화 평균)
  - **섭취량 환산**(1인분·2인분·직접입력 g)
  - 일일 권장 대비 % 자동 계산

## 슬라이드 5 — 만성질환 맞춤 피드백 (앱 흐름 ⑤)
- **중앙(폰)**: "🧂 나트륨 높음 — 혈압 관리 참고" 등 안내 화면
- **왼쪽(왜)**
  - **"필라이즈가 못하는 만성질환자 영역으로 차별화"**
  - 일반인 기준이 아닌 당뇨·고혈압 맞춤
- **오른쪽(기술·성과)**
  - 나트륨·당류 임계 경고, 일일권장 대비
  - 컴플라이언스: **진단·처방 아님**(정보 제공) — 의료법 안전선

## 슬라이드 6 — 성능 개선 여정 (요약, 폰 대신 그래프 가능)
- **중앙**: 실험 비교 맵 이미지(막대=정확도 상승) — 폰 대신 차트
- **왼쪽(왜)**
  - 좋아만 보이는 수치를 의심 → **과적합 진단**
  - "데이터가 부족한가?" 가설 검증
- **오른쪽(기술·성과)**
  - train 0.99 vs val 0.88 = **과적합 확인**
  - **데이터 증량(cap↑)** → val **+0.018**, 과적합 갭 0.117→0.097
  - 약점 클래스는 **실데이터 보강**(파일럿 진행)
  - 전체: **0.82 → 0.90**

## 슬라이드 7 — 마무리 / 향후
- **중앙(폰)**: 종합 결과 화면(5종 분석 미리보기)
- **왼쪽(왜)**: 사진 1장 → 만성질환 맞춤 관리의 입구
- **오른쪽(향후)**: 실환경 데이터 수집, 영양제·활동 통합, 의료데이터 연계
- 한 줄: **"한식을 정확히 읽어 만성질환자 맞춤 관리로 — 경쟁사가 안 하는 영역"**

## 부록 — 미리캔버스 작성 팁
- **폰 목업**: 무료 mockup 에셋 사용, 안에 streamlit 데모 캡처 넣기(실제 화면이 설득력↑)
- **좌우 균형**: 좌·우 글자 수를 비슷하게(각 3~4줄). 좌는 문장형, 우는 **숫자+화살표**(0.67→0.90)로 대비
- **색**: 좌(연한 그레이/블루=스토리), 우(그린=성과·상승), 폰은 화이트
- **일관성**: 모든 슬라이드 좌상단에 앱흐름 단계 표시(①②③…)로 진행감
- **수치는 굵게**: +0.226, 0.82→0.90 등 핵심 숫자만 강조해 한눈에
