# Lemon Aid 개발자용 기획서 작성 전 팀 검토 보고서

> 참고 문서: `01-product-overview.md`, `02-product-spec.md`, `03-frontend.md`, `04-backend-api.md`, `05-data-model.md`, `06-ai-agents.md`, `07-algorithms.md`, `08-compliance-safety.md`, `10-demo-release.md`

## 1. 보고서 목적

이 문서는 Lemon Aid의 개발자 전달용 최종 기획서를 만들기 전, 현재 `guide` 문서에 정리된 기능 흐름과 기술 구현안을 팀이 검토하고 확정하기 위한 내부 보고서다.

제품 방향과 MVP 목표는 현재 확정된 것으로 본다. 다만 프론트엔드, 백엔드, 데이터 모델, 이미지 처리, AI Agent, 병원 데이터 연동 기반, 안전/컴플라이언스 구현 방식은 AI와 함께 초안화된 내용이므로 팀 검토와 확정이 필요하다.

따라서 이 문서는 개발자에게 바로 전달하는 최종 명세서가 아니라, 최종 명세서로 전환하기 전 팀이 결정해야 할 질문을 정리하는 문서다.

## 2. 현재 확정된 기획 방향

Lemon Aid는 영양제, 음식, 복약 정보, 건강 데이터를 바탕으로 사용자에게 건강관리 참고 정보를 제공하는 AI 헬스케어 Assistant MVP다.

현재 확정된 방향은 다음과 같다.

| 구분 | 내용 |
|------|------|
| 서비스 성격 | 만성질환자와 예방 단계 사용자를 위한 AI 영양·복약 관리 보조 서비스 |
| 핵심 메시지 | 사진 한 장으로 시작해, 병원 기록과 생활 데이터를 기억하는 AI Agent |
| MVP 핵심 입력 | 영양제 라벨 사진, 음식 사진 또는 식단 입력, 복약 정보, 건강 데이터 |
| MVP 핵심 출력 | 부족 가능 영양소, 권장 섭취량 참고, 주의 성분, 체중 변화 예측, 활동 권고 |
| 장기 방향 | 향후 병원성 데이터와 생활 데이터를 연결한 개인화 건강관리 Agent |
| 안전 기준 | 진단·처방·치료 판단이 아닌 참고 정보 제공 |
| MVP 제외 | 실제 병원 데이터 연동, 의료기관 알선, 처방, 치료 효과 보장 |

팀 확인 질문:

| 질문 | 검토 이유 |
|------|-----------|
| 위 내용을 팀의 최종 제품 방향으로 확정해도 되는가? | 이후 개발자용 기획서의 최상위 기준이 된다. |
| MVP에서 반드시 보여줘야 하는 핵심 가치는 "사진 기반 분석"인가, "병원 데이터 연동 가능성"인가, 아니면 둘 모두인가? | 시연 구성과 개발 우선순위가 달라진다. |
| 만성질환자를 1차 사용자로 확정할 것인가, 예방 단계 직장인까지 같은 비중으로 다룰 것인가? | 온보딩 질문, 화면 문구, 분석 기준이 달라진다. |

## 3. 개발 대상 기능 범위

현재 `guide` 초안 기준 개발 대상 기능은 다음과 같이 정리되어 있다.

| 기능 영역 | 현재 초안 내용 | 팀이 결정해야 할 질문 | 결정 영향 |
|-----------|----------------|----------------------|-----------|
| 온보딩 | 사용자 프로필, 만성질환, 복약 정보, 건강 데이터 권한 입력 | MVP에서 필수 입력과 선택 입력을 어떻게 나눌 것인가? | 개인정보 동의, 이탈률, 분석 정확도 |
| 영양제 분석 | 영양제 라벨 촬영 후 OCR·LLM·식약처 매칭 | 영양제 분석을 MVP 1순위로 확정할 것인가? | 백엔드, OCR, 데이터 매칭 우선순위 |
| 음식/식단 분석 | 음식 사진 또는 식단 입력으로 영양소 산출 | 음식 사진 인식까지 구현할 것인가, 수동/텍스트 입력 중심으로 갈 것인가? | 이미지 AI 난이도, 시연 안정성 |
| 건강 데이터 | 걸음수, 체중, 심박 등 시계열 데이터 | 실제 HealthKit/Health Connect 연동을 할 것인가, 시연용 데이터로 대체할 것인가? | 모바일 권한, 심사, 개발 기간 |
| 복약 정보 | 사용자가 복용 중인 약 정보를 입력 | 약 이름을 자유 입력으로 받을 것인가, 샘플 선택형으로 받을 것인가? | 안전 문구, 데이터 표준화, 분석 책임 |
| AI Agent | 분석, 개인화, 챗봇, 평가 Agent 구조 | MVP에서 4개 Agent를 모두 분리할 것인가, 일부를 통합할 것인가? | 구현 복잡도, 디버깅, 일정 |
| 결과 화면 | 부족 영양소, 주의 가능성, 권장 기준, 활동 권고 표시 | 결과를 몇 개 화면으로 나눌 것인가? | UX, QA, 안전 문구 배치 |
| 응모권 UX | 사진 기록 참여 기반 응모권 | MVP 시연에 포함할 것인가? | 개발 범위, 차별화 포인트 |

개발자용 최종 기획서에는 위 기능별로 화면, 입력값, 출력값, 실패 상태, 완료 기준이 별도로 정리되어야 한다.

## 4. 이미지 처리 범위 검토

Lemon Aid의 핵심 입력은 사진이다. 따라서 이미지 처리는 단순 업로드 기능이 아니라, 분석 정확도, 개인정보, 저장 정책, 실패 대응, 시연 안정성에 직접 영향을 준다.

현재 초안에는 영양제 라벨·음식 사진 촬영, OCR, 이미지 저장, SHA-256 중복 감지, OCR 실패 시 재촬영 또는 수동 입력 흐름이 포함되어 있다. 다만 팀이 최종 확정해야 할 항목이 남아 있다.

| 검토 항목 | 팀이 결정해야 할 질문 | 선택지 또는 검토 방향 | 결정 영향 |
|-----------|----------------------|----------------------|-----------|
| 이미지 대상 | MVP에서 어떤 이미지를 처리할 것인가? | 영양제 라벨만 / 영양제+음식 / 영양제+음식+검사표·처방 관련 이미지 | 개발 난이도와 개인정보 리스크 |
| 입력 방식 | 카메라 촬영만 허용할 것인가, 갤러리 업로드도 허용할 것인가? | 카메라만 / 갤러리 포함 / 시연용 샘플 이미지 선택 포함 | 사용성, 시연 안정성 |
| 이미지 전처리 | 회전 보정, 용량 제한, 흐림 감지, 해상도 제한을 MVP에 넣을 것인가? | 최소 제한만 / 전처리 포함 / 실패 감지만 포함 | OCR 정확도, 개발 기간 |
| OCR 실패 대응 | OCR 실패 시 어떤 흐름을 제공할 것인가? | 다시 촬영 / 수동 입력 / 샘플 이미지 선택 / 일부 결과 표시 후 수정 | 사용자 이탈, QA 범위 |
| 원본 이미지 저장 | 원본 사진을 저장할 것인가? | 분석 후 삭제 / 사용자 동의 후 저장 / 시연용만 저장 | 개인정보, 스토리지, 동의 UI |
| 이미지 중복 처리 | 같은 사진을 다시 분석하지 않게 할 것인가? | SHA-256 해시 적용 / MVP 제외 / 시연 데이터만 캐시 | 비용 절감, 구현 복잡도 |
| 사용자 수정 | OCR 결과를 사용자가 수정할 수 있게 할 것인가? | 필수 제공 / 영양제만 제공 / MVP 제외 | 분석 신뢰도, UI 복잡도 |
| 이미지 민감도 | 어떤 이미지를 민감정보로 볼 것인가? | 영양제 사진은 일반정보, 처방·검사표 이미지는 민감정보 / 모든 건강 관련 이미지를 민감하게 처리 | 동의 정책, 저장 정책 |

팀 검토 포인트:

- 영양제 라벨 이미지는 MVP 핵심이므로 우선 처리 대상으로 보는 것이 자연스럽다.
- 음식 사진은 서비스 차별화에는 중요하지만, 실제 이미지 인식까지 구현하면 난이도가 크게 올라갈 수 있다.
- 처방전, 약 봉투, 검사표 이미지는 향후 가치가 크지만 개인정보와 의료정보 리스크가 높으므로 MVP에서 직접 처리할지 신중히 결정해야 한다.
- 시연 안정성을 위해 실제 촬영 흐름과 샘플 이미지 흐름을 함께 준비할 필요가 있다.

## 5. 이미지 분석을 위한 데이터 검토

이미지 분석은 사진만으로 완료되지 않는다. OCR 또는 이미지 인식 결과를 표준 데이터와 매칭해야 사용자에게 의미 있는 결과를 제공할 수 있다.

| 데이터 종류 | 현재 초안 내용 | 팀이 결정해야 할 질문 | 결정 영향 |
|-------------|----------------|----------------------|-----------|
| 영양제 기준 데이터 | 식약처 건강기능식품 원료·제품 데이터 활용 | 제품명 기준 매칭을 할 것인가, 성분명 기준 매칭을 할 것인가? | 매칭 정확도, 데이터 정제 범위 |
| 성분명 표준화 | OCR 텍스트를 LLM으로 구조화 | 비타민D, Vitamin D, cholecalciferol 같은 표현을 어떻게 표준화할 것인가? | 데이터 모델, QA 케이스 |
| 함량·단위 변환 | mg, μg, IU 등 단위가 섞일 수 있음 | 단위 변환 규칙을 MVP에 포함할 것인가? | 분석 정확도, 안전 리스크 |
| 음식 영양소 데이터 | 식품영양성분 Open API, 농진청 데이터 언급 | 음식 사진 분석 결과를 어떤 영양소 DB와 연결할 것인가? | API 연동, 데이터 출처 표시 |
| 사용자 보정 데이터 | 사용자가 OCR 결과를 수정할 수 있음 | 사용자가 수정한 값을 저장해 다음 분석에 재사용할 것인가? | 개인화, 개인정보, 캐싱 |
| 캐시 데이터 | OCR 결과 캐싱, 동일 사진 중복 방지 | 캐싱을 MVP에 넣을 것인가, 데모 안정화용으로만 둘 것인가? | 비용, 성능, 구현 범위 |
| 데이터 출처 | 앱 내 출처 표시 필요 | 결과 화면마다 출처를 표시할 것인가, 별도 출처 페이지로 둘 것인가? | 신뢰도, 컴플라이언스 |
| 데이터 품질 | OCR과 LLM 결과는 오류 가능 | 낮은 신뢰도 결과를 사용자에게 어떻게 표시할 것인가? | 안전 문구, UX |

팀 검토 포인트:

- 영양제 분석은 "OCR 텍스트 추출"보다 "표준 데이터와의 매칭"이 핵심이다.
- 데이터 출처가 불명확하면 결과 신뢰도와 컴플라이언스 모두 약해진다.
- 단위 변환과 성분 표준화는 건강 관련 결과에 직접 영향을 주므로, MVP에서도 최소 기준이 필요하다.

## 6. 향후 병원 데이터 연동 기반 검토

본 MVP는 실제 병원 데이터 연동을 수행하지 않는다. 다만 Lemon Aid의 장기 방향이 병원성 데이터 기반 개인화 Agent이므로, 향후 병원 데이터가 들어올 수 있는 구조와 시연용 대체 데이터는 검토해야 한다.

이 섹션의 목적은 실제 연동을 구현하자는 것이 아니라, 개발자용 최종 기획서에서 병원 데이터가 들어올 자리를 어떻게 표현할지 정하는 것이다.

| 검토 항목 | 팀이 결정해야 할 질문 | 선택지 또는 검토 방향 | 결정 영향 |
|-----------|----------------------|----------------------|-----------|
| 병원 데이터 범위 | 향후 어떤 병원 데이터를 가져온다고 가정할 것인가? | 검진 결과 / 검사 수치 / 진단명 태그 / 복약 정보 / 진료 요약 | 데이터 모델, 개인화 기준 |
| MVP 대체 데이터 | 실제 병원 데이터 대신 무엇을 사용할 것인가? | 가상 프로필 / 공개 데이터 / Kaggle 시연 데이터 / 팀이 만든 샘플 데이터 | 시연 신뢰도, 윤리 리스크 |
| 데이터 형식 | 향후 표준 연동을 고려할 것인가? | 내부 JSON 우선 / FHIR KR Core 참고 / LDB 인터페이스 설계만 작성 | 확장성, 개발 범위 |
| 사용자 동의 | 병원 데이터 연동 전 어떤 동의가 필요한가? | 시연용 고지 / 항목별 민감정보 동의 / 향후 연동 동의 화면 목업 | 법적 안전성, UX |
| 민감정보 저장 | 질환, 복약, 검사값을 저장할 것인가? | 시연 프로필에만 포함 / DB 저장 / 세션 처리 | 개인정보 리스크, 백엔드 설계 |
| 개인화 기준 | 병원 데이터가 들어오면 분석 결과가 어떻게 달라지는가? | 질환별 주의 영양소 / 복약 주의 가능성 / 활동 권고 가중치 | Agent 설계, 알고리즘 |
| 시연 표현 | 실제 연동처럼 보이지 않게 어떻게 표시할 것인가? | "시연용 데이터" 배지 / "향후 연동 가정" 문구 / 발표 자료에서만 설명 | 과장 표현 방지 |
| LDB/FHIR 연결성 | 개발자용 문서에 어느 수준까지 포함할 것인가? | 개념 설명만 / 인터페이스 예시 / 실제 API 스펙 초안 | 산출물 범위 |

팀 검토 포인트:

- 실제 병원 데이터를 가져오지 않으면서도, 향후 연동 가능성을 보여주는 구조가 필요하다.
- 시연용 데이터는 실제 환자 데이터처럼 보이게 하면 안 된다.
- 검사값, 질환 태그, 복약 정보는 민감정보이므로 저장 여부와 동의 흐름을 먼저 확정해야 한다.
- "병원 기록을 기억하는 Agent"라는 메시지는 강점이지만, MVP에서는 "향후 연동을 가정한 참조 구조"로 표현하는 것이 안전하다.

## 7. AI Agent 및 기술 구현 질문

현재 초안은 4개 Agent 구조와 OCR, LLM Tool Use, Pydantic 검증, 의료 표현 필터를 포함한다. 이 구조는 서비스 방향을 잘 보여주지만 MVP 구현 난이도가 높을 수 있으므로 팀 확정이 필요하다.

| 검토 항목 | 현재 초안 내용 | 팀이 결정해야 할 질문 | 결정 영향 |
|-----------|----------------|----------------------|-----------|
| Agent 구조 | 분석, 개인화, 챗봇, 평가 Agent | 4개 Agent를 모두 분리할 것인가, MVP에서는 하나의 분석 흐름으로 합칠 것인가? | 구현 복잡도, 발표 차별화 |
| LLM 역할 | OCR 텍스트 구조화, 개인화 판단, 챗봇 응답 | LLM이 판단하는 범위와 규칙 기반으로 처리할 범위를 어떻게 나눌 것인가? | 안전성, 비용, 재현성 |
| OCR Provider | Google Cloud Vision, CLOVA 폴백 | 실제 외부 OCR API를 사용할 것인가, 데모용 OCR 결과를 준비할 것인가? | 비용, API 키, 시연 안정성 |
| 백엔드 API | FastAPI 기반 API 목록 | 실제 백엔드를 어느 수준까지 구현할 것인가? | 팀 작업 분담, QA |
| 캐싱 | Redis OCR 캐시, KDRIs 캐시 | MVP에 캐시를 구현할 것인가, 후속 개선으로 둘 것인가? | 성능, 비용, 일정 |
| 비동기 처리 | 여러 이미지 OCR/LLM 병렬 처리 | 병렬 처리를 MVP에 넣을 것인가? | 안정성, 코드 복잡도 |
| 오류 처리 | OCR 실패, LLM 실패, 의료 표현 차단 | 실패 상태별 화면과 API 응답을 얼마나 구체화할 것인가? | QA 범위, 사용자 신뢰 |
| 비용 제한 | OCR/LLM 호출 비용 산정 | 호출량 제한, 샘플 데이터, 캐시 중 무엇으로 비용을 통제할 것인가? | 운영 비용, 데모 안정성 |

팀 검토 포인트:

- Agent 구조는 발표 차별화에 유리하지만, 개발 단계에서는 단순화가 필요할 수 있다.
- 건강 관련 결과는 LLM 단독 판단보다 규칙, 출처, 검증 흐름을 함께 보여주는 것이 안전하다.
- 기술적으로 멋진 구조보다 8주 MVP에서 안정적으로 시연 가능한 구조가 우선이다.

## 8. 헬스케어 안전/개인정보 검토 질문

Lemon Aid는 건강 정보, 복약 정보, 영양제, 병원 데이터 연동 가능성을 다루므로 고위험 도메인으로 취급해야 한다. 개발자용 최종 기획서에는 안전 문구와 개인정보 처리 기준이 기능 명세와 함께 들어가야 한다.

| 검토 항목 | 팀이 결정해야 할 질문 | 결정 영향 |
|-----------|----------------------|-----------|
| 의료 표현 제한 | "진단", "처방", "치료", "완치", "질환입니다" 같은 표현을 어디서 차단할 것인가? | 프롬프트, 필터, QA 기준 |
| 결과 문구 | 결과를 "주의 가능성", "권장량 대비", "전문가 상담 권장" 형태로 통일할 것인가? | 사용자 신뢰, 법적 안전성 |
| 면책 문구 | 모든 결과 화면에 표준 고지를 넣을 것인가, 일부 화면에만 넣을 것인가? | UI, 컴플라이언스 |
| 민감정보 동의 | 만성질환, 복약, 검진기록, 걸음수, 심박수를 항목별 동의로 받을 것인가? | 온보딩, 저장 정책 |
| 사진 데이터 | 식단/영양제 사진을 일반정보로 볼 것인가, 건강 관련 이미지로 더 엄격히 볼 것인가? | 저장, 삭제, 동의 |
| 데이터 삭제 | 사용자가 삭제하거나 탈퇴할 때 어떤 데이터까지 삭제할 것인가? | 백엔드 정책, 개인정보처리방침 |
| 출처 표시 | KDRIs, 식약처, 농진청, AI 분석 출처를 어디에 표시할 것인가? | 화면 설계, 신뢰도 |
| 위급 상황 | 위험 신호 또는 심각 증상 표현이 들어오면 어떻게 안내할 것인가? | 챗봇 안전 정책 |

팀 검토 포인트:

- 결과는 진단이나 처방처럼 표현하지 않는다.
- 입력이 불완전하거나 OCR/LLM 추출이 틀릴 수 있을 때는 불확실성을 표시한다.
- 특정 영양제, 성분, 약이 누구에게나 안전하다고 단정하지 않는다.
- 복약 또는 질환 정보가 포함되는 순간 민감정보 동의와 저장 정책이 필요하다.
