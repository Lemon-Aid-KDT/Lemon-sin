## 1. 보고서 목적

Lemon Aid는 음식과 영양제 분석을 출발점으로, 만성질환자와 예방 관리 사용자가 자신의 섭취·활동·복약 맥락을 이해하도록 돕는 AI 건강관리 보조 서비스다.

이번 보고서는 서비스 방향을 다시 설명하기보다, 현재 기획서에 정리된 기술 선택이 8주 MVP와 향후 레몬헬스케어 서비스 확장 관점에서 적절한지 확인받기 위한 자료다.

## 2. 전제와 MVP 범위

| 구분 | 현재 기준 |
|------|-----------|
| 서비스 성격 | 진단·처방·치료가 아닌 건강관리 참고 서비스 |
| MVP 입력 | 영양제 라벨, 음식 사진 또는 텍스트, 사용자 프로필, 활동 데이터 |
| MVP 출력 | 부족 영양소, 권장 섭취량, 체중 예측, 활동 권고, 목적별 분석 |
| 병원 데이터 | 실제 연동은 제외하고 mock/Kaggle/수동 입력으로 대체 |
| 후속 확장 | LDB/FHIR 기반 검사값, 복약, 진단명 태그 연동 가능성 검토 |

## 3. 과제 해결 기술 흐름

1. 사용자가 영양제 라벨 또는 음식 사진을 입력한다.
2. OCR과 구조화 로직으로 제품명, 성분, 함량, 음식명, 섭취량 후보를 추출한다.
3. 식약처, 농진청, KDRIs 등 공식 데이터와 매칭한다.
4. 음식과 영양제 섭취량을 영양소 단위로 합산한다.
5. 사용자 프로필, 활동 데이터, 시연용 질환·복약 정보를 참고해 5종 결과를 만든다.
6. 챗봇은 결과 설명과 알림 등록을 보조하되, 사용자의 확인 후 저장한다.

## 4. 기술 스택 요약

| 영역 | 선택 기술 | 확인받고 싶은 점 |
|------|-----------|------------------|
| 모바일 | Flutter, Riverpod, Dio/Retrofit, camera, health 패키지 | 건강의신 또는 후속 앱과 연결할 때 Flutter 참조 앱 방향이 적절한가 |
| 백엔드 | Python, FastAPI, Pydantic, SQLAlchemy, JWT | 분석 중심 API 경계가 8주 MVP에 적절한가 |
| 데이터 | PostgreSQL, TimescaleDB, Redis, S3 호환 스토리지 | 민감정보 저장 범위와 삭제 정책을 어느 수준으로 둘 것인가 |
| OCR/AI | Google Vision, CLOVA OCR, Claude, OpenAI 백업 | 분석은 알고리즘으로 처리하고 3개 Agent가 보조 역할을 담당하는 방향이 맞는가 |
| 연동 | HealthKit, Health Connect, 향후 LDB/FHIR | 실제 병원 연동 전 어떤 인터페이스를 설계해야 하는가 |

## 5. CTO 확인 질문

1. MVP에서 영양제·음식 분석을 모두 보여주는 것이 적절한가, 영양제 분석을 더 강하게 가져가는 것이 좋은가?
2. Flutter 기반 독립형 참조 앱이 레몬헬스케어 후속 적용 관점에서 적절한가?
3. FastAPI 기반 분석 API가 사진 분석, 섭취량 합산, 5종 출력, 챗봇 액션을 관리하는 범위가 적절한가?
4. 만성질환, 복약, 검사값, 사진 데이터 중 MVP에서 실제 저장해도 되는 항목은 무엇인가?
5. 실제 병원 데이터가 없을 때 mock/Kaggle/수동 입력 중 어떤 대체 데이터가 가장 설득력 있는가?
6. 향후 LDB/FHIR 연동을 고려하면 가장 먼저 연결할 데이터는 검사값, 복약, 진단명 태그 중 무엇인가?
7. LLM을 건강 판단자가 아니라 OCR 구조화와 설명 보조자로 제한하는 원칙이 충분한가?
8. 의료 표현 필터와 사용자 미리보기 승인 흐름이 안전장치로 충분한가?
9. CTO 관점에서 시연 전 반드시 줄여야 할 기술 리스크는 무엇인가?

## 6. 결론

이 보고서의 목표는 기술을 더 많이 추가하는 것이 아니라, 현재 기획된 기술 선택을 헬스케어 서비스 관점에서 검증받는 것이다.

확인받은 내용은 개발자용 기획서, API 설계, DB 저장 범위, 안전 문구, 시연 우선순위에 반영한다.
