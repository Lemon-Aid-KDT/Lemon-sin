Slide 1 · 설계 배경
왜 Multi-Agent
구조인가?
❌ 하나의 AI가 다 하면
🔴의료 경계 판단과 답변 생성이 뒤섞여 오류 원인을 추적하기 어렵다
🔴안전 정책을 부분만 바꾸려 해도 전체 로직을 수정해야 한다
🔴메모리, 검수 근거, LLM 생성이 한 곳에 뭉쳐 유지보수가 힘들다
✅ 역할을 분리하면
🟢각 Agent가 한 가지 책임만 가져 오류가 생긴 지점을 바로 특정할 수 있다
🟢의료 안전 정책은 Safety Agent만 수정하면 전체에 반영된다
🟢LLM은 표현만, 판단은 Agent — 역할이 명확하다
Slide 2 · Agent 구성
4개 Agent 역할
🎯 Orchestrator Agent 요청 분류 → 하위 Agent 호출 → 결과 조합 → 최종 응답 반환
🧠
Context & Memory Agent
사용자 메모리 4종 + 앱 데이터 + 검수 근거를 수집해 컨텍스트 조립
📋
Planning Agent
컨텍스트 기반으로 AnswerPlan / AnalysisPlan 구조 설계
⚠️
Safety & Boundary Agent
의료 경계 판단 — 진단·처방 결정은 여기서 차단, unknown은 backlog 기록
🖨️
Rendering Agent
Plan → LLM으로 자연스러운 한국어 변환 + sources[] 첨부
Slide 3 · 요청 흐름
요청이 흘러가는 방식
1
📱 사용자 요청
채팅 질문 / 분석 요청 / 체크리스트 요청 입력
2
🎯 Orchestrator 분류
채팅 / 분석 / 의료 boundary 중 어떤 유형인지 판단하고 하위 Agent 호출
3
🧠 Context Agent — 컨텍스트 수집
관련 메모리(profile / behavior / conversation / safety) + 앱 데이터 + 검수 근거 선별
4
📋 Planning Agent — 답변 구조 설계
AnswerPlan 또는 AnalysisPlan 객체 생성 — LLM이 아닌 Agent가 구조 결정
5
⚠️ Safety Agent — 의료 경계 확인
진단·처방·복용량 결정 여부 검사 → normal / boundary / unknown 판정 후 Rendering으로 전달
6
🖨️ Rendering Agent — 최종 답변 생성
LLM이 Plan을 자연스러운 한국어로 변환 + sources[] 첨부 → Orchestrator에게 반환
7
🔒 사용자 확인 후 앱 반영
체크리스트 저장·알림 등록·기록 수정 — 사용자가 승인한 것만 실행, Memory 갱신
Slide 4 · 핵심 원칙
두 가지 설계 원칙
🤖
LLM은 판단자가 아니라 표현 보조자입니다.
의료 경계 판단, 답변 구조, 우선순위 결정은 Agent가 합니다.
LLM은 그 결과를 자연스러운 한국어로 바꾸는 역할만 담당합니다.
🔒
사용자가 확인하지 않은 행동은 절대 자동 실행하지 않습니다.
체크리스트 추가, 알림 등록, 공식 기록 저장 — 모든 앱 데이터 변경은
반드시 사용자 승인 후에만 이루어집니다.

📄 발표 대본

⏱ Slide 1 — 약 20초

저희 Lemon Aid Agent는 개인화 건강관리 오케스트레이터입니다.

처음에는 하나의 AI가 모든 걸 처리하는 방식을 생각했는데요,
문제는 의료 경계 판단, 메모리 관리, 답변 생성이 섞이면
어디서 오류가 났는지 추적하기 어렵고,
의료 안전 정책을 일관되게 유지하기 힘들다는 점이었습니다.

그래서 각 책임을 전문 Agent로 분리했습니다.

⏱ Slide 2 — 약 30초

구조는 Orchestrator 아래 4개 하위 Agent로 이루어져 있습니다.

첫 번째, Context & Memory Agent는
사용자의 개인화 메모리와 앱 데이터를 수집합니다.

두 번째, Planning Agent는
수집된 정보를 바탕으로 AnswerPlan을 만듭니다.

세 번째, Safety & Boundary Agent는
의료적으로 위험한 결정을 여기서 차단합니다.

네 번째, Rendering Agent는
Plan을 자연스러운 한국어 답변으로 변환합니다.

⏱ Slide 3 — 약 30초

요청이 들어오면 Orchestrator가 분류합니다.
채팅인지, 분석인지, 의료 boundary인지 판단한 뒤

Context Agent가 관련 메모리와 검수 근거를 모아오고,
Planning Agent가 답변 구조를 설계합니다.

Safety Agent가 의료 경계를 확인한 후,
Rendering Agent가 LLM을 사용해 최종 답변을 생성합니다.

단, 모든 앱 데이터 변경은 사용자 확인 후에만 실행됩니다.

⏱ Slide 4 — 약 10초

핵심은 두 가지입니다.

LLM은 판단자가 아니라 표현 보조자이고,

사용자가 확인하지 않은 행동은 절대 자동 실행하지 않습니다.
