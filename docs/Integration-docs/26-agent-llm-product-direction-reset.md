## 1. 왜 재정리하는가

Agent/LLM 설계는 빠르게 구현되면서 "근거 기반 챗봇", "앱 컨텍스트 기반 agent",
"LLM/RAG 고도화"가 여러 문서에 나뉘어 정리됐다. 그 결과 원래 제품 의도와 현재 구현
용어가 부분적으로 섞였다.

이 문서는 새 구현을 바로 시작하기 위한 문서가 아니다. 먼저 Lemon Aid가 만들려는
Agent와 LLM의 정체성, 학습 방식, 메모리, 분석 점수, 체크리스트, 의료 지식 경계를 다시
하나의 기준으로 고정한다.

## 2. 한 줄 방향

Lemon Aid Agent는 앱의 모든 데이터를 마음대로 바꾸는 자동 실행자가 아니라, 음식,
영양제, 프로필, 복약, 생활습관, 대화 맥락을 모아 무엇을 분석해야 하는지, 어떤 답변과
실천안을 줄지, 어떤 앱 액션을 제안할지 결정하는 개인화 건강관리 오케스트레이터다.

실제 저장, 알림 등록, 분석 실행, 체크리스트 추가 같은 행동은 사용자 확인 후 실행한다.

## 3. Agent의 정체성

Agent는 다음 책임을 가진다.

- 음식, 영양제, 사용자 프로필, 복약, 생활습관, 대화 메모리를 읽는다.
- 현재 질문이나 분석 요청에 필요한 컨텍스트를 고른다.
- 앱 기록과 검수 근거를 바탕으로 `AnswerPlan` 또는 `AnalysisPlan`을 만든다.
- 사용자가 이해할 수 있는 답변, 분석 카드, 실천 체크리스트, CTA를 제안한다.
- 사용자가 승인한 행동만 앱 데이터에 반영한다.
- 사용자의 실천 결과와 반복 패턴을 개인화 메모리에 반영한다.

Agent는 다음 책임을 갖지 않는다.

- 사용자 몰래 공식 프로필, 복약, 질환, 알림, 체크리스트를 수정하지 않는다.
- 질병을 진단하거나 치료 여부를 결정하지 않는다.
- 약 시작, 중단, 증량, 감량, 개인 복용량을 결정하지 않는다.
- 검수되지 않은 의료 지식을 LLM 일반 지식으로 채워 말하지 않는다.

## 4. 챗봇의 정체성

Lemon Aid 챗봇은 범용 AI 비서가 아니라 건강관리 동반자다.

인사, 공감, 부담을 줄이는 짧은 대화는 가능하다. 하지만 중심 역할은 음식, 영양제,
복약 주의, 생활습관, 만성질환 관리, 앱 기록 기반 분석과 실천 코칭이다.

건강과 무관한 질문은 길게 확장하지 않고, Lemon Aid가 도울 수 있는 건강관리 맥락으로
부드럽게 돌린다.

## 5. LLM의 역할

LLM은 판단자가 아니라 표현과 구조화 보조자다.

LLM이 할 수 있는 일:

- 검수 근거와 앱 컨텍스트로 만들어진 plan을 자연스러운 한국어 답변으로 바꾼다.
- `AnswerPlan`, section JSON, 요약 문장 같은 구조화 출력을 생성한다.
- OCR 또는 라벨 텍스트를 정해진 스키마로 구조화하는 보조 역할을 할 수 있다.
- 사용자의 부담을 낮추는 말투와 설명 흐름을 만든다.

LLM이 하면 안 되는 일:

- 검수 근거에 없는 의료 사실을 만든다.
- 개인의 진단, 치료 여부, 검사수치 판단, 복용량, 병용 가능 여부를 결정한다.
- raw OCR, raw chat, raw prompt, internal trace를 근거처럼 사용한다.
- RAG 검색 결과를 검수 없이 그대로 사용자 답변에 넣는다.

## 6. 학습의 두 층

Lemon Aid의 학습은 두 층으로 나눈다.

### 6.1 사용자별 학습

1차 학습은 사용자별 개인화 메모리와 실천 기록 갱신이다.

Agent는 사용자가 확인한 음식, 영양제, 복약, 프로필, 채팅 신호, 체크리스트 수행 결과,
분석 결과를 요약 기억하고 다음 분석, 챗봇 답변, 체크리스트, 앱 액션 제안에 반영한다.

예시:

- 아침 식사를 자주 거른다.
- 라면, 찌개, 국물류가 반복되어 나트륨 축이 자주 올라온다.
- 혈압약 복용을 채팅에서 언급했다.
- 저녁보다 아침 체크리스트 수행률이 높다.
- 두부는 잘 먹고 유제품은 싫어한다.
- 영양제 병용 질문을 반복해서 묻는다.
- 너무 일반적인 답변보다 구체적인 음식/행동 후보를 선호한다.

### 6.2 서비스 품질 학습

2차 학습은 서비스 전체 품질을 높이기 위한 장기 단계다.

충분한 사용자 동의, 비식별 데이터, 검수, 평가 체계가 쌓인 뒤 OCR, 음식 인식, 질문
분류, 추천 랭킹, 답변 품질 평가 모델을 개선한다.

LLM fine-tuning은 의료 지식 주입용이 아니라 형식, 말투, 요약 품질 개선용 장기 후보로
제한한다.

## 7. 모델 학습, 개인화 메모리, deterministic 정책 업데이트의 차이

| 구분 | 무엇이 바뀌나 | 대상 | 예시 | 역할 |
| --- | --- | --- | --- | --- |
| 개인화 메모리 | 사용자별 요약, 패턴, 선호, 주의 맥락 | 한 사용자 | 아침 결식, 나트륨 반복, 선호 음식, 체크리스트 성공률 | 개인별 맞춤화 |
| deterministic 정책 업데이트 | 검수된 규칙, 알고리즘, 안전 정책 | 모든 사용자 또는 조건군 | KDRIs, 복약 boundary, unknown 정책, 분석 최소 조건 | 안전하고 일관된 판단 |
| 모델 학습 | AI 모델 파라미터 또는 별도 추천 모델 | 서비스 전체 | OCR 모델 개선, 음식 인식 개선, 질문 분류, 추천 랭킹 | 장기 품질 개선 |

MVP는 개인화 메모리와 deterministic 정책 업데이트를 우선한다. 모델 학습은 서비스가
더 완벽해지기 위한 장기 고도화 단계다.

## 8. 메모리 구조

Lemon Aid는 raw 저장소와 agent memory를 분리한다.

### 8.1 Raw 저장소

- `raw_chat_archive`: 원문 대화 저장소. 보관 기간, 동의, 삭제, 개인정보 정책이 필요하다.
- `raw_prompt_log`: 내부 감사와 디버깅용 로그. 개인화 메모리로 사용하지 않는다.

raw chat은 저장할 수 있지만, agent가 매번 원문 전체를 기억처럼 쓰지는 않는다. 대화가
길어지면 핵심만 `conversation_memory`로 요약한다.

raw prompt는 사용자 개인화 메모리가 아니다. 내부 정책, 시스템 프롬프트, 도구 호출,
provider payload가 섞일 수 있으므로 감사와 디버깅 목적으로만 제한한다.

### 8.2 Agent memory

| 메모리 | 의미 | 사용 방식 |
| --- | --- | --- |
| `profile_memory` | 키, 몸무게, 나이, 성별, 질환, 복약, 운동, 흡연, 음주, 선호/회피 같은 사용자 프로필 | 앱 공식 기록과 채팅 언급을 신뢰도와 함께 구분 |
| `behavior_memory` | 음식 패턴, 영양제 체크율, 체크리스트 수행률, 알림 반응, 반복 실패/성공 패턴 | 체크리스트 난이도, 수, 시간대, 표현 방식 조정 |
| `conversation_memory` | 최근 대화 요약, 자주 묻는 주제, 말투/답변 길이 선호, 관심사 | rolling summary로 갱신하고 필요할 때만 prompt에 주입 |
| `safety_memory` | 복약, 질환, 상호작용, 응급, 검사수치 관련 주의 맥락 | 답변 전 boundary와 caution 판단에 사용 |

### 8.3 메모리 압축 원칙

한 사용자에 대해 DB에는 많은 기록을 저장할 수 있다. 하지만 매 답변에 모든 기록을 넣으면
안 된다.

답변 시에는 아래만 선택적으로 사용한다.

- 최근 대화 일부
- `conversation_memory` 요약
- 질문과 관련 있는 `profile_memory`
- 질문과 관련 있는 `behavior_memory`
- 질문과 관련 있는 `safety_memory`
- 현재 앱 데이터 snapshot
- reviewed evidence

## 9. 채팅에서 나온 건강 정보의 경계

채팅에서 나온 건강 정보는 확인 버튼 없이도 memory에는 남길 수 있다. 단, 공식 앱 기록을
자동 수정하지 않는다.

예를 들어 사용자가 "혈압약 먹고 있어"라고 말하면 `safety_memory`에는 "사용자가 혈압약
복용을 언급함"으로 저장하고 다음 답변에서 주의 맥락으로 사용한다.

하지만 `user_medications` 공식 DB에 확정 복약 정보로 등록하거나 분석에서 강하게
사용하려면 별도 등록 또는 확인 흐름을 제안한다.

## 10. OCR preview와 사진 분석 후보

OCR preview와 사진 분석 후보는 memory에 넣지 않는다.

의미:

- OCR preview: 사진을 찍은 뒤 앱이 "아마 이 제품/음식 같다"고 추정한 상태
- 사진 분석 후보: 사용자가 아직 맞다고 저장하지 않은 음식, 제품, 성분 후보

이 정보는 사용자가 확정 저장하기 전까지 건강 분석 근거가 아니다. 사용자가 확정한 음식과
영양제만 confirmed 기록으로 들어간다.

## 11. 음식/영양제 알고리즘과 agent의 책임 분리

음식과 영양제의 인식, DB 매칭, 단위 변환, 영양성분/함량 계산은 deterministic 분석
파이프라인이 담당한다.

Agent는 그 결과를 사용자 프로필, 복약/질환, 생활습관, 개인화 메모리, 대화 맥락과 합쳐
해석하고, 우선순위와 실천 계획을 만든다.

LLM은 OCR/라벨 텍스트 구조화 보조에는 쓸 수 있지만, 최종 칼로리, 영양성분, 함량 판단은
DB와 알고리즘 결과를 기준으로 한다.

## 12. 분석 탭과 점수 정의

Lemon Aid 분석 탭은 `오늘 분석`과 `스마트 분석`을 가진다.

### 12.1 오늘 분석

이름: `오늘 현재 분석 점수`

오늘 등록된 음식, 오늘 체크된 영양제, 오늘 체크리스트, 오늘 대화 맥락을 기준으로 현재
식사, 영양제, 체크리스트 관리 상태를 보여주는 점수다.

오늘 분석은 하루 최종 건강 평가가 아니라, 현재까지 등록된 기록 기준 생활관리 상태다.

### 12.2 스마트 분석

이름: `스마트 생활관리 점수`

누적 기록, 실천 패턴, 반복 영양축, 체크리스트 수행률, 대화 메모리를 바탕으로 장기 관리
습관의 안정도를 보여주는 점수다.

스마트 생활관리 점수는 질병 위험 예측이나 건강 상태 판정이 아니다. 점수가 오른다는 것은
질병 위험이 낮아졌다는 뜻이 아니라, 앱에 기록된 기준으로 사용자가 더 일관되게 관리하고
있다는 뜻이다.

금지 표현:

- 건강이 좋아졌습니다.
- 질병 위험이 낮아졌습니다.
- 혈압/당뇨가 개선되고 있습니다.
- 이 점수면 안전합니다.

권장 표현:

- 기록 기준으로 관리 흐름이 좋아지고 있습니다.
- 최근 나트륨 높은 식사가 줄고 체크리스트 수행률이 올라갔습니다.
- 이번 주는 아침 결식 패턴이 줄었습니다.
- 복약/영양제 주의 질문이 반복되어 확인 항목을 유지합니다.

## 13. 체크리스트 생성과 학습

Agent는 체크리스트 후보를 자동 생성할 수 있다. 하지만 사용자의 일일 체크리스트에 실제
추가하거나 알림을 등록하기 전에는 사용자 확인을 받는다.

기본 원칙:

- 기본 모드: 우선순위 높은 체크리스트 1~3개를 제안한다.
- 확장 모드: 사용자가 더 원하면 카테고리별 추가 후보를 제공한다.
- 저장 원칙: 사용자가 선택한 항목만 저장한다.
- 학습 원칙: 수행률, 거절 항목, 반복 실패, 선호 시간대, 난이도를 `behavior_memory`에 반영한다.
- 안전 원칙: 의료/복약 관련 체크리스트는 복용, 중단, 증량 지시가 아니라 확인, 기록, 전문가 문의 준비로 제한한다.

확장 모드 예시:

- 식사: 저녁 국물 절반 남기기, 단백질 반찬 추가하기
- 영양제: 오늘 복용 여부 체크하기, 제품 라벨 성분 확인하기
- 활동: 10분 걷기, 엘리베이터 대신 계단 1층
- 기록: 저녁 식사 사진 저장하기
- 복약 주의: 새 영양제 시작 전 복용 중인 약과 함께 확인하기

## 14. 의료 지식, RAG, source governance

의료, 영양, 복약 지식은 LLM의 일반 지식에 맡기지 않고 검수된 source/evidence에서 가져온다.

RAG는 검수 evidence를 찾는 검색 계층이다. 검색 결과는 바로 prompt에 넣지 않고
`AnswerCard` 같은 내부 답변 프레임으로 정규화한다.

검수된 지식이 없으면 모른다고 말하거나 boundary로 닫는다. LLM은 그 프레임을 사용자
맥락에 맞게 설명할 뿐, 새로운 의료 사실이나 병용 가능 여부를 만들지 않는다.

초기에는 답변 범위가 좁아 보일 수 있다. 이 문제는 LLM 일반 지식으로 채우는 방식이 아니라
아래 운영 루프로 해결한다.

```text
unknown topic
-> safe metadata backlog
-> 공식 source 또는 검수 자료 확인
-> evidence / boundary / allowed wording / blocked wording 등록
-> AnswerCard 정규화 테스트
-> golden test 추가
-> 다음부터 답변 가능
```

## 15. 의료 boundary: 답변 금지가 아니라 결정 금지

Lemon Aid의 의료 boundary는 답변 금지가 아니라 결정 금지다.

응급, 복약, 검사수치, 병용 질문에서도 reviewed medical evidence와 사용자 메모리를 바탕으로
가능한 위험 범주, 일반 원리, 확인할 정보, 상담 준비 질문, 낮은 위험 행동은 설명한다.

하지만 아래는 하지 않는다.

- 진단명 확정
- 치료 여부 결정
- 약 시작, 중단, 증량, 감량 결정
- 개인 복용량 결정
- 병용 가능/불가 단정
- 응급 여부 확정

### 15.1 응급/위험 질문

예: "가슴이 아프고 숨이 차."

Agent가 할 수 있는 일:

- 사용자 메모리에서 고혈압, 흡연, 심혈관 질환, 복약, 최근 과로 같은 관련 위험 맥락을 확인한다.
- 검수된 응급/증상 source 기반으로 가능한 위험 범주를 짧게 설명한다.
- 원인을 진단하지 않는다.
- 마지막은 즉시 행동 안내로 닫는다.

답변 방향:

```text
가슴 통증과 숨참은 단순 피로일 수도 있지만, 심장이나 폐 관련 응급 신호일 수 있습니다.
특히 고혈압, 흡연, 심혈관 질환 이력이 있거나 통증이 지속되면 더 조심해야 합니다.
Lemon Aid가 원인을 진단할 수는 없지만, 이런 조합은 응급 평가가 필요한 범위에 들어갑니다.
지금 증상이 지속되거나 심하면 119에 연락하거나 가까운 응급실로 이동하세요.
```

### 15.2 검사수치/치료 판단 질문

예: "LDL 130이면 약 먹어야 해?"

Agent가 할 수 있는 일:

- LDL이 무엇인지 설명한다.
- 일반적으로 함께 보는 위험 요인을 설명한다.
- 사용자 메모리에 있는 질환, 흡연, 복약, 생활습관 맥락을 주의 정보로 연결한다.
- 병원 상담 전에 정리할 정보를 알려준다.

Agent가 하지 않는 일:

- 이 수치면 약을 먹어야 한다고 말하지 않는다.
- 치료 여부를 결정하지 않는다.

### 15.3 약/영양제 병용 질문

예: "혈압약 먹는데 마그네슘 같이 먹어도 돼?"

Agent가 할 수 있는 일:

- 검수 evidence가 있으면 일반 원리, 확인 항목, 주의 조건을 설명한다.
- 사용자 메모리에 복약 언급이 있으면 주의 맥락으로 반영한다.
- 제품 라벨, 함량, 약 이름, 신장/간 기능, 이상 증상처럼 확인할 정보를 안내한다.

Agent가 하지 않는 일:

- 병용 가능 또는 불가를 단정하지 않는다.
- 복용 시작, 중단, 증량, 감량을 지시하지 않는다.

## 16. 현재 구현과 부족한 부분

현재 로컬 worktree 기준으로 이미 구현 흔적이 있는 영역:

- `UserHealthContextSnapshot`
- `ContextResolver`
- `AnswerPlan`
- `AnalysisPlan`
- `ChatRenderer`
- `AnalysisRenderer`
- `BoundaryRenderer`
- `UnknownRenderer`
- `food_records`
- `user_medications`
- reviewed evidence gate
- `unknown_no_reviewed_source`
- `sources[]` 응답 계약

아직 제품 목표 기준으로 더 확인하거나 강화해야 하는 영역:

- 음식 사진에서 실제 g/인분 단위 영양성분 DB 매칭이 agent 컨텍스트로 안정적으로 들어오는지 확인
- 영양제 OCR/성분/함량/단위 파이프라인과 confirmed supplement 계약의 실제 연결 확인
- `profile_memory`, `behavior_memory`, `conversation_memory`, `safety_memory`의 DB 모델과 갱신 정책 설계
- raw chat archive와 conversation summary memory의 보관, 삭제, 동의 정책 설계
- 스마트 생활관리 점수의 세부 알고리즘, UI copy, safety wording 검증
- 체크리스트 수행률과 거절 항목을 `behavior_memory`로 반영하는 루프 구현
- RAG/vector DB를 reviewed source governance와 `AnswerCardNormalizer` 뒤에 붙이는 retrieval eval
- boundary 답변을 "짧은 차단"이 아니라 "결정 금지 안에서 충분히 설명"하는 형태로 품질 개선

## 18. 참고한 공개 memory 패턴

아래 공개 문서들은 Lemon Aid 메모리 구조가 일반적인 agent memory 패턴과 크게 어긋나지
않는지 확인하기 위한 참고다.

- [OpenAI Agents SDK Memory](https://openai.github.io/openai-agents-js/guides/sandbox-agents/memory): 대화 전체가 아니라 유용한 요약, 선호, 메모리를 agent prompt에 주입하는 방식.
