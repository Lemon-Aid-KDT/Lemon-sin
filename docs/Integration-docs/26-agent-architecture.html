🧑 개인화 메모리 (MVP 우선)

  • 사용자별 요약·패턴·선호·주의 맥락
  • 아침 결식, 나트륨 반복, 체크리스트 성공률
  • 다음 분석·답변·체크리스트에 반영

📐 Deterministic 정책 업데이트 (MVP 우선)

  • 검수된 규칙·알고리즘·안전 정책 갱신
  • KDRIs, 복약 boundary, unknown 정책
  • 모든 사용자 또는 조건군에 일관 적용

🔬 모델 학습 (장기 고도화)

  • 충분한 동의·비식별 데이터 축적 후
  • OCR, 음식 인식, 질문 분류 모델 개선
  • LLM fine-tuning: 말투·형식 품질만 (의료 지식 주입 아님)

✅ Agent가 할 수 있는 것 (결정 금지 안에서 설명)

  • 검수 evidence 기반 위험 범주·일반 원리 설명
  • 사용자 메모리 맥락 연결 (복약·질환·생활습관)
  • 상담 전 정리할 정보 안내
  • 응급 질문: 가능한 위험 범주 + 즉시 행동 안내 (119 등)
  • 병용 주의 조건·확인 항목 설명
  • 체크리스트 후보 생성 (사용자 확인 후 저장)

❌ Agent가 하지 않는 것

  • 진단명 확정 / 치료 여부 결정
  • 약 시작·중단·증량·감량·개인 복용량 결정
  • 병용 가능/불가 단정
  • 응급 여부 확정
  • 검수 없는 의료 사실 생성 (LLM 일반 지식 채우기)
  • 사용자 몰래 공식 프로필·복약·알림 수정

📌 구성요소 역할 요약

구성요소 핵심 역할 하지 않는 것
Agent 컨텍스트 선별 → Plan 생성 → 답변·실천안·CTA 제안 몰래 공식 데이터 수정, 진단·치료 결정
LLM Plan을 자연어로 변환, 구조화 출력, OCR 보조 검수 없는 의료 사실 생성, RAG 결과 무검증 사용
ContextResolver 매 요청마다 필요한 메모리·앱데이터·evidence만 선택 모든 기록을 항상 주입 (token 낭비·노이즈)
AnswerCard / RAG 검수된 의료·영양 근거 검색 및 AnswerCardNormalizer 정규화 검색 결과 바로 사용자에게 전달
Memory 4종 개인화 맥락 유지 (profile / behavior / conversation / safety) raw chat/prompt를 개인화에 직접 사용
BoundaryRenderer 결정 금지 범위 내 최대한 상세 설명 + 상담 준비 안내 "모릅니다" 단답, 또는 진단·처방
사용자 확인 게이트 모든 앱 액션 — 사용자 명시 승인 후 실행 자동 저장, 자동 알림 등록, 공식 기록 자동 수정
Deterministic 파이프라인 음식/영양제 OCR → DB 매칭 → 영양성분 계산 → Agent context 전달 LLM이 최종 칼로리·함량 판단

🤖 Multi-Agent 계층 구조

Orchestrator Agent가 요청을 분류하고, 전문화된 4개 하위 Agent를 호출하여 결과를 조합합니다. 각 박스에 마우스를 올리면 상세 설명이 표시됩니다.

🎯 Orchestrator Agent
사용자 요청 분류 → 하위 Agent 호출 → 결과 조합 → 최종 응답 반환
📥 입력
사용자 요청 텍스트
📤 출력
최종 응답 (채팅 / 분석 / boundary)
request_classify response_compose agent_dispatch
🧠 Context & Memory Agent
개인화 컨텍스트 수집
메모리 4종 중 관련분 선택 · 앱 데이터 조회 · Evidence 검색 → Snapshot 조립
📥 입력
요청 유형 + 사용자 ID
📤 출력
UserHealthContextSnapshot
memory_read app_data_query evidence_search context_filter
📋 Planning Agent
Plan 생성
Snapshot 기반으로 AnswerPlan / AnalysisPlan 구조 결정 · 섹션·우선순위 설정
📥 입력
UserHealthContextSnapshot
📤 출력
AnswerPlan / AnalysisPlan
plan_build llm_structure section_rank
⚠️ Safety & Boundary Agent
의료 경계 판단
응급·복약·병용 질문 탐지 · boundary 정책 적용 · unknown topic → backlog 기록
📥 입력
AnswerPlan + safety_memory
📤 출력
방향 판정 (normal / boundary / unknown)
safety_check boundary_policy unknown_backlog_write emergency_detect
🖨️ Rendering Agent
최종 출력 생성
Renderer 선택 · LLM으로 자연어 변환 · structured output 생성 · sources[] 첨부
📥 입력
Plan + 방향 판정
📤 출력
최종 응답 텍스트 + sources[]
llm_generate renderer_select sources_attach structured_output

각 Agent가 어떤 Tool을 어떤 시점에 호출하는지 정리한 매핑입니다.

🔄 Memory 생명주기 (읽기 → 선택 → 사용 → 갱신)

4종 메모리가 하나의 요청 사이클 안에서 어떻게 흐르는지 단계별로 표시합니다.

1
요청 수신 Orchestrator가 요청 유형 분류
request_classify
2
Memory 읽기 4종 메모리 DB 조회
질문과 무관한 메모리는 제외
memory_read
3
컨텍스트 조립 관련 메모리만 선택 → Snapshot 생성
앱 데이터 + Evidence 포함
context_filter
4
Plan 생성 + LLM Snapshot → Plan → LLM 렌더링
Safety 판정 후 응답 생성
llm_generate
5
사용자 확인 제안된 행동에 대해 사용자 승인
(체크리스트·알림·저장)
6
Memory 갱신 승인된 행동 기반 갱신
다음 사이클에 반영
memory_write
갱신되는 메모리: behavior_memory 체크리스트 수행률·거절 항목·반복 패턴 갱신  |  conversation_memory rolling summary 갱신 (전체 raw 채팅 아님)  |  safety_memory 새로운 복약·상호작용 언급 추가  |  profile_memory 명시 확인된 건강 정보만 갱신 (채팅 언급만으로는 공식 등록 안 함)
