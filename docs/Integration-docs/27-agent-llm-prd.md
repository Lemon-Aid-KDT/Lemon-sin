# 27. Agent/LLM PRD

> Status: product requirements draft
> 작성일: 2026-06-04
> 기준 문서: [26-agent-llm-product-direction-reset.md](./26-agent-llm-product-direction-reset.md)
> 기준 worktree: `feat/ai-agent-backend-integration`

## 1. 목적

이 문서는 Lemon Aid Agent/LLM의 제품 요구사항을 다시 정의한다.

Lemon Aid는 만성질환자와 건강관리가 필요한 사용자가 음식, 영양제, 복약, 생활습관,
대화 맥락을 연결해 자신의 관리 상태를 이해하고, 오늘 실천할 행동을 고를 수 있게 돕는
AI 건강관리 서비스다.

Agent는 자동 실행자가 아니라 개인화 건강관리 오케스트레이터다. LLM은 의료 판단자가
아니라 검수 근거와 앱 컨텍스트를 사용자가 이해할 수 있는 답변, 분석, 체크리스트로
표현하는 보조 계층이다.

## 2. 제품 문제

현재까지의 Agent/LLM 문서는 근거 기반 챗봇, 앱 컨텍스트 기반 agent, LLM/RAG 고도화가
여러 문서에 나뉘어 있다. 사용자가 원한 핵심은 단순한 FAQ 챗봇이 아니라 아래를 모두
연결하는 개인화 agent다.

- 음식 사진/기록에서 나온 칼로리, 영양성분, g/인분 단위 정보
- 영양제 사진/기록에서 나온 성분, 함량, 단위, 복용 정보
- 키, 몸무게, 나이, 성별, 질환, 복약, 운동, 흡연, 음주
- 채팅 대화에서 드러난 관심사, 선호, 반복 질문, 실천 패턴
- 오늘 분석과 누적 스마트 분석 결과
- 검수된 의료/영양/복약 evidence

이 정보가 연결되지 않으면 챗봇은 일반 답변만 반복하고, 분석 탭은 사용자에게 실제로 무엇을
해야 하는지 충분히 알려주지 못한다.

## 3. 제품 목표

### 3.1 핵심 목표

- Agent는 앱 컨텍스트와 개인화 메모리를 읽고, 무엇을 분석하고 답해야 하는지 결정한다.
- 음식/영양제 알고리즘 결과와 사용자 프로필, 복약, 생활습관, 채팅 메모리를 연결한다.
- 오늘 분석과 스마트 분석을 통해 현재 관리 상태와 누적 생활관리 안정도를 보여준다.
- 사용자가 실행할 수 있는 체크리스트 후보를 제안하고, 수행 결과를 다음 추천에 반영한다.
- 의료/영양/복약 답변은 reviewed evidence와 source governance를 통과한 지식만 사용한다.
- 의료 boundary는 답변 금지가 아니라 결정 금지로 정의한다.

### 3.2 비목표

- 질병 위험 예측을 제품의 핵심 점수로 만들지 않는다.
- 검사수치만으로 치료 여부를 판단하지 않는다.
- 개인별 복용량, 약 시작/중단/증량/감량을 결정하지 않는다.
- LLM 일반 의료 지식으로 검수되지 않은 질문에 답하지 않는다.
- OCR preview나 사진 분석 후보를 사용자 확정 기록처럼 사용하지 않는다.
- raw chat 전체를 매 답변의 장기 memory처럼 사용하지 않는다.

## 4. 대상 사용자

- 만성질환이 있거나 가족력, 복약, 생활습관 때문에 식사와 영양제 관리를 신경 쓰는 사용자
- 음식과 영양제를 직접 기록하되, 어떤 의미인지 해석받고 싶은 사용자
- "오늘 뭘 먹지?", "이 영양제 괜찮을까?", "내가 뭘 실천하면 좋지?"처럼 생활관리 질문을 하는 사용자
- 긴 의료 설명보다 오늘 할 행동과 확인할 정보를 알고 싶은 사용자

## 5. 핵심 사용자 시나리오

### 5.1 오늘 식사 조정

사용자가 점심으로 라면을 기록하고 저녁 식단을 묻는다.

Agent는 음식 기록의 영양축, 사용자 프로필, 복약/질환 주의 맥락, 과거 선호를 읽고,
나트륨을 낮추는 저녁 후보와 체크리스트를 제안한다.

성공 기준:

- "채소와 단백질을 드세요"처럼 일반론만 말하지 않는다.
- 실제 음식 후보와 조리/섭취 방식을 제안한다.
- 점수는 건강 판정이 아니라 오늘 기록 기준 생활관리 상태로 설명한다.

### 5.2 영양제/복약 주의

사용자가 "혈압약 먹는데 마그네슘 같이 먹어도 돼?"라고 묻는다.

Agent는 복약 memory, 영양제 label 정보, reviewed evidence를 확인한다. 병용 가능 여부를
단정하지 않고, 확인해야 할 약 이름, 제품 라벨, 함량, 신장/간 기능, 이상 증상, 상담 준비
질문을 설명한다.

성공 기준:

- 검수 evidence가 있으면 일반 원리와 확인 항목을 충분히 설명한다.
- 검수 evidence가 없으면 LLM 일반 지식으로 답하지 않는다.
- 약 시작/중단/증량/감량은 지시하지 않는다.

### 5.3 스마트 분석

사용자가 몇 주간 앱을 사용한다.

Agent는 누적 음식 기록, 영양제 체크, 체크리스트 수행률, 대화 메모리, 사용자 프로필을
바탕으로 `스마트 생활관리 점수`와 반복 패턴을 보여준다.

성공 기준:

- 점수는 질병 위험이나 건강 상태 판정이 아니라 생활관리 안정도로 설명한다.
- 반복되는 영양축, 실천 성공률, 선호 시간대, 실패 패턴을 보여준다.
- 사용자가 다음 주에 실천할 수 있는 후보를 1~3개 우선 제안한다.

### 5.4 대화 기반 개인화

사용자가 여러 대화에서 "아침은 자주 못 먹어", "두부는 괜찮아", "너무 긴 답변은 싫어"라고 말한다.

Agent는 raw chat 전체를 매번 쓰지 않고, 핵심만 `conversation_memory`,
`behavior_memory`, `profile_memory`에 요약한다. 다음 답변은 더 짧고 구체적인 아침 대안이나
두부 기반 식사 후보를 제안한다.

성공 기준:

- 채팅 원문 전체를 장기 prompt로 넣지 않는다.
- 선호, 반복 관심사, 답변 스타일은 확인 없이 memory에 반영할 수 있다.
- 복약/질환처럼 공식 기록에 영향을 주는 정보는 memory에 남기되 공식 DB를 자동 수정하지 않는다.

## 6. 기능 범위

### 6.1 MVP 포함

- 앱 컨텍스트 기반 Agent 응답
- 사용자별 memory 4종 구조
- raw chat archive와 agent memory 분리
- 오늘 현재 분석 점수
- 스마트 생활관리 점수
- 체크리스트 기본 모드와 확장 모드
- 체크리스트 수행률과 거절 항목의 `behavior_memory` 반영
- reviewed evidence 기반 의료/영양/복약 답변
- unknown backlog와 evidence 승격 루프
- 의료 boundary의 "결정 금지 안에서 설명" 답변

### 6.2 MVP 이후

- 추천 랭킹 모델 학습
- OCR/음식 인식 모델 개선
- 답변 품질 평가 모델
- 대규모 vector index
- LLM fine-tuning

## 7. 메모리 제품 요구사항

Lemon Aid는 사용자를 더 잘 이해하기 위해 메모리를 유지한다. 메모리는 원문 저장소와
agent memory로 분리한다.

| 메모리 | 제품 요구 |
| --- | --- |
| `profile_memory` | 고정 프로필과 채팅 언급 기반 프로필을 신뢰도와 함께 구분한다. |
| `behavior_memory` | 음식 패턴, 체크리스트 수행률, 거절 항목, 선호 시간대, 반복 실패를 저장한다. |
| `conversation_memory` | 긴 대화에서 핵심 요약, 관심사, 말투 선호를 rolling summary로 유지한다. |
| `safety_memory` | 복약, 질환, 상호작용, 응급 관련 주의 맥락을 답변 전 boundary에 반영한다. |

raw chat은 저장할 수 있지만, 장기 agent memory로 직접 쓰지 않는다. raw prompt는 개인화
메모리가 아니라 감사/디버깅 로그다.

## 8. 분석 제품 요구사항

### 8.1 오늘 현재 분석 점수

오늘 등록된 음식, 오늘 체크된 영양제, 오늘 체크리스트, 오늘 대화 맥락을 기준으로 현재
생활관리 상태를 보여준다.

### 8.2 스마트 생활관리 점수

누적 기록, 실천 패턴, 반복 영양축, 체크리스트 수행률, 대화 메모리를 바탕으로 장기
생활관리 습관의 안정도를 보여준다.

두 점수 모두 질병 위험 예측, 건강 상태 판정, 치료 효과 판정이 아니다.

## 9. 체크리스트 제품 요구사항

- 기본 모드는 우선순위 1~3개를 제안한다.
- 사용자가 더 원하면 확장 모드로 카테고리별 후보를 제공한다.
- 실제 저장과 알림 등록은 사용자가 선택한 항목만 반영한다.
- 수행 여부, 거절, 반복 실패, 선호 시간대, 난이도는 `behavior_memory`에 반영한다.
- 의료/복약 관련 체크리스트는 확인, 기록, 상담 준비로 제한한다.

## 10. 의료 지식 제품 요구사항

- 의료/영양/복약 지식은 reviewed evidence와 source governance를 통과해야 한다.
- RAG는 evidence를 찾는 검색 계층이며, 검색 결과는 `AnswerCard` 같은 내부 프레임으로 정규화한다.
- 검수 지식이 없으면 `unknown_no_reviewed_source` 또는 boundary로 닫는다.
- 응급/복약/검사수치/병용 질문도 답변 금지가 아니라 결정 금지로 다룬다.
- 가능한 위험 범주, 일반 원리, 확인할 정보, 상담 준비 질문, 낮은 위험 행동은 설명한다.

## 11. 성공 기준

- 사용자는 "내 기록을 보고 말한다"는 느낌을 받는다.
- 답변은 일반론보다 음식, 행동, 확인 항목이 구체적이다.
- 체크리스트는 너무 많지 않고 실행 가능하다.
- 사용자별 memory가 다음 답변과 분석에 반영된다.
- 검수되지 않은 건강 사실은 답변에 들어가지 않는다.
- 점수 문구가 질병 위험이나 건강 상태 판정처럼 보이지 않는다.
- 의료 boundary 답변은 차갑게 막지 않고, 결정 금지 안에서 충분히 설명한다.

## 12. 비성공 기준

- 범용 챗봇처럼 건강과 무관한 대화를 길게 이어간다.
- LLM 일반 의료 지식으로 병용 가능 여부를 말한다.
- raw chat 전체를 장기 memory처럼 prompt에 넣는다.
- OCR preview를 확정 영양제/음식 기록처럼 사용한다.
- 스마트 점수를 건강 개선, 질병 위험 감소처럼 표현한다.
- 체크리스트를 사용자 승인 없이 저장하거나 알림으로 등록한다.
- 사용자가 말한 복약 정보를 공식 복약 DB에 자동 등록한다.
