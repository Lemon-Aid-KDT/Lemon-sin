## 📋 한 줄 요약

> 사용자가 올린 음식 사진을 **① 음식/비음식 게이트 → ② 음식 영역 검출(박싱) → ③ 음식 종류 분류 → 칼로리 매핑** 의 다단계(카스케이드) 구조로 처리한다. 각 단계를 가벼운 전용 모델로 분리해 무게를 줄이고, 인식 실패 시 사용자에게 재촬영을 유도한다. 현재 **①번(음식/비음식 게이트)이 구현 완료**(테스트 정확도 97.0%)이며, ②·③번은 확장 예정이다.

## 1. 목표와 범위

- **목표**: 음식 사진 1장 → 어떤 음식인지 식별 → 영양소(칼로리) 매핑의 입력을 만든다.
- **설계 원칙**:
  - 모든 처리를 한 모델에 몰지 않고 **단계별 경량 모델로 분리**해 전체 무게를 낮춘다.
  - **게이트(1단계)** 에서 음식이 아니거나 흐림·어두움으로 인식이 어려우면 **사용자에게 재촬영을 요청**한다.
  - **클래스 확장**(음식 종류 추가)을 전제로, 재학습 비용이 낮은 구조를 택한다.
- **이 문서의 범위**: 이미지 → 음식 종류 후보 산출까지. 영양소 수치 계산·매핑은 `data/nutrition_reference/` 및 후속 문서에서 다룬다.

## 2. 전체 구조 — 3단 카스케이드

```text
사용자 사진
   │
   ▼
[모델 1] 음식 / 비음식 게이트  ──(비음식·흐림·어두움)──▶ "다시 촬영해 주세요"
   │ 통과(음식)
   ▼
[모델 2] 음식 영역 검출(박싱)  ──(음식 미검출)──▶ "음식이 잘 보이게 다시 촬영"
   │ 박스마다 크롭(확대)
   ▼
[모델 3] 음식 종류 분류(크롭 1장씩)  ──(DB에 없는 음식)──▶ 사용자 라벨 입력 → 서버 저장
   │
   ▼
음식 이름 → 칼로리/영양소 매핑
```

**카스케이드 이점**
- 각 모델이 한 가지만 잘하면 되어 단순·경량.
- 게이트와 검출 단계에서 막아 **잘못된 입력을 조기에 걸러내고 재촬영을 유도**.
- 모델 2(검출)는 학습 없이도 동작하므로, **DB에 없는 음식도 박스는 가능** → 사용자 라벨링으로 데이터가 쌓이는 선순환(active learning).

---

## 3. 모델 1 — 음식/비음식 게이트 (구현 완료)

### 3.1 방식
- **CLIP ViT-L/14 임베딩 + 경량 분류 헤드(로지스틱 회귀, linear probe)**.
- 이미지를 CLIP으로 768차원 벡터로 변환한 뒤, 그 위에 가벼운 분류기를 학습.
- GPU 없이 CPU만으로 학습·추론 가능(백본은 추론 시 1회, 헤드 학습은 수 초).

### 3.2 학습 데이터
- 네이버 지도 식당 리뷰 사진을 CLIP 제로샷으로 1차 자동 분류한 결과를 라벨로 사용.
  - food 약 28,000장 / not_food 약 11,000장.
- 학습엔 클래스 균형을 위해 **클래스당 11,000장(총 22,000장)** 임베딩을 추출해 사용.
- 라벨은 자동 분류라 일부 오류가 있으나, 게이트 학습에는 충분.

### 3.3 성능 (test)
| 지표 | 값 |
|------|-----|
| 정확도 | **97.0%** |
| ROC-AUC | **0.997** |
| food precision / recall | 0.977 / 0.964 |

게이트 임계값(음식일 확률 ≥ 임계값이면 통과) 권장값 **0.6~0.7**:
- 0.7에서 비음식 오통과 약 0.8%, 음식 통과 약 94.8%.
- "흐림·어두움"은 별도 클래스가 아니라 **확률이 임계값 미만 → 재촬영**으로 처리(표준 방식).
- 임계값·재촬영 판정 같은 제어/UX는 모델 바깥(앱·Streamlit)에서 담당.

### 3.4 산출물
- 추론 모듈: 이미지 → 음식일 확률(0~1)만 반환.
- 헤드 가중치 + 메타데이터(정확도·권장 임계값) 저장.
- 클래스 확장·검수를 위해 "라벨 오류 의심" 목록도 함께 산출(모델이 강하게 반대한 예시).

## 4. 모델 2 — 음식 영역 검출/박싱 (확장 예정)

### 4.1 핵심 판단
- **박스 라벨(주석) 데이터가 없으므로**, 박스 라벨이 필요한 일반 YOLO 지도학습은 현 시점에서 불리.
- 대신 **제로샷 오픈-보캐뷸러리 검출**(텍스트 프롬프트로 "food" 등을 검출, 학습 불필요)을 1차 채택.

### 4.2 후보 비교 (초기 검토)
| 모델 | 특징 | 비고 |
|------|------|------|
| **GroundingDINO** | 제로샷 검출 정확도 우수 | transformers 내장. 초기 비교에서 주 음식·반찬을 일관되게 박싱 |
| OWLv2 | transformers 내장, 빠름 | 일부 주 음식 누락·과분할 관찰 |
| YOLO-World | 오픈보캐뷸러리 + 빠름 | `ultralytics` 설치 필요 |

> 초기 4장 비교에서는 **GroundingDINO가 더 안정적**. CPU 기준 장당 약 3~5초로, 게이트 통과 사진에만 1회 실행되므로 허용 범위.

### 4.3 확장 경로
- 정확도·속도를 더 높여야 하면, AIHub 등 **박스 라벨 데이터 확보 후 RT-DETR / YOLOv11 fine-tune**(지도학습)으로 전환.
- 검출 결과는 박스별로 크롭하여 모델 3에 한 장씩 전달.

---

## 5. 모델 3 — 음식 종류 분류 (확장 예정)

### 5.1 요구사항
- 현재 보유 음식 클래스 약 92~97종, **시간이 지나며 계속 추가**.
- DB에 없는 음식도 처리(오픈셋) → 사용자 입력으로 데이터 축적.

### 5.2 권장 방식 — 프로토타입(중심점) + CLIP 임베딩
- 각 음식 = "그 음식 사진들의 CLIP 임베딩 평균 벡터(프로토타입)".
- **클래스 추가 = 새 음식 사진 몇 장의 평균 벡터 1개 추가** → 전체 재학습 불필요.
- **모름 판정**: 입력이 어떤 프로토타입과도 유사도가 임계값 미만이면 "DB에 없는 음식"으로 보고 사용자에게 입력 요청 → 서버 저장(다음 학습 자료).
- 정확도를 더 끌어올릴 땐, 데이터 축적 후 로지스틱 헤드 추가 학습(수 초) 또는 파인튜닝.
- 한계: 얼린 CLIP은 매우 유사한 음식(예: 김치찌개 vs 부대찌개) 구분이 다소 약할 수 있음 → 플라이휠로 개선.

### 5.3 taxonomy 연계
- `data/food_images/manifests/taxonomy.json` 기준:
  `cuisine_type`(한/양/중/일/기타) × `meal_component`(main/soup/side/dessert) → `dish_name`.
- 모델 3은 우선 `dish_name` 후보를 산출하고, 상위 cuisine/meal 라벨은 파생 또는 보조 헤드로 확장 가능.

## 6. 경량화 전략 — 백본 공유

- "모델 3개"라고 무게가 3배가 되는 것이 아니라, **백본 개수**가 무게를 결정한다.
- **모델 1(게이트)과 모델 3(분류)은 같은 CLIP 백본을 공유**한다.
  - 게이트는 전체 사진 임베딩, 분류는 크롭 임베딩 — **백본 가중치는 메모리에 1개만** 올리고 입력만 다르게 넣음.
  - 헤드(분류기·프로토타입)는 수십 KB 수준.
- 실제 메모리에 올라가는 백본은 **2개**: CLIP(모델 1·3 공유) + 검출기(모델 2).
- 서버 추론(FastAPI) 기준 충분히 감당 가능. 완전 온디바이스(모바일)로 갈 경우에만 경량화(distill → MobileNet/TFLite)를 별도 검토.

## 7. 데이터 구조 연계

팀 데이터 규약(`data/food_images/`)에 맞춘다.

| 단계 | 폴더 | 비고 |
|------|------|------|
| 원본 | `raw/` (aihub / web_crawl / friend_contributed) | 깃 미포함, 외부 스토리지 |
| 가공 | `interim/`, `processed/` | 학습 입력 |
| 분할 | `splits/` (train/val/test.csv) | |
| 인덱스/분류체계 | `manifests/` (taxonomy.json, classes.json) | |
| 품질 격리 | `quarantine/` (duplicates/low_quality/ambiguous) | 학습 split 오염 방지 |

- **대용량 이미지·모델 가중치·임베딩(.npz)·헤드(.joblib)는 깃에 커밋하지 않는다**(manifest·스크립트·코드만).

## 8. 컴플라이언스 / 기능 플래그

- 음식 이미지 학습(image learning), 멀티모달 LLM, YOLO 등 학습·외부 추론 기능은 팀 규약상 **기능 플래그 기본 OFF**, 발주처/동의 게이트 전까지 비활성.
- 사용자 기여 이미지(친구·사용자 제공)는 수집·이용 동의 흐름(`docs/Nutrition-docs/17-image-collection-consent-plan.md` 참고) 하에서만 학습 자료로 사용.
- 서비스 방향은 진단·처방이 아닌 **건강관리 보조**(칼로리·영양 정보 제공).
