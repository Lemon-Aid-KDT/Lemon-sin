## 📐 리허설 일정 (시연 1주 전부터)

### 리허설 #1: 풀 런쓰루 (D-7)

```
목적: 전체 흐름 확인, 시간 측정

진행:
  1. 발표 자료 + 시연 디바이스 준비
  2. 처음부터 끝까지 한 번 (15-20분)
  3. 시간 초과/부족 체크
  4. 어색한 전환·모호한 문구 기록

체크포인트:
  □ 슬라이드 자동 전환 vs 수동 클릭
  □ 시연 디바이스에서 시드 데이터 정상 표시
  □ 음성 명료도 (녹음 후 청취)
  □ 전체 시간 ≤ 20분
```

### 리허설 #2: 압박 모드 (D-3)

```
목적: 외부 변수 시뮬레이션

진행:
  1. 발주처 관점에서 듣는 청중 1명 (지도교수 등)
  2. 의도적 방해 시도:
     - "근데 그게 정말 되나요?" (Q&A 끼어들기)
     - 시연 중 "Wi-Fi 끊겼어요" 시뮬레이션
  3. 의도적 시연 실패:
     - 시연 디바이스에 일부러 데이터 삭제
     - 백업 영상 재생 연습

체크포인트:
  □ 패닉 없이 백업 시연 영상 재생
  □ Q&A 끼어들기 시 "좋은 질문 — 후반부에" 처리
  □ 압박 상황에서도 30초 내 복구
```

### 리허설 #3: 최종 점검 (D-1)

```
목적: 디테일 점검 + 자신감 확보

진행:
  1. 진행자 + 시연자 + 백업 인원 모두 함께
  2. 처음부터 끝까지 (방해 X)
  3. 마지막 5분 — Q&A 시뮬레이션 (백업 인원이 질문)

체크포인트:
  □ 모든 슬라이드 자연스럽게 전환
  □ 시연 디바이스 충전 100%
  □ 명함, 자료 인쇄본, 백업 USB 준비
  □ 시연 장소 도착 시각 확인 (1시간 여유)
```

## 🎤 예상 질문 (30+ 카테고리별)

### 카테고리 A: 기술적 질문 (★★★)

**Q1. OCR 정확도가 얼마나 됩니까?**
```
A: 종합비타민 라벨 100장 기준 평균 85% 인식률입니다.
   복잡한 한글 라벨은 80% 수준입니다.
   사용자가 결과를 검토·수정할 수 있는 UI를 제공하여
   100% 정확한 데이터를 보장합니다.
   향후 LDB 의료기관 데이터로 학습하면 더 향상될 수 있습니다.
```

**Q2. Claude API 의존도가 높은데 비용은?**
```
A: 영양제 1회 분석에 약 ₩30 수준입니다.
   사용자 1만 명 × 월 5회 = 월 ₩1,500,000 정도.
   Tool Use 캐싱으로 시스템 프롬프트 비용 90% 절감 중.
   백업으로 OpenAI GPT 통합 — 단가 비교 가능.
   장기적으로는 자체 모델 검토 가능합니다.
```

**Q3. 백엔드 응답 속도는?**
```
A: 캐시 미스 첫 호출 약 4~6초, 캐시 히트 < 500ms.
   OCR 1.5s + LLM 2.5s + 매칭/진단 < 50ms + DB 50ms.
   대부분의 시간은 외부 API에 의존.
   동일 라벨 재분석은 Redis 캐싱으로 즉시 응답.
```

**Q4. Hall 동적 모델이 7-step보다 정확하다는 근거는?**
```
A: NIH(국립보건원)에서 제안한 모델로,
   Hall et al. 2011 논문에서 메타분석 1,000건 검증된 모델입니다.
   특히 30~90일 장기 예측에서 단순 모델 대비 15% 정확도 향상.
   다만 학생 프로젝트의 단순화 버전이라
   실제 임상에서는 추가 검증 필요합니다.
```

### 카테고리 B: 사업·시장 질문 (★★★)

**Q5. 필라이즈와 정확히 뭐가 다른가요?**
```
A: (★ 가장 자주 받는 질문)
   필라이즈는 건강한 직장인의 영양제 추천에 강점이 있고,
   저희는 만성질환자 + 의료데이터 영역에 특화되어 있습니다.

   구체적으로:
   - 페르소나 B (52세 만성질환자) 시연에서 보여드린 v4 활동점수,
     UL 초과 경고, 만성질환자용 KDRIs는 필라이즈에 없습니다.
   - LDB 130여 의료기관과 연계 시 처방 데이터 + 영양 데이터 통합
     — 후발주자가 따라올 수 없는 영역입니다.
```

**Q6. 사업화 모델은?**
```
A: 학생 프로젝트 단계라 결정된 것은 없지만,
   현실적인 옵션 3가지를 검토할 수 있습니다:

   1. B2B: LDB 의료기관에 SaaS 형태로 제공
      → 의사가 환자의 영양·활동 패턴을 진료에 활용
   2. B2C 구독: 만성질환 관리 월 ₩4,900
      → AI 분석 + 영양사 상담 (선택)
   3. 영양제 마켓플레이스
      → 식약처 인정 제품만 큐레이션 (광고 X)
```

**Q7. 실제 환자에게 사용하면 의료법 위반 아닌가요?**
```
A: 핵심 우려이고, 학생 팀이 가장 신경 쓴 부분입니다.

   ✓ "진단", "처방", "치료" 단어 0건 — 자동 테스트로 강제
   ✓ 모든 화면에 면책 고지 ("의사·약사 상담을 대체하지 않음")
   ✓ 식약처 인정 기능성 표시만 사용 — 임의 효능 X
   ✓ 데이터 가명처리 — 의료법·개인정보보호법 준수

   "정보 제공" 영역에서 운영하며, 실제 의료 행위는
   의료진과 분리됩니다. (의료기기법 등급 미해당)
```

### 카테고리 C: 데이터·개인정보 질문 (★★)

**Q8. 사용자 건강 데이터는 어디에 저장되나요?**
```
A: 백엔드 PostgreSQL에 가명처리된 형태로 저장됩니다.
   - 이름·이메일·전화번호 등 PII는 별도 테이블, 암호화
   - 영양·활동 데이터는 user_id (UUID)만 키로 사용
   - HealthKit/Health Connect 데이터는 백엔드 전송 시
     별도 동의 화면을 거쳐야 함
   - 사용자가 탈퇴 시 30일 이내 완전 삭제
```

**Q9. 외부 API에 사용자 데이터가 노출되지 않나요?**
```
A: 노출 범위를 최소화했습니다:
   - 영양제 사진 → Google Cloud Vision (이미지만, PII 없음)
   - OCR 텍스트 → Claude API (성분 텍스트만, 사용자 정보 X)
   - 분석 결과 → 백엔드 DB만 저장
   사용자 식별 정보는 외부에 절대 전송되지 않습니다.
```

### 카테고리 D: 기술 깊이 질문 (★)

**Q10. 시스템 장애 시 어떻게 대응하나요?**
```
A: 3단계 백업 전략:
   1. OCR: Google Cloud Vision → CLOVA OCR (자동 폴백)
   2. LLM: Claude → OpenAI GPT (자동 폴백)
   3. DB: PostgreSQL + Redis (캐시 hit ratio 모니터링)

   추가로 4-Tier 테스트 (단위 + 통합 + E2E + 성능)로
   배포 전 자동 검증.
```

**Q11. 확장성은? 사용자 100만 명 가능한가요?**
```
A: 학생 프로젝트 단계라 부하 테스트는 1만 명 동시 접속까지.
   현재 아키텍처로 100만 명 지원하려면:
   - 수평 확장: FastAPI 컨테이너 N개 + 로드 밸런서
   - DB 분리: 읽기 복제본, 인덱스 튜닝
   - 캐싱 강화: Redis Cluster + CDN
   - LLM 비용: 월 ~$15,000 (현재 단가 기준)
```

### 카테고리 E: 어려운 질문 (★★ 대비)

**Q12. 이 정도면 그냥 ChatGPT 써도 되지 않나요?**
```
A: 좋은 질문입니다.
   ChatGPT는 일반 대화에 강하지만:
   1. 사용자별 KDRIs 컨텍스트 자동 적용 X
   2. 식약처 인정 기능성 DB 연동 X
   3. 의료법 표현 자동 검증 X
   4. HealthKit/Health Connect 통합 X
   5. 영양제 사진 → 단위 환산 자동 처리 X

   범용 AI 대비 도메인 특화의 가치를 제공합니다.
```

**Q13. 학생들이 만든 거라 신뢰성이 부족하지 않나요?**
```
A: 정직한 답변:
   네, 학생 프로젝트입니다.
   하지만 다음을 보장합니다:
   - 모든 코드는 4-Tier 테스트 (단위·통합·E2E·성능) 통과
   - 의료법 표현 자동 검증 — 위반 0건
   - 인수인계 산출물 (가이드 25~29) 으로 후속 개발자가 이어받기 쉬움

   상용화 전 의료기관 임상 검증 + 보안 감사 권장.
   학생 팀의 작업은 PoC 단계로 보시는 게 정확합니다.
```

**Q14. AI가 잘못된 권고를 할 수도 있는데, 책임은 누가?**
```
A: 약관·면책 고지에 명시:
   - 본 서비스는 "참고 정보"이며, 의료 행위가 아님
   - 의료진 상담을 대체하지 않음
   - 사용자 본인이 최종 판단

   기술적 안전장치:
   - "진단", "처방" 표현 자동 차단
   - UL 초과 시 "전문가 상담" 강제 안내
   - 식약처 인정 외 효능 주장 금지

   상용화 시 PL 보험 + 의료법 자문 변호사 검토 필수입니다.
```

```markdown
2. 30초 내 복구
3. 그래도 안 되면 백업 영상 → "기술적 이슈로 영상으로 대신..." 멘트

### 시나리오 A: 백엔드 응답 없음
🔧 즉시 (5초):
   - 시연자 디바이스 → "다시 시도" 버튼
   - 백업 인원이 백엔드 헬스체크 (curl /health)

🔧 30초:
   - Docker 컨테이너 재시작 (백엔드 한 줄)
   - 또는 백업 인원이 미리 준비한 다른 환경으로 전환

🔧 실패:
   - "지금 네트워크 상태가 좋지 않은데 시연 영상으로 보여드리겠습니다"
   - 백업 영상 재생

### 시나리오 B: OCR 인식 실패
🔧 즉시:
   - "이 케이스는 추가 학습이 필요한 부분입니다"
   - 미리 준비한 다른 영양제 사진 사용 (백업 1, 2)

🔧 발생 가능 원인:
   - Cloud Vision 일시 장애 → CLOVA로 자동 폴백 (정상 작동 시)
   - 신뢰도 < 0.85 → 폴백 후에도 실패 가능

### 시나리오 C: LLM 응답 5초 이상 대기
🔧 즉시:
   - "AI가 분석 중입니다 — 평소 약 3초 소요됩니다"
   - 동시에 다음 슬라이드 미리 설명
   - 30초 후 결과 안 나오면 → 백업 영상

### 시나리오 D: 디바이스 배터리 소진
🔧 즉시:
   - 시연자 → 백업 디바이스로 전환
   - 진행자 → 백엔드 + 자료 측면으로 전환

🔧 예방:
   - 시연 시작 전 충전 100%
   - 백업 디바이스 1대 추가 충전
   - 시연 도중 충전 케이블 연결 (가능하면)

### 시나리오 E: Wi-Fi 끊김
🔧 즉시:
   - 모바일 핫스팟 활성화 (학생 팀 폰)
   - 디바이스 네트워크 전환

🔧 30초:
   - 안 되면 → 백업 영상

### 시나리오 F: API 쿼터 소진
🔧 사전 예방:
   - 시연 시작 전 모든 API 키 잔액 확인
   - Claude API: 최소 $5 잔액
   - Cloud Vision: 무료 한도 1,000건 확인

🔧 발생 시:
   - "방금 잔액이 모자란 것 같은데..." 솔직히 인정
   - "백업 영상으로 보여드리겠습니다"
```

## ✅ Definition of Done

- [ ] 리허설 #1, #2, #3 모두 완료 (각 회차 기록)
- [ ] 예상 질문 30개+ 작성 + 답변 스크립트
- [ ] 어려운 질문 대응 — 솔직함 + 강점 강조
- [ ] 트러블슈팅 매뉴얼 (장애 시나리오 6개+)
- [ ] 백업 시연 영상 2개 (페르소나 A, B)
- [ ] 시연 후 피드백 양식 준비
- [ ] 시연 당일 리더 1명 + 시연자 1명 + 백업 1명 역할 분담 명확
- [ ] 모든 디바이스 100% 충전, 백업 디바이스 1대
- [ ] 인쇄본 발표 자료 1부 (전원 다운 시)

## 💡 구현 팁

### 자신감 = 준비

리허설 3회 + 예상 질문 30개 답변 스크립트 = 자신감.
무대공포는 준비 부족의 신호.

### 솔직함이 무기

```
✅ "학생 프로젝트라서 이 부분은 미흡합니다. 하지만 ..."
✅ "지금은 모르겠는데 확인해서 답변드리겠습니다"

❌ 모든 질문에 자신감 있는 척 (티남)
❌ 모르는 걸 안다고 답변 (나중에 들통남)
```

### 시연자 vs 진행자 역할 분리

```
진행자: 발주처 응대, 발표 흐름 컨트롤, Q&A 메인
시연자: 디바이스 조작에 집중, 음성은 최소
백업: Q&A 보조, 트러블슈팅, 메모

→ 1인 다역 X (실수 확률 ↑)
```
