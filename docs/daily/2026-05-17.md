#### 다른 팀원이 알아야 할 내용

- 모델 1 Google Vision은 readiness 기준 `ready`로 보이나, 모델 2 PaddleOCR은 runtime 설치와 `ENABLE_LOCAL_OCR=true` 설정 전에는 `not_configured` 상태다.
- 여러 이미지 업로드는 현재 비교/반복 요청 테스트 중심이며, 여러 장을 하나의 제품 분석으로 병합하는 backend merge 정책은 별도 구현이 필요하다.
- ngrok URL은 임시 공개 테스트 경로이므로 실행 상태와 계정 설정에 따라 바뀐다. 테스트 종료 후 backend/ngrok 공개 노출을 닫아야 한다.
- `AUTH_MODE=disabled`는 임시 공개 테스트용이며 운영 배포 설정이 아니다.
- OCR 결과와 건강/영양제 분석 결과는 사용자 확인 없이 최종 기록으로 저장하면 안 된다.

#### 관련 학습 내용

- OCR provider는 사진 속 글자를 읽는 외부/로컬 엔진이다. Google Vision은 외부 서비스이므로 동의와 개인정보 처리가 필요하고, PaddleOCR은 로컬 fallback 후보지만 런타임 설치와 모델 관리가 필요하다.
- ROI는 이미지 안에서 실제 분석해야 할 영역이다. 영양제 병 여러 개가 같이 찍힌 사진에서는 어느 제품을 읽을지 먼저 정해야 한다.
- Governance gate는 모델이나 rule을 바로 운영에 올리지 않고, test split, safety metric, raw data leak 여부를 확인하는 배포 전 판단 장치다.
- Action contract는 화면이 단순 오류 문구를 보여주는 대신 "다시 촬영", "제품 영역 선택", "추가 사진 요청" 같은 사용자 행동으로 이어지게 하는 약속이다.

#### 검토 필요 사항

- 원격 커밋이 없으므로 실제 파일 반영 상태는 PR 또는 다음 커밋에서 확인해야 한다.
- Google Vision은 외부 OCR이므로 raw image, raw OCR text, provider raw payload 저장 정책과 사용자 동의 흐름을 명확히 해야 한다.
- PaddleOCR 정확도나 blur/glare threshold는 공식 성능 claim이 아니라 내부 fixture와 팀 리뷰 기준으로 보정해야 한다.
- ngrok 공개 테스트는 임시 인증 비활성화 상태가 포함되어 있어, 테스트 종료 후 프로세스 종료와 URL 비공개 상태 확인이 필요하다.
- health/nutrition 문구는 "진단", "치료", "처방", "안전 보장"처럼 오해될 표현을 피하고, "현재 입력 정보 기준", "주의가 필요할 수 있음", "전문가 상담 권장" 수준으로 제한해야 한다.

### `origin/jongpil-tech`

#### 작업 목적

당일 새 커밋은 없지만, `taedong-design`의 TODO에서 종필 파트는 DUR 룰 엔진 v0와 영양 알고리즘 통합 담당으로 지정되어 있다.

#### 오늘 기준 해석

- `data/clean/supplement_drug_rules_v0.json`과 `data/clean/kfda_dur_unified.parquet`를 바탕으로 영양제와 의약품 주의 정보를 확인하는 backend rule engine이 다음 작업으로 보인다.
- 제안 API는 `POST /api/v1/safety/check`이며, 입력은 사용자 만성질환, 복용 의약품, 영양제 후보이고 출력은 안전/주의/금기와 근거로 정리되어 있다.

#### 검토 필요 사항

- DB schema가 먼저 확정되어야 API가 안정적으로 룰을 조회할 수 있다.
- yeong-tech의 OCR 결과 shape와 종필의 safety check 입력 shape가 맞아야 한다.

### `origin/sunghoon-database`

#### 작업 목적

당일 새 커밋은 없지만, `taedong-design`의 TODO에서 성훈 파트는 DUR parquet PostgreSQL 적재, 만성질환/동의 플래그, 인증 안정화로 지정되어 있다.

#### 오늘 기준 해석

- 만성질환 정보와 동의 audit은 민감정보 처리에 해당하므로 암호화/분리 저장, 동의 이력, 로그 노출 방지가 필요하다.
- OAuth refresh token rotation과 JWT 만료/갱신 흐름은 태동 mobile 인증 흐름과 맞춰야 한다.

## 5. 오늘의 파트별 학습 정리

### Backend/API

- `/api/v1/supplements/analyze`는 영양제 사진을 서버로 보내 OCR과 분석 결과를 받는 핵심 API다.
- `POST /api/v1/safety/check`는 아직 TODO 단계지만, 복약/영양제 주의 흐름을 담당할 가능성이 있어 안전 문구와 DB 근거가 먼저 필요하다.
- `/ready`는 provider 설정 상태를 빠르게 확인하는 용도이고, heavy OCR/model smoke test는 별도 CLI나 CI job으로 분리하는 방향이 맞다.

### DB

- DUR parquet와 rule JSON은 DB 적재 전 단계의 정제 데이터다.
- 만성질환, 복약, 동의 audit은 민감정보 처리 대상이므로 스키마뿐 아니라 암호화, 접근 제어, 로그 정책을 같이 봐야 한다.
- OCR 원문과 원본 이미지는 분석 편의를 위해 쉽게 저장하고 싶어도 개인정보와 건강 이미지에 해당할 수 있어 저장 최소화 원칙을 지켜야 한다.

### Mobile/UI

- 카메라 화면은 단순 촬영 화면이 아니라 OCR 실패, 품질 경고, 추가 사진 요청, 제품 영역 선택까지 안내해야 한다.
- iPhone Safari/Firebase/ngrok 환경에서는 HTTPS origin, mixed content, localhost 해석 문제가 실제 테스트를 막을 수 있다.
- 결과 화면은 `success`, `partial`, `retry`, `error` 같은 상태를 구분해 사용자에게 다음 행동을 보여줘야 한다.

### Auth/Security

- OAuth, JWT, refresh token, secure storage는 한 흐름으로 검증되어야 한다.
- `AUTH_MODE=disabled`는 공개 테스트에서는 위험하므로 로컬/임시 테스트에만 제한해야 한다.
- Google Vision 같은 외부 OCR은 이미지가 외부로 전송되므로 명시적 동의와 저장/삭제 정책이 필요하다.

### AI/OCR

- Google Vision은 1차 OCR provider 후보이고, PaddleOCR은 로컬 fallback/비교 실험 후보로 보인다.
- ROI/품질 분석은 OCR 이전에 "이 사진을 분석해도 되는가"를 판단하는 layer다.
- fine-tuning과 correction rule은 raw OCR을 무조건 믿는 방식이 아니라 사람이 확인한 확정값과 fixture 기반 평가를 전제로 해야 한다.

### Docs

- 오늘 문서 작업은 실제 구현 완료 선언보다 팀 조율과 다음 우선순위 정리에 초점이 있다.
- `PROJECT_GUIDE.md`에 v6.5 데이터 수집 시스템, 데이터 라이선스, OCR 테스트 흐름을 반영할지 검토가 필요하다.

## 8. 프로젝트 방향 피드백

- PROJECT_GUIDE와 잘 맞는 방향: 영양제 라벨 OCR, 사용자 확인, 만성질환/복약 정보 기반 주의 흐름, 5종 출력 대시보드의 입력 기반을 실제 테스트 가능한 상태로 좁혀 가고 있다.
- 현재 작업에서 비어 있는 연결 지점: test UI와 Flutter 실제 화면, OCR preview 응답과 최종 저장 API, DUR rule DB와 safety check API, Agent input schema가 아직 하나의 계약으로 고정되지 않았다.
- 발표/시연 관점에서 다음에 먼저 정해야 할 것: "iPhone 촬영 -> Google Vision 분석 -> 사용자 확인 preview -> 안전한 저장 전 상태"까지의 단일 시연 경로를 먼저 고정하는 것이 좋다.
- 안전/개인정보/의료 표현 관점에서 조심해야 할 것: 복약/영양제 주의는 진단·처방이 아니라 현재 입력 정보 기준의 참고 안내로 표현해야 하며, raw image/raw OCR text/raw provider payload 저장은 최소화해야 한다.

## 14. 왜 이렇게 진행하는가

Lemon Aid의 MVP 가치는 사용자가 음식/영양제 사진을 찍고, 앱이 그 정보를 구조화한 뒤, 만성질환·복약·건강 데이터와 연결해 조심해야 할 점과 부족/과다 가능성을 이해하기 쉽게 보여주는 데 있다.

그래서 OCR 테스트 UI나 데이터 수집은 별도 실험이 아니라 MVP의 첫 입력 품질을 안정화하는 작업이다. 사진이 흐리거나 성분표가 없거나 여러 제품이 섞인 경우를 구분하지 못하면, 이후 Agent와 대시보드가 아무리 좋아도 잘못된 입력 위에서 동작하게 된다.

또한 복약/영양제 주의 흐름은 안전상 민감하다. DUR 데이터와 만성질환 정보가 들어오는 순간부터는 기능 속도보다 "사용자 확인", "불확실성 표시", "전문가 상담 안내", "동의와 저장 최소화"가 먼저다.
