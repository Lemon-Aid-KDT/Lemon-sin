#### 관련 학습 내용

대규모 폴더 재구성은 기능 구현보다 머지 리스크가 크다. 파일 이름이 같아도 경로가 바뀌면 다른 브랜치의 수정과 충돌할 수 있고, CI 경로나 import path도 함께 바뀐다. 따라서 이런 브랜치는 "그대로 병합"보다 어떤 파일을 기준 구조로 채택할지 먼저 합의하는 편이 안전하다.

#### 검토 필요 사항

- `PROJECT_GUIDE.md`와 `guide.html`이 함께 변경되어 docs-sync 관점 확인이 필요하다.
- `backend/Nutrition-backend` 경로가 기존 `backend/src` 중심 가이드와 달라졌다면 팀 표준 경로 결정을 먼저 해야 한다.
- `.github` 삭제는 CI/리뷰 자동화에 직접 영향을 줄 수 있어 Infra/팀 리드 확인이 필요하다.
- 원본 데이터, PDF, ZIP, Office 파일 제외 정책은 적절하지만, 실제 커밋 tree 스캔 결과를 PR에서도 다시 확인해야 한다.

## 5. 오늘의 파트별 학습 정리

### Backend/API

Nutrition backend는 `yeong-tech`에서 가장 많이 정리된 파트다. FastAPI, OCR intake, KDRIs lookup, 부족 영양소 분석, 체중 예측, 개인정보/동의, regulated OCR intake가 한 폴더로 모였다. 다만 경로가 바뀌면 기존 API import, test path, CI command도 바뀌므로 팀 표준 경로 합의가 필요하다.

### DB

오늘 직접 DB migration 커밋은 확인되지 않았다. 대신 Agent 로드맵은 DB/auth/profile과 consent gate를 선행 단계로 둔다. 개인정보·건강 데이터를 저장하기 전 사용자 identity, consent, audit log 구조가 먼저 있어야 한다는 의미다.

### Mobile/UI

`changmin-model-develop`의 preview 모델은 Mobile에 직접 영향을 준다. UI는 `preview`, `needs_review`, `approved=false`, warnings, evidence 같은 상태를 표시할 수 있어야 한다. `yeong-tech` TODO는 Mobile 담당자가 `mobile/uiux/`, `mobile/flutter_app/`, `assets/` 기준으로 작업하라고 제안한다.

### Auth/Security

Agent harness와 yeong-tech README 모두 동의와 개인정보를 중요한 선행 조건으로 둔다. 외부 OCR/LLM은 기본 차단, 동의 철회 시 Agent 실행 차단, raw OCR text 저장 금지 같은 기준이 반복해서 나타난다.

### AI

오늘 AI 작업은 실제 모델 성능 고도화보다 계약과 안전성 준비에 가깝다. Agent harness, safety policy, supplement preview 모델은 모두 "AI가 만든 결과를 바로 확정하지 않고 검토 가능한 구조로 만든다"는 방향이다.

### Docs

문서 변경이 많다. 특히 `yeong-tech`는 README, docs 폴더, guide, todo-list를 크게 재정리했고, `changmin-aiagent`는 구현 로드맵을 새로 만들었다. 문서가 실제 코드 경로와 맞는지 계속 동기화해야 한다.

## 8. 프로젝트 방향 피드백

- PROJECT_GUIDE와 잘 맞는 방향: AI가 건강 판단을 확정하지 않고, preview와 safety policy를 통해 사용자 확인·전문가 상담·동의 흐름을 먼저 세우는 점은 프로젝트 방향과 잘 맞는다.
- 현재 작업에서 비어 있는 연결 지점: preview 모델, Agent harness, Nutrition backend 구조가 각각 따로 존재한다. 이 셋을 연결할 API schema와 저장 정책이 아직 보고서 기준으로는 확정되지 않았다.
- 발표/시연 관점에서 다음에 먼저 정해야 할 것: 영양제 사진/OCR 결과가 preview 화면까지 도달하는 최소 end-to-end 흐름을 정해야 한다. 실제 provider는 mock이어도 괜찮지만, 상태값과 사용자 승인 흐름은 고정해야 한다.
- 안전/개인정보/의료 표현 관점에서 조심해야 할 것: OCR 원문, 처방전·검사표, 건강 데이터, 복약 정보는 저장·전송·로그 범위를 최소화해야 한다. "진단", "처방", "치료"처럼 보이는 문구는 사용자 화면과 Agent 응답 모두에서 차단해야 한다.

## 14. 왜 이렇게 진행하는가

Lemon Aid는 단순 분석 앱이 아니라 건강 데이터, 복약 정보, 식단, 영양제 기록을 함께 다루는 MVP다. 그래서 기능을 빨리 붙이는 것보다 "누가 동의했는지", "어떤 데이터를 저장하는지", "AI 결과가 확정인지 preview인지", "의료 표현이 안전한지"를 먼저 정해야 한다.

영양제 preview 모델은 사용자가 확인하기 전까지 분석 결과를 확정하지 않게 만든다. Agent harness는 실제 알림이나 저장 같은 side effect 없이 Agent 흐름을 검증하게 만든다. `yeong-tech`의 workspace 정리는 팀원들이 각 파트 작업 위치를 혼동하지 않게 만든다. 세 작업은 서로 다르지만, 모두 MVP 통합 전에 계약을 안정화하는 역할을 한다.

| confirm supplement API | 미정 | 사용자가 preview를 승인한 뒤 실제 영양제 기록으로 저장 | 추정, 설계 필요 |
| Agent harness runner | 내부 스크립트 | scenario와 fixture를 실행해 Agent 흐름을 평가 | harness skeleton |

### DB 변경사항

| 테이블명 | 변경 내용 | 영향 받는 파트 |
|---|---|---|
| 미확정 | 오늘 날짜 커밋에서 새 migration 자체는 확인되지 않음 | DB |
| consent/audit 관련 테이블 | Agent 로드맵상 선행 필요로 언급 | Backend, AI, Mobile, Safety |
| supplement preview 저장소 | preview 결과를 저장할지, 메모리 응답만 할지 결정 필요 | Backend, DB, Mobile |

### 화면 / 기능 흐름 변경사항

```text
영양제 사진 또는 OCR 텍스트
  -> 성분 후보 추출
  -> 성분명/함량/단위/근거/confidence 정리
  -> 공식 기준 매칭
  -> preview 또는 needs_review
  -> 사용자 확인/수정
  -> 승인 후 저장
```

```text
Agent 제안
  -> harness scenario에서 먼저 검증
  -> 금지 표현과 disclaimer 확인
  -> tool preview 반환
  -> 사용자 승인 전 side effect 없음
```

### 인증 / 개인정보 / 안전 검토

- 외부 OCR/LLM은 기본 차단 방향이 확인된다.
- raw OCR text 저장 금지와 consent revoked 차단 시나리오는 실제 API에서도 유지해야 한다.
- 처방전·검사표 OCR intake는 사용자 확인 흐름으로 제한하고 복용량 변경 지시로 이어지지 않게 해야 한다.
- 금지어 목록은 시작점이며, 실제 사용자 문구에서는 진단·치료·처방으로 오해될 수 있는 표현을 추가 검수해야 한다.
