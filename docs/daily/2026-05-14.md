- 전체 로드맵은 backend core algorithms, nutrition standards, OCR/LLM ingestion, Flutter MVP, advanced analysis, demo/handover 순서를 제안한다.
- A1 핸드오프는 식단 인식의 DTO, 예외, 시드 데이터, mock fixture, seed integrity test만 범위로 잡고, YOLO/GCV 실제 adapter나 FastAPI 통합은 A2/A3 이후로 미룬다.
- 이 계획은 `PROJECT_GUIDE.md`의 음식·식단 분석 흐름과 맞지만, 현재 오늘 커밋 기준으로 실제 구현이 올라온 것은 아니다. 따라서 “다음 작업 후보”로만 반영한다.

## 5. 오늘의 파트별 학습 정리

### Backend/API

인증 API는 이제 이메일 인증, 소셜 로그인, 토큰 refresh, 로그아웃, rate-limit까지 함께 봐야 한다. 화면이 로그인 버튼을 누르면 서버는 “누구인지 확인하고, 우리 앱에서 쓸 임시 출입증을 발급하고, 너무 자주 시도하면 잠시 막는” 역할을 한다.

### DB

오늘 직접 main 구조 DB migration을 확인한 것은 아니지만, 인증 쪽은 이메일 인증 기록과 사용자 계정 정책이, yeong-tech 쪽은 KDRIs 데이터와 regulated feature 설정이 저장 기준에 영향을 준다. DB는 기능 뒤에서 정보를 저장하는 공간이므로, 어떤 데이터가 어느 조건에서 저장되는지 팀이 먼저 합의해야 한다.

### Mobile/UI

모바일은 로그인 화면, 이메일 인증 화면, splash 분기, 설정 로그아웃이 실제 인증 상태와 연결되었다. 사용자는 “로그인했다/안 했다”에 따라 다른 화면을 보게 되고, 마지막 로그인 방식 힌트도 보게 된다.

### Auth/Security

OAuth key와 token은 앱 코드에 직접 넣지 않는다. 모바일은 환경값으로 key를 받고, 백엔드는 외부 provider 토큰을 다시 검증한 뒤 자체 JWT를 발급해야 한다. Redis rate-limit는 이메일 인증과 로그인 남용을 줄이는 장치다.

### AI/Data/Safety

KDRIs, 만성질환자 우선순위, OCR/LLM은 건강 관련 판단처럼 보일 수 있다. 오늘 작업에서 중요한 원칙은 “사용자가 확인하기 전까지 OCR/LLM 결과는 확정 데이터가 아니고, 만성질환 우선순위는 치료 지시가 아니라 우선 확인 대상 정렬”이라는 점이다.

### Docs

문서는 기능의 보조물이 아니라 팀이 같은 방향으로 움직이게 하는 계약이다. 특히 오늘은 연구 근거 문서, branch-local todo, handoff 문서가 실제 다음 작업 순서 판단에 쓰였다.

## 8. 프로젝트 방향 피드백

- PROJECT_GUIDE와 잘 맞는 방향: 인증을 먼저 실제 앱 흐름에 연결하고, 영양 기준과 OCR 파이프라인을 구조화하는 방향은 “사진 한 장 → 개인화 분석 → 5종 출력” MVP 목표와 맞다.
- 현재 작업에서 비어 있는 연결 지점: 로그인 이후 동의/프로필/만성질환/복약 입력으로 이어지는 온보딩 연결, OCR 결과가 5종 출력 대시보드로 이어지는 API 계약, 외부 OCR 동의 UX가 아직 daily 기준에서는 확인되지 않는다.
- 발표/시연 관점에서 다음에 먼저 정해야 할 것: 데모 계정 또는 이메일 로그인 fallback, OCR 실패 시 캐시/수동 입력 fallback, 김건강 시나리오에서 실제로 끊김 없이 보여줄 범위를 정해야 한다.
- 안전/개인정보/의료 표현 관점에서 조심해야 할 것: KDRIs와 만성질환 우선순위는 “진단/치료/처방”이 아니라 참고 정보와 상담 권장으로만 표현해야 한다. 외부 OCR은 이미지 전송 동의와 보존 정책이 선행되어야 한다.

## 14. 왜 이렇게 진행하는가

Lemon Aid는 개인 건강 데이터와 음식·영양제 기록을 연결하는 앱이다. 따라서 사용자를 안정적으로 식별하는 인증 흐름이 먼저 있어야 한다. 로그인과 토큰 저장이 흔들리면 만성질환, 복약, 영양제 기록을 누구의 데이터로 저장해야 하는지 확정할 수 없다.

영양 기준과 OCR 파이프라인은 5종 출력의 재료다. 사진 한 장을 찍어도 제품명, 성분, 함량, 음식명, 섭취량을 구조화하지 못하면 개인화 Agent와 평가 Agent가 설명할 수 있는 입력이 부족하다.

문서와 branch-local todo를 같이 보는 이유는 커밋만으로는 “다음에 무엇을 하려는지”가 보이지 않기 때문이다. 오늘 `yeong-tech`의 todo와 handoff는 커밋 분할, CI hardening, OCR provider 구현 순서를 직접 설명하고 있어, 병합 계획에 중요한 근거가 된다.

| 식단 인식은 MVP 핵심 흐름 일부 | jongpil-tech에 A1 계획은 있으나 오늘 구현 커밋 없음 | 다음 작업 후보로만 봐야 함 | 참고 |

## 16. 오늘 변경된 공통 설계

### API 변경사항

| API | Method | 설명 | 상태 |
|---|---|---|---|
| 이메일 인증 코드 발송/검증 | POST 추정 | 회원가입 후 6자리 코드 검증 흐름 | `taedong-design`에서 구현, 계약 확인 필요 |
| `/auth/google` | POST | Google token을 백엔드가 검증하고 JWT 발급 | 구현, 기존 브랜치와 통합 필요 |
| `/auth/kakao` | POST | Kakao token을 백엔드가 검증하고 JWT 발급 | 구현, 기존 브랜치와 통합 필요 |
| token refresh/logout | POST 추정 | access 만료 시 refresh, 로그아웃 시 refresh revoke | 구현, 모바일 인터셉터와 연결 |
| `/api/v1/supplements/analyze` | POST 계획 | OCR adapter를 통해 OCR+parse preview 생성 | `yeong-tech` docs/35 계획 단계 |

### DB 변경사항

| 테이블명 | 변경 내용 | 영향 받는 파트 |
|---|---|---|
| `email_verifications` 추정 | 이메일 인증 코드, purpose, 만료/검증 상태 저장 | Auth, Backend |
| Redis key | `rl:{purpose}:{email}` 류 rate-limit key 사용 | Auth, Infra |
| KDRIs data files | KDRIs 2025 row와 source manifest/checksum 관리 | Data, AI, Backend |
| chronic priority reference | 만성질환별 nutrient priority/caution rule | AI, Safety |

### 화면 / 기능 흐름 변경사항

```text
Splash
  -> AuthState bootstrap
  -> 미인증이면 Login
  -> 인증이면 Home

Login
  -> Email login 또는 Google/Kakao OAuth
  -> 백엔드 인증 API 호출
  -> access/refresh token secure storage 저장
  -> 최근 로그인 방식 저장
  -> router guard가 홈으로 이동

Settings
  -> 로그아웃 확인
  -> refresh revoke + secure storage clear
  -> Login으로 이동
```

### 인증 / 개인정보 / 안전 검토

- OAuth key와 client secret은 모바일 코드에 평문으로 들어가면 안 된다.
- refresh token 저장과 revoke 정책은 개인정보 보호 관점에서 검토가 필요하다.
- 외부 OCR 사용 시 이미지와 OCR 텍스트가 외부 서비스로 전송될 수 있으므로 별도 동의와 보존 정책이 필요하다.
- KDRIs/만성질환 우선순위는 진단, 치료, 처방처럼 보이는 표현을 피해야 한다.
