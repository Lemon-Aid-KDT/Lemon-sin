PROJECT_GUIDE 기준으로는 W3~W4 코어 구현에서 W5~W6 Agent 통합 준비로 넘어가는 중간 단계로 보인다. 음식·영양제 분석 결과를 병원성 데이터, 복약 정보, 만성질환 기준과 연결하려면 백엔드의 인증·동의·감사 게이트가 먼저 안정화되어야 하고, 그 위에 Agent 입출력 계약이 붙어야 한다. 오늘 작업은 그 순서를 비교적 잘 따르고 있다.

다만 오늘 작업은 건강정보, 복약/영양제 주의, OCR 원문, 활동점수, 건강 추세를 모두 건드린다. 실제 병합 전에는 API 응답 shape, DB migration 적용 순서, 민감 건강정보 동의 타입, Agent 안전 문구, raw OCR 노출 여부를 팀 단위로 확인해야 한다. 특히 AI Agent 결과가 사용자에게 조언처럼 보일 수 있으므로 진단·치료·처방으로 읽히지 않게 문구 검토가 필요하다.

- 건강 코칭 문구는 provider가 무엇이든 동일한 safety guard와 사용자 승인 흐름을 거쳐야 한다.

#### 검토 필요 사항

- `origin/changmin-aiagent`와 구현이 중복되므로, 실제 PR은 하나의 기준 브랜치로 정리하는 편이 안전하다.
- Ollama/OpenAI-compatible adapter를 공식 개발 문서 기준으로 검토하고, README나 운영 문서에 필요한 링크와 환경변수를 명시해야 한다.

- Backend/Auth 파트는 `require_analysis_write` scope와 `SENSITIVE_HEALTH_ANALYSIS` 동의가 실제 로그인/동의 UI와 연결되는지 확인해야 한다.
- AI 파트는 default reference range가 코드에 하드코딩되어 있는 부분을 KDRIs/공식 기준 데이터와 어떻게 연결할지 정해야 한다.

#### 관련 학습 내용

- API scope는 사용자가 로그인했는지만 보는 것이 아니라, 이 사용자가 어떤 종류의 API를 호출할 권한이 있는지 나누는 기준이다.
- 미리보기 상태는 AI가 바로 기록하거나 실행하지 않고, 사용자가 OCR 결과와 제안을 확인한 뒤 승인하도록 멈추는 단계다.
- raw OCR text 차단은 사진에서 읽은 원문이 그대로 응답에 섞여 개인정보나 부정확한 건강 문구로 노출되는 일을 막기 위한 조치다.

#### 검토 필요 사항

- Agent 라우트가 새로 추가되므로 OpenAPI 문서, 모바일 API client, 인증/동의 UI가 함께 업데이트되어야 한다.
- default reference range는 시연용 기준으로 보이며, 실제 개인화 기준은 나이, 성별, 질환, 복약, KDRIs/공식 자료와 연결해야 한다.
- audit event에 provider, cost, debug trace 여부가 기록되므로 민감정보가 metadata에 섞이지 않는지 검토해야 한다.

## 5. 오늘의 파트별 학습 정리

### Backend/API

백엔드는 이제 단순 계산 함수 호출에서 인증·동의·감사 로그가 붙은 건강정보 API로 이동하고 있다. 활동점수, 예측, AI daily coaching 모두 사용자의 민감 건강정보를 다루므로, API route마다 scope와 consent를 명확히 둬야 한다.

### DB

`health_daily_summaries`는 날짜별 건강 요약을 저장하는 성격이 강해 TimescaleDB hypertable 후보가 됐다. 이를 위해 기본키에 `measured_date`를 포함하는 migration이 추가됐으며, 실제 병합 전에는 기존 데이터와 외래키 영향 여부를 확인해야 한다.

### Mobile/UI

모바일은 새 AI Agent API가 `preview`와 `completed`를 구분한다는 점을 화면 흐름에 반영해야 한다. OCR 결과가 미확정이면 결과 화면에서 바로 확정 코칭을 보여주기보다 사용자가 확인·수정·승인하는 UI가 필요하다.

### Auth/Security

오늘 작업의 핵심은 로그인 이후에도 건강 분석 동의를 별도로 확인하는 구조다. 또한 API key 방식 OCR, raw OCR text, debug trace, audit metadata는 모두 개인정보·민감정보 노출 가능성이 있어 운영 설정과 로그 정책이 함께 검토되어야 한다.

### AI/Agent

Agent는 독립 Python 패키지에서 백엔드 API로 붙기 시작했다. deterministic provider, fake/local LLM, Ollama/OpenAI-compatible adapter가 함께 등장했으므로, provider 교체 가능성보다 먼저 입력·출력 계약과 안전 문구를 고정해야 한다.

### Docs

AI Agent 쪽 문서는 한국어 번역, architecture, decision log, PR description, todo가 함께 정리됐다. 다만 기능 브랜치가 여러 갈래로 생겼으므로, 최종 병합 기준 브랜치와 문서 위치를 정리해야 한다.

## 8. 프로젝트 방향 피드백

- PROJECT_GUIDE와 잘 맞는 방향: 인증·동의·감사 로그를 먼저 붙이고, 그 위에 Agent daily coaching을 붙이는 순서는 만성질환자 건강정보를 다루는 MVP 방향과 잘 맞는다.
- 현재 작업에서 비어 있는 연결 지점: Mobile 결과 화면이 `preview`, `requires_user_approval`, `safety_warnings`, `actions`를 어떻게 보여줄지 아직 커밋 기준으로는 확인되지 않는다.
- 발표/시연 관점에서 다음에 먼저 정해야 할 것: AI daily coaching demo payload, consent가 없을 때 화면, OCR 미확정일 때 preview 화면, 확정 후 결과 화면을 하나의 시연 경로로 고정해야 한다.
- 안전/개인정보/의료 표현 관점에서 조심해야 할 것: sodium, protein, chronic condition, medication 같은 결과가 "질병 판단"이나 "복용 지시"처럼 보이지 않게 "입력 정보 기준", "주의가 필요할 수 있음", "전문가 상담 권장" 표현을 유지해야 한다.

```text
6. Safety 문구, audit metadata, raw OCR 노출 여부 QA
7. 통합 테스트와 시연용 demo payload 고정
```

이 순서는 확정 지시가 아니라 팀 논의를 위한 제안이다. 특히 DB와 API 계약은 이후 모바일과 AI가 동시에 의존하므로 먼저 안정화하는 편이 병합 리스크가 낮다.

## 14. 왜 이렇게 진행하는가

Lemon Aid는 음식과 영양제 사진을 분석하는 앱처럼 보이지만, 실제 MVP 가치는 개인의 건강정보, 복약 정보, 활동 데이터, OCR 결과를 함께 해석하는 데 있다. 그래서 단순히 Agent를 먼저 붙이는 것보다, 인증된 사용자와 동의된 건강정보만 분석하고, 결과가 사용자 확인을 거친 뒤 기록되는 구조가 먼저 필요하다.

오늘 백엔드 작업은 그 안전한 기반을 다지는 역할이고, AI Agent 작업은 그 기반 위에서 개인화 설명과 행동 제안을 만드는 역할이다. 이 둘이 따로 진행되면 빠르게 보일 수는 있지만, API 계약과 안전 문구가 맞지 않으면 시연 직전에 화면과 서버가 어긋난다. 따라서 DB/API/동의 계약을 먼저 공유하고, Agent 출력과 모바일 화면을 그 계약에 맞추는 순서가 합리적이다.

## 15. 기획 기준과 실제 작업 차이

| Guide 기준 | 실제 커밋에서 보이는 진행 | 차이 또는 위험 | 결정 상태 |
|---|---|---|---|
| 3개 Agent 통합 흐름 | 현재는 daily health coaching Agent와 adapter 중심 | 개인화·평가·챗봇 3개 Agent의 최종 책임 분리가 아직 명확하지 않음 | 검토 필요 |
| KDRIs/공식 기준 기반 개인화 | default reference range가 코드 상수로 들어감 | 시연용 값과 공식 기준이 섞일 위험 | 팀 결정 필요 |
| 사용자 미리보기 후 승인 | 미확정 OCR은 preview로 멈추는 테스트 존재 | 모바일 화면 구현이 아직 확인되지 않음 | 일시적 차이 |
| 의료/영양 안전 표현 | safety guard와 raw text 차단 추가 | 실제 사용자 문구 전체 QA 필요 | 안전 검토 필요 |
| LDB/건강의신 직접 연동 X | 독립 백엔드/Agent API 중심 | 현재 방향은 참조 앱 MVP와 일치 | 유지 |

## 16. 오늘 변경된 공통 설계

### API 변경사항

| API | Method | 설명 | 상태 |
|---|---|---|---|
| `/api/v1/ai-agent/daily-coaching` | POST | 음식/영양제/건강 추세 입력을 받아 daily coaching Agent 결과 반환 | 신규, 검토 필요 |
| `/api/v1/activity/score` | POST | 활동점수 계산 전 인증·민감 건강정보 동의·감사 로그 적용 | 수정, 공유 필요 |
| predictions 관련 API | 추정 | 예측 API에 인증·동의·감사 gate 적용 | 수정, 검토 필요 |
| OCR factory/readiness | 내부 | Google Vision/PaddleOCR provider 선택과 staging 경고 | 수정, 운영 설정 검토 |

### DB 변경사항

| 테이블명 | 변경 내용 | 영향 받는 파트 |
|---|---|---|
| `health_daily_summaries` | 기본키를 `(id, measured_date)`로 확장 | DB, Backend, Migration, Tests |
| `health_daily_summaries` | TimescaleDB hypertable opt-in 준비 | DB, Infra, Backend |

### 화면 / 기능 흐름 변경사항

```text
사용자 로그인
-> 민감 건강정보 분석 동의 확인
-> 음식/영양제 OCR 또는 건강 추세 입력
-> AI daily coaching API 호출
-> OCR 미확정이면 preview + 사용자 승인 요청
-> 확정 입력이면 completed 결과와 findings/recommendations/actions 표시
-> 안전 문구와 action은 사용자 확인을 거쳐 실행
```

### 인증 / 개인정보 / 안전 검토

- `SENSITIVE_HEALTH_ANALYSIS` 동의가 활동점수, 예측, AI daily coaching의 선행 조건으로 들어갔다.
- 클라이언트가 보낸 `user_id`를 신뢰하지 않고 인증된 사용자 subject로 덮어쓰는 흐름은 개인정보 보호 관점에서 적절하다.
- raw OCR text와 unsafe trend text가 응답에 그대로 나오지 않도록 차단하는 테스트가 추가됐다.
- audit metadata에 민감정보가 들어가지 않는지 별도 검토가 필요하다.
- Agent 메시지는 진단, 치료, 처방처럼 읽히지 않도록 Safety Rules 기준 문구 QA가 필요하다.
