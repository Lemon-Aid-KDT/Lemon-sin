#### 관련 학습 내용

- confirmed payload는 사용자가 preview 값을 확인한 뒤에만 서버가 최종 기록으로 다루는 데이터다.
- Agent memory는 원본 이미지나 OCR 전문을 오래 저장하는 공간이 아니라, 다음 코칭에 필요한 요약 패턴만 남기는 구조다.
- SGLang은 로컬 GPU에서 LLM을 OpenAI-compatible API처럼 제공하는 런타임이다. 이 프로젝트에서는 외부 API가 아니라 local/self-hosted 설명 레이어 후보로 쓰인다.
- secure headers는 브라우저나 API client가 더 안전한 방식으로 통신하도록 서버 응답에 붙이는 보안 설정이다.

- `agent_memory`와 `agent_runs` migration이 실제 PostgreSQL 환경에서 main 브랜치의 기존 migration 순서와 충돌하지 않는지 확인해야 한다.
- confirmed food는 nutrient가 비어 있을 수 있으므로, evaluation/Agent가 빈 값을 보고 추정 영양소를 만들지 않는지 확인해야 한다.
- health consent가 필요한 영양제 저장/분석 API에서 동의 누락 시 403 흐름과 모바일 문구가 맞는지 확인해야 한다.
- `PROJECT_GUIDE.md`와 `guide.html` 변경이 docs-sync 규칙을 지켜 생성된 것인지 다시 확인해야 한다.
- `mobile/flutter_app`와 taedong root `mobile/` 중 어느 쪽이 다음 canonical 앱인지 팀 결정이 필요하다.

## 5. 오늘의 파트별 학습 정리

### Backend/API

- `/ai-agent/daily-coaching`, `/supplements/analyze`, `/supplements` 저장, privacy/memory, secure headers가 함께 움직였다.
- API는 화면과 서버가 주고받는 약속이므로, 모바일 confirmed payload와 backend schema가 먼저 고정되어야 한다.
- health consent가 필요한 분석/저장 API는 단순 오류가 아니라 개인정보·민감정보 처리 흐름으로 봐야 한다.

### DB

- `agent_memory`, `agent_runs` 계열 저장 구조가 중요해졌다.
- 이 저장소는 원본 이미지/OCR 전문을 장기 보관하는 공간이 아니라, 다음 코칭에 필요한 요약 기억을 저장하는 방향이다.
- migration은 backend integration branch 기준으로 보이므로, main의 기존 migration과 순서 충돌 여부를 따로 검증해야 한다.

### Mobile/UI

- `mobile/flutter_app`는 daily coaching, supplement preview, food confirm, dashboard shell을 갖춘 현재 통합 후보 앱이다.
- `origin/taedong-design`는 더 풍부한 UI source지만, auth/API 계약이 다르기 때문에 직접 merge보다 port 전략이 필요하다.
- OCR 정확도는 모바일 촬영 UX와도 연결된다. 정면/평면/흐림 없는 촬영 안내가 실제 성능에 영향을 줄 수 있다.

### Auth/Security

- OAuth `.env`, token store, SPKI pinning, secure headers가 모두 등장했다.
- 실제 key와 pin은 커밋 대상이 아니며, production 도메인 확정 전에는 placeholder와 문서 기준을 분리해야 한다.
- 민감 건강정보 동의가 없으면 분석/저장 흐름이 막혀야 하고, 모바일은 이를 사용자가 이해할 수 있는 문구로 보여줘야 한다.

### AI/OCR

- LLM은 음식 인식기나 영양 계산기가 아니라 설명/코칭 문장화 레이어다.
- 음식 영양소 계산과 영양제 기준 판단은 deterministic service, 기준 데이터, 사용자 확인 결과가 맡아야 한다.
- OCR은 합성 데이터와 실사 데이터의 정확도 차이가 크므로, 실사 UX와 데이터셋 검수가 품질 개선의 핵심이다.

### Docs

- 오늘 문서 변경은 단순 보고서가 아니라 구현 경계, 완료 체크리스트, OCR baseline, 운영 smoke, taedong UI port 전략을 담는다.

## 8. 프로젝트 방향 피드백

- PROJECT_GUIDE와 잘 맞는 방향:
  confirmed preview, 사용자 승인, deterministic 판단, LLM 설명 레이어, memory 요약 저장은 "AI는 거드는 손길"이라는 가이드 방향과 잘 맞는다.

  food recognition과 영양 DB lookup은 아직 구현하지 않고 수동 confirmed food로 대체하고 있다. 이는 안전한 임시 경계지만, 5종 출력으로 가려면 표준 음식명과 영양소 계산 계약이 필요하다.
- 발표/시연 관점에서 다음에 먼저 정해야 할 것:
  `mobile/flutter_app`를 기준으로 `대시보드 -> 영양제 촬영/preview -> 사용자 확인 -> daily coaching -> memory 반영` 흐름을 먼저 끝까지 보여주는 편이 좋다. taedong UI port는 이 계약이 안정된 뒤 진행하는 것이 충돌을 줄인다.
- 안전/개인정보/의료 표현 관점에서 조심해야 할 것:
  OCR preview, Agent memory, daily coaching 문구는 진단·치료·처방처럼 보여서는 안 된다. raw image, raw OCR text, raw LLM response, prompt 전문은 저장/노출하지 않는 기준을 계속 유지해야 한다.

## 14. 왜 이렇게 진행하는가

Lemon Aid MVP는 사용자가 음식이나 영양제를 촬영하고, AI가 읽은 정보를 사용자가 확인하고, 그 확인된 데이터만 건강 코칭과 memory에 반영하는 흐름이다. 이 순서가 중요한 이유는 건강·영양·복약 데이터가 민감하고, OCR이나 LLM 출력은 틀릴 수 있기 때문이다.

오늘 작업은 이 경계를 코드와 문서에 동시에 세우는 단계다. 음식 모델이 아직 없기 때문에 모바일이 영양소를 추정하지 않고 수동 confirmed food만 넘기는 것은 좋은 임시 선택이다. 영양제 OCR도 preview와 confirm을 나누는 것이 안전하다. LLM을 로컬 SGLang으로 연결하더라도 판단은 deterministic engine이 맡아야 한다.

OCR 95% 목표도 같은 이유로 측정 기준이 필요하다. 합성 데이터에서는 목표에 가까워 보여도 실사 사진에서 크게 떨어지면, 사용자 촬영 UX와 품질 검사가 없이는 실제 앱 품질이 보장되지 않는다. 따라서 다음 작업은 "화면을 더 많이 만드는 것"보다 "확인된 한 흐름을 끝까지 안전하게 검증하는 것"을 우선해야 한다.

### 인증 / 개인정보 / 안전 검토

- confirmed 전 OCR preview와 food input은 최종 memory/run log에 저장하지 않는 기준을 유지해야 한다.
- 민감 건강정보 분석 동의가 필요한 endpoint는 동의 누락 시 blocked/403 흐름을 명확히 해야 한다.
- LLM 출력은 "진단", "치료", "처방", 제품 구매 유도처럼 보이면 fallback 처리해야 한다.
- local SGLang은 loopback/self-hosted 범위에서만 사용하고, 외부 LLM 전송은 명시적으로 금지 또는 별도 동의가 필요하다.
- SPKI pin, OAuth key, SGLang key placeholder 등은 실제 secret을 저장소에 넣지 않아야 한다.

## 17. 오늘의 팀 결정사항

1. 커밋 문서 기준으로는 `mobile/flutter_app`를 현재 backend/Agent 통합의 canonical 앱으로 유지하고, taedong root `mobile/` 직접 merge는 보류하는 방향이 기록됐다.
2. 음식 인식 모델과 nutrient DB lookup은 이번 단계에서 구현하지 않고, 사용자가 확인한 음식 입력만 Agent payload로 넘기는 경계가 기록됐다.
3. LLM은 판단자가 아니라 설명/문장화 계층이며, SGLang은 local/self-hosted provider 후보로 유지하는 방향이 재확인됐다.
4. OCR 95% 목표는 실사 데이터 기준으로 아직 미달이며, 모바일 촬영 UX와 실사 라벨링 확대가 후속 작업으로 정리됐다.
