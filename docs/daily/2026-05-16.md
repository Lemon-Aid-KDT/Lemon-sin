## 5. 오늘의 파트별 학습 정리

### Backend/API

오늘 새 API 구현 커밋은 없지만, 다음 작업 순서는 더 선명해졌다. DB/auth/profile과 consent gate가 먼저 있고, 그 다음 supplement preview API, meal input DTO/API, Agent tool preview API가 붙어야 한다.

### DB

DB migration 자체는 확인되지 않았다. 하지만 사용자 identity, profile, consent, audit log가 여러 roadmap의 선행 조건으로 반복된다. 저장할 정보와 저장하지 말아야 할 OCR 원문 범위를 먼저 정해야 한다.

### Mobile/UI

Mobile은 아직 당일 커밋이 없지만 후행 의존성이 크다. 영양제 preview, 식단 review flag, Agent tool preview는 모두 사용자가 확인하고 승인하는 화면을 필요로 한다.

### Auth/Security

JWT/OIDC 강화, consent gate, 외부 OCR/LLM opt-in, secret 제외 정책이 핵심이다. 특히 외부 provider를 켜려면 `.env.example`, 동의 UI, 로그 정책, 테스트 skip 조건을 함께 맞춰야 한다.

### AI/OCR

AI는 값을 확정하는 역할보다 구조화·검수·설명 보조 역할로 제한되는 흐름이다. VLM Inspector는 값을 제안하지 않고 판정만 하도록 제한하는 방향이 안전하다.

### Docs

오늘의 실질 작업은 문서 기반 조율이다. 브랜치마다 TODO 문서가 있지만 위치와 용어가 다르므로, daily report가 다음 작업 순서를 묶어 주는 역할을 한다.

## 8. 프로젝트 방향 피드백

- PROJECT_GUIDE와 잘 맞는 방향: "AI는 거드는 손길"이라는 기준에 맞게 preview, review flag, consent gate, safety filter를 먼저 세우려는 방향은 적절하다.
- 현재 작업에서 비어 있는 연결 지점: DB/auth/profile, Mobile approval UI, 실제 FastAPI preview endpoint가 아직 서로 연결되지 않았다.
- 발표/시연 관점에서 다음에 먼저 정해야 할 것: 영양제 분석 preview와 음식 인식 mock 결과 중 하나라도 end-to-end로 보여줄 최소 흐름을 선택해야 한다.
- 안전/개인정보/의료 표현 관점에서 조심해야 할 것: 외부 OCR/LLM 전송, OCR 원문 저장, 만성질환자 우선순위 표현은 모두 안전 검토가 필요하다.

## 14. 왜 이렇게 진행하는가

오늘 새 기능 커밋은 적지만, 브랜치별 TODO를 보면 다음 작업들이 서로 강하게 의존한다. Agent는 사용자 context와 동의 상태가 있어야 안전하게 실행되고, Mobile은 Backend가 내려주는 preview/review 상태값이 있어야 화면을 만들 수 있다. OCR provider는 개인정보 전송과 비용 문제가 있어 기본값과 동의 정책이 먼저 필요하다.

따라서 5월 16일 보고서는 "무엇이 새로 구현됐는가"보다 "무엇을 먼저 합의해야 다음 구현이 덜 흔들리는가"를 정리하는 데 목적이 있다.
