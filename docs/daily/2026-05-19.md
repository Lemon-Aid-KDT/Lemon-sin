#### 다른 팀원이 알아야 할 내용

- AI Agent 통합은 아직 문서 기준으로 `changmin-aiagent`에 분리 유지하는 전략이다.
- FastAPI backend에 붙일 때는 `AgentInput`, `AgentOutput`, preview, trace, safety guard 계약을 다시 확인해야 한다.
- OCR 결과가 사용자 확인 전이면 preview 상태로 멈추는 구조가 PROJECT_GUIDE의 "미리보기 후 승인" 방향과 맞다.
- LLM provider가 Ollama, OpenAI-compatible, fake 중 무엇이든 건강 판단 자체를 맡기지 않는다는 원칙을 유지해야 한다.

#### 관련 학습 내용

- deterministic engine은 같은 입력이면 같은 결과를 내는 계산 코드다. 건강·영양 기준처럼 리뷰와 테스트가 필요한 영역에 적합하다.
- LLM은 문장을 자연스럽게 설명하는 데 쓰되, 질환 판정이나 복약 판단을 직접 내리게 하면 위험하다.
- adapter는 앱 API와 AI 내부 모델처럼 모양이 다른 두 구조를 이어주는 변환 계층이다.
- trace는 왜 그런 결과가 나왔는지 남기는 실행 흔적이지만, 사용자에게 보여주기 전 안전 검토가 필요하다.

#### 검토 필요 사항

- 문서의 설명 원칙과 실제 backend 통합 브랜치의 코드가 계속 일치하는지 확인해야 한다.
- 사용자에게 노출되는 AI 문구는 "현재 입력된 정보 기준", "주의가 필요할 수 있음", "전문가 상담 권장" 같은 완충 표현을 유지해야 한다.
- 장기 기억, 승인 흐름, raw OCR 노출 방지, trace redaction은 구현 완료로 단정하지 말고 통합 테스트로 확인해야 한다.

## 5. 오늘의 파트별 학습 정리

### Backend/API

- 인증, 프로필, supplement register, 사용자 조회, 오류 응답 shape가 여러 브랜치에서 언급됐다.
- API는 화면과 서버가 주고받는 약속이므로, 모바일 UI가 먼저 커져도 서버의 요청/응답 형태가 고정되지 않으면 통합 때 흔들릴 수 있다.

### DB

- `origin/taedong-design`는 social login columns migration을 수정했다.
- `origin/yeong-tech` 문서는 consent_records를 테스트에서 직접 조작하는 절차를 포함한다.
- DB 변경은 마이그레이션 적용 순서와 기존 사용자 데이터 영향까지 확인해야 한다.

### Mobile/UI

- 회원가입 10-step, 약관 모달, 로그인, 하단 탭, 중앙 카메라 FAB, 디자인 토큰이 크게 추가됐다.
- PROJECT_GUIDE의 핵심 사용자 흐름인 "온보딩 -> 홈 -> 카메라 -> 분석 결과"의 앞부분이 구체화되는 중으로 보인다.

### Auth/Security

- OAuth 키, `.env`, `.env.example`, iOS URL scheme, 토큰 저장, 소셜 로그인 컬럼이 같이 움직였다.
- 실제 키나 토큰이 커밋되지 않았는지 확인하고, 개발/운영 환경변수 표를 맞춰야 한다.

### AI/OCR/Data

- AI Agent는 판단과 설명을 분리하는 문서가 추가됐다.
- 영양제 OCR은 오류 상황 검증 문서와 스크립트가 생겼다.
- 음식 이미지 분석은 AI Hub/Roboflow/YOLO 변환 TODO가 생겼지만, train 이미지 누락으로 아직 학습 완료 단계는 아니다.

### Docs

- 오늘은 문서 작업 비중이 높다. 다만 문서가 단순 기록이 아니라 팀 간 계약, 검증 절차, 데이터 파이프라인 순서를 정리하는 역할을 한다.
- `PROJECT_GUIDE.md` 자체를 바꾸는 브랜치가 있으므로, 실제 병합 전 guide 동기화 영향 검토가 필요하다.

## 8. 프로젝트 방향 피드백

- PROJECT_GUIDE와 잘 맞는 방향:
  모바일 온보딩, 동의, 카메라 진입, OCR 검증, AI Agent 설명 구조가 모두 "사용자 확인 후 건강 데이터를 조심스럽게 분석한다"는 방향과 맞다.
- 현재 작업에서 비어 있는 연결 지점:
  모바일이 보내는 회원가입/프로필/OCR 요청 shape와 백엔드가 받는 schema가 확정됐는지 아직 커밋만으로는 불명확하다. 음식 이미지 인식 결과도 영양소 DB와 연결되는 표준 음식명 규칙이 더 필요하다.
- 발표/시연 관점에서 다음에 먼저 정해야 할 것:
  시연 첫 흐름은 `회원가입/동의 -> 홈 -> 카메라 -> OCR/음식 분석 미리보기 -> 사용자 승인 -> 결과` 순서로 고정하는 것이 좋다. 이 흐름에 없는 고급 기능은 병합 우선순위를 낮춰도 된다.
- 안전/개인정보/의료 표현 관점에서 조심해야 할 것:
  건강정보 동의, 복약/영양제 OCR, AI 코칭 문구, 음식 인식 결과는 모두 사용자가 의학적 판단처럼 오해할 수 있다. "입력 정보 기준", "주의가 필요할 수 있음", "전문가 상담 권장" 표현과 raw OCR/trace 비노출 기준을 유지해야 한다.

## 14. 왜 이렇게 진행하는가

Lemon Aid MVP는 사용자가 건강 데이터를 입력하고, 사진을 찍고, 결과를 확인하고, 필요한 경우 AI 설명을 받는 흐름이다. 이 흐름은 겉으로 보면 모바일 화면이 먼저 보이지만, 실제로는 인증, 동의, DB 저장, OCR, 음식/영양제 분석, Agent 설명, 안전 문구가 순서대로 연결되어야 한다.

그래서 회원가입과 동의가 먼저 안정되어야 개인 건강정보를 다룰 수 있다. 카메라 FAB가 있어도 OCR API와 오류 처리, 사용자의 확인/승인 흐름이 없으면 시연에서 막힌다. AI Agent가 있어도 deterministic engine과 safety guard가 없으면 건강 조언처럼 보일 위험이 있다. 음식 이미지 모델도 클래스 매핑과 데이터 사용 기준이 정리되지 않으면 영양소 계산까지 이어지기 어렵다.

오늘 작업은 이 선행 조건을 각 파트가 동시에 밀어 올리는 단계다. 다음 단계에서는 "무엇을 많이 만들었는가"보다 "한 사용자 흐름이 끝까지 이어지는가"를 기준으로 병합 순서를 정해야 한다.

| 테이블명 | 변경 내용 | 영향 받는 파트 |
|---|---|---|

| `consent_records` | 문서상 동의 누락 테스트에서 직접 삭제 대상으로 언급 | Privacy, QA, Backend |

### 화면 / 기능 흐름 변경사항

```text
Splash/Login
  -> 약관 모달
  -> 회원가입 10-step
  -> 홈/챗/점수/설정 하단 4탭 + 중앙 카메라 FAB
  -> 음식 또는 영양제 촬영/등록 흐름으로 연결 예정
```

커밋 기준으로는 OCR 통합 테스트를 위한 화면 진입점과 회원가입 기반이 준비되는 중이다. 실제 카메라 촬영 후 분석 결과 미리보기까지의 end-to-end 동작은 별도 검증이 필요하다.

### 인증 / 개인정보 / 안전 검토

- OAuth 키와 URL scheme 설정은 실제 비밀키가 커밋되지 않았는지 확인해야 한다.
- `.env`는 커밋 금지이고, `.env.example`에는 예시값과 키 이름만 있어야 한다.
- 건강정보/복약/영양제/질환 정보 입력은 민감정보 동의와 연결되어야 한다.
- AI Agent 결과와 OCR 오류 문구는 진단·치료·처방처럼 보이지 않아야 한다.
- raw OCR text, trace, fixture 이미지는 개인정보나 저작권 문제가 없는지 확인해야 한다.

## 17. 오늘의 팀 결정사항

1. 커밋만으로 확정된 팀 결정은 확인되지 않았다.
2. 문서 기준으로는 AI Agent를 `changmin-aiagent` 브랜치에 분리 유지하고, 통합 타이밍은 나중에 정하는 전략이 기록되어 있다.
3. 음식 이미지 데이터는 Roboflow 50개 클래스와 AI Hub 매핑을 사용하되, 직접 대응이 없는 5개 클래스는 억지 매칭하지 않는 방향이 제안되어 있다.
