# Lemon Aid 팀 작업 공유 문서

이 문서는 팀원이 각자 다른 파트를 맡아도 서로의 작업을 쉽게 이해할 수 있도록
매일 공유하는 진행 기록입니다. 코드를 잘 모르는 사람도 읽을 수 있게, "무엇을
만들었는지"보다 "앱에서 어떤 역할을 하는지"와 "다른 파트와 어떻게 이어지는지"를
먼저 설명합니다.

`PROJECT_GUIDE.md`는 프로젝트의 기준 기획서입니다. 이 문서는 그 기준에 맞춰
실제로 진행된 작업을 쉽게 풀어 쓰는 보조 문서입니다.

기록 위치는 `changmin-plan` 브랜치의 `docs/daily/` 폴더입니다. 자동화로 작성할 때도
팀원이 GitHub에 푸시한 브랜치의 변경 내용을 먼저 확인한 뒤, 이 기준에 맞춰 문서를
갱신하고 커밋과 푸시까지 이어갑니다.

## 1. 작성 흐름

작업 보고서는 하루 단위로 끊어 쓰지만, 내용은 매일 이어져야 합니다. 오늘 보고서를
쓸 때는 반드시 전날 보고서를 먼저 읽고, 어제의 작업이 오늘 어떤 작업으로
이어졌는지 적습니다.

처음 pull한 날:

- 본인 파트가 무엇인지 확인한다.
- `PROJECT_GUIDE.md`에서 그 파트의 목표를 먼저 읽는다.
- 오늘 작업이 그 목표 중 어느 부분인지 설명한다.
- 아직 연결되지 않은 UI, DB, API, AI, 데이터 요소를 표시한다.

매일 보고서를 쓸 때:

- 전날 보고서를 먼저 읽는다.
- 전날 진행한 내용 중 오늘 이어서 한 부분을 적는다.
- 새로 바뀐 결정이나 연결된 파트가 있으면 추가한다.
- 단순히 "오늘 무엇을 했다"가 아니라 "어제 흐름에서 오늘 무엇이 쌓였다"를 쓴다.
- 작성 후 `docs/daily/` 변경분만 확인하고 `changmin-plan` 브랜치에 커밋한다.

예시:

```md
어제는 사용자 정보를 저장할 공간을 만들었다.
오늘은 그 저장 공간을 이용해 로그인할 때 사용자를 확인하는 기능을 만들었다.
그래서 이제 로그인 화면이 서버와 연결될 준비가 되었다.
```

## 2. GitHub 확인 흐름

매일 작성할 때는 로컬 기억만 사용하지 않고, GitHub에 올라온 변경을 기준으로
확인합니다. 다만 모든 커밋을 기술적으로 나열하지 않고, 팀원이 학습할 수 있도록
"앱 흐름에서 어떤 의미가 있는 변화인지"로 바꿔 적습니다.

확인 순서:

- `changmin-plan` 브랜치의 전날 daily 문서를 먼저 읽는다.
- 팀원이 당일 푸시한 브랜치와 커밋 메시지를 확인한다.
- 변경 파일을 보고 UI, DB, API, AI, 데이터, 문서 중 어느 파트인지 분류한다.
- 실제 코드 변경이 작아도 사용자 흐름이나 다른 파트 연결에 의미가 있으면 기록한다.
- 아직 병합되지 않았거나 확인이 필요한 내용은 "검토 필요"로 표시한다.

자동화로 작성할 때:

- 새 daily 문서는 `docs/daily/` 아래에 둔다.
- 템플릿이나 작성 기준을 바꾸는 경우 `docs/daily/team-progress.md`를 수정한다.
- 하루 기록을 추가한 뒤 `docs/daily/` 변경분을 다시 읽어 깨진 링크와 어색한 문구를 확인한다.
- 커밋 메시지는 `docs(daily): ...` 형식으로 작성한다.
- 푸시는 `origin/changmin-plan`로만 진행한다.

## 3. 오늘의 흐름 요약

하루 동안 팀 전체가 어떤 부분을 진행했는지 짧게 정리합니다. 단순 작업 목록이
아니라, 전날부터 오늘까지 프로젝트 흐름이 어떻게 이어졌는지 설명합니다.

좋은 요약은 다음 질문에 답합니다.

- 전날 어떤 작업에서 이어졌나?
- 오늘 앱의 어떤 부분이 조금 더 완성되었나?
- 아직 임시 데이터로만 확인한 부분은 무엇인가?
- 다른 팀원 작업과 새로 연결된 부분은 무엇인가?
- 기획서와 다르게 진행한 부분이 있다면 무엇인가?

예시:

```md
## 2026-05-13 전체 요약

어제는 사용자 정보를 저장할 기본 구조를 만들었다. 오늘은 그 구조를 바탕으로
로그인 화면이 서버에 이메일과 비밀번호를 보낼 수 있도록 요청 흐름을 정리했다.
영양제 분석 결과 화면은 아직 임시 데이터로 확인 중이지만, 백엔드에서 분석 결과를
어떤 형태로 보내야 하는지 UI와 맞춰 볼 수 있게 되었다.
```

## 4. 파트별 작업 설명

파트별 작업은 전문 용어보다 사용자 흐름을 기준으로 설명합니다.

예를 들어 "users 테이블 생성"이라고만 쓰지 않습니다. 대신 "사용자가 회원가입할
때 입력한 정보를 저장할 공간을 만들었다"처럼 적습니다.

작성 항목:

- 날짜
- 담당 파트
- 작업한 브랜치
- 내 파트 목표
- 그 목표 중 오늘 건드린 부분
- 전날 보고서에서 이어진 부분
- 오늘 한 일
- 쉽게 설명하면
- 연결되는 부분
- 기획서와 다른 점
- 결정 이유

템플릿:

```md
### 2026-05-13 / 백엔드 / feat/backend-auth

- 내 파트 목표:
  사용자의 정보와 앱의 주요 기록을 안전하게 저장하고, 화면에서 필요한 기능을
  서버 API로 제공한다.

- 그 목표 중 오늘 건드린 부분:
  로그인 화면이 사용할 사용자 확인 기능을 만들었다.

- 전날 보고서에서 이어진 부분:
  전날에는 사용자 정보를 저장할 공간을 만들었다. 오늘은 그 저장 공간을 이용해
  이메일과 비밀번호가 맞는지 확인하는 흐름으로 이어졌다.

- 오늘 한 일:
  사용자가 회원가입하고 로그인할 수 있도록 서버 쪽 기본 기능을 만들었다.

- 쉽게 설명하면:
  앱에서 이메일과 비밀번호를 입력했을 때, 서버가 "가입된 사용자 맞는지" 확인할
  수 있는 구조를 만든 것이다.

- 연결되는 부분:
  - UI: 로그인 화면에서 이 기능을 호출해야 한다.
  - DB: 사용자 정보가 저장되는 users 테이블과 연결된다.
  - 보안: 비밀번호는 그대로 저장하지 않고 암호화해서 저장해야 한다.

- 기획서와 다른 점:
  원래는 이메일 인증까지 포함되어 있었지만, 우선 MVP에서는 로그인 성공/실패
  흐름을 먼저 연결하기로 했다.

- 결정 이유:
  발표 시연에서는 이메일 인증보다 "앱에서 로그인 후 홈 화면으로 이동하는 흐름"이
  더 중요하기 때문이다.
```

## 5. 서로 연결되는 부분

각 파트의 작업이 따로 떨어져 보이지 않도록, "이 작업이 어디로 이어지는지"를
적습니다.

특히 DB, API, UI, AI 작업은 아래처럼 풀어 씁니다.

| 작업 종류 | 쉽게 설명하는 방식 | 예시 |
|----------|-------------------|------|
| DB | 정보를 저장하는 공간 | 사용자의 이메일, 비밀번호, 복약 정보를 저장하는 공간 |
| API | 화면과 서버가 주고받는 약속 | 로그인 화면이 서버에 이메일과 비밀번호를 보내고 결과를 받는 통로 |
| UI | 사용자가 직접 보는 화면 | 로그인 버튼, 영양제 분석 결과 카드, 오류 안내 문구 |
| AI | 사용자의 입력을 해석하거나 설명을 돕는 기능 | 영양제 사진에서 제품명을 찾고, 주의 성분을 쉬운 말로 설명 |
| 데이터 | AI와 알고리즘이 참고하는 기준 자료 | 식약처 영양제 정보, KDRIs 영양 기준, 음식 영양성분 자료 |

연결 예시:

```md
### 영양제 분석 흐름

- 데이터:
  영양제 이름과 성분 정보를 저장한 기준 자료를 준비한다.

- AI:
  사용자가 찍은 사진에서 제품명과 성분명을 읽어낸다.

- 백엔드:
  AI가 읽은 이름을 기준 자료와 비교해서 가장 가까운 영양제를 찾는다.

- UI:
  찾은 결과를 사용자가 확인하고 수정할 수 있는 화면으로 보여준다.

- 사용자 입장:
  사진을 찍으면 앱이 영양제 정보를 자동으로 채워 주고, 사용자는 틀린 부분만
  고치면 된다.
```

## 6. 기획서와 달라진 점

`PROJECT_GUIDE.md`와 실제 진행이 달라졌다면 반드시 이유를 남깁니다.

다르게 진행한 이유가 기록되어야 나중에 팀원이 "왜 이렇게 만들었지?"라고 다시
추적하지 않아도 됩니다.

작성 기준:

- 원래 기획서에는 무엇이라고 되어 있었나?
- 실제로는 어떻게 진행했나?
- 왜 그렇게 바꿨나?
- 임시 결정인가, 최종 결정인가?
- `PROJECT_GUIDE.md`도 수정해야 하는가?

템플릿:

```md
### 2026-05-13 / 이메일 인증 제외

- 원래 기획서:
  회원가입 후 이메일 인증까지 포함한다.

- 실제 진행:
  MVP에서는 회원가입과 로그인 성공/실패 흐름만 먼저 만든다.

- 바꾼 이유:
  발표 시연에서는 이메일 인증보다 로그인 후 주요 화면으로 이동하는 흐름이 더
  중요하다. 이메일 발송 설정은 외부 서비스와 연결이 필요해서 통합 일정에 영향을
  줄 수 있다.

- 결정 상태:
  임시 결정. MVP 이후 이메일 인증을 다시 붙일 수 있다.

- 기획서 수정 필요:
  필요함. 인증 범위를 "MVP 우선 구현"과 "후속 구현"으로 나눠 적어야 한다.
```

## 작성 원칙

- 전날 보고서를 먼저 읽고 이어지는 흐름을 남긴다.
- 처음 작성할 때는 본인 파트의 목표를 먼저 확인한다.
- 코드를 모르는 사람도 이해할 수 있게 쓴다.
- 파일명이나 함수명만 나열하지 않는다.
- "이 작업이 사용자에게 어떻게 보이는지"를 먼저 설명한다.
- DB 작업은 "어떤 정보를 저장하는 공간인지"로 설명한다.
- API 작업은 "화면과 서버가 주고받는 약속"으로 설명한다.
- AI 작업은 "사용자의 어떤 입력을 어떻게 도와주는지"로 설명한다.
- 기획과 달라진 선택은 숨기지 않고 이유를 남긴다.
- 아직 확정되지 않은 내용은 "임시", "확정", "검토 필요" 중 하나로 표시한다.
- 커밋이나 파일명은 근거로만 쓰고, 본문은 사용자가 이해할 수 있는 흐름으로 쓴다.
