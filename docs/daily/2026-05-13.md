## 5. 오늘의 파트별 학습 정리

### Auth / Mobile

소셜 로그인은 모바일 앱이 외부 제공자에서 토큰을 받고, 백엔드가 그 토큰을 다시 검증한 뒤 우리 서비스용 토큰을 발급하는 구조다. 모바일에서 로그인 버튼이 생겼다고 끝나는 것이 아니라, provider 설정, 서버 검증, DB 사용자 연결, 토큰 저장, 로그아웃까지 하나의 흐름으로 맞아야 한다.

### Backend / DB

소셜 로그인은 기존 이메일/비밀번호 로그인과 데이터 조건이 다르다. 소셜 계정은 비밀번호가 없을 수 있고, 이메일도 항상 안정적으로 제공된다고 가정하기 어렵다. 그래서 DB nullable 정책과 계정 병합 정책을 함께 검토해야 한다.

### Tooling / CI

팀 프로젝트에서는 Python 버전, 줄바꿈, 테스트 임시 파일처럼 기능과 직접 관련 없어 보이는 요소도 협업 안정성에 영향을 준다. CI 기준을 한 번 정하면 문서와 로컬 개발 환경도 같은 기준으로 맞춰야 한다.

### AI / OCR / Nutrition

## 8. 프로젝트 방향 피드백

### PROJECT_GUIDE와 잘 맞는 방향

- 소셜 로그인 작업은 `PROJECT_GUIDE.md`의 인증·온보딩 흐름을 실제 앱과 서버로 연결하는 기반이다. 개인화 건강 데이터는 사용자 식별이 먼저 안정화되어야 하므로, 로그인 흐름을 먼저 잡는 방향은 타당하다.
- `yeong-tech`의 OCR/LLM/vision 작업은 핵심 기능 명세의 “분석 알고리즘 + 3개 Agent 통합 구조”와 직접 연결된다. 특히 영양제 사진에서 제품명·성분을 읽고 구조화하는 작업은 5종 출력의 출발점이다.
- `work-space/jongpil`의 Python 3.13/CI 정리는 눈에 띄는 사용자 기능은 아니지만, 여러 팀원이 각자 브랜치에서 작업하는 현재 상황에서는 머지 안정성을 높이는 필수 기반이다.
- `changmin-plan`의 daily 문서화 작업은 팀원이 서로 다른 파트를 맡아도 프로젝트 전체를 설명할 수 있게 만드는 운영 장치다. 발표와 회고 자료로도 이어질 수 있다.

### 현재 비어 있는 연결 지점

- 소셜 로그인은 화면과 서버가 각각 생겼지만, 로그인 성공 후 `PROJECT_GUIDE.md`의 동의, 기본정보, 만성질환, 복약, 목적 선택, 건강 데이터 권한 흐름으로 이어지는지는 아직 확인되지 않는다.
- AI/OCR 파이프라인은 풍부하게 진행됐지만, 모바일 카메라 화면과 5종 출력 대시보드까지 한 번에 이어지는지는 별도 통합 검증이 필요하다.
- 건강/영양 분석 결과가 사용자에게 어떤 문구로 보이는지 아직 중요하다. `PROJECT_GUIDE.md`의 R2 의료법·약사법 리스크와 R8 DTx 오인 리스크를 기준으로 문구 검수가 필요하다.

### 발표/시연 관점에서 다음에 먼저 정해야 할 것

- 김건강 시나리오 기준으로 “로그인 → 프로필 로드 → 영양제 사진 분석 → 5종 출력 → 챗봇 알림 등록” 중 어디까지 실제로 끊김 없이 보여줄지 정해야 한다.
- AI/OCR이 실패했을 때 사용할 백업 사진, 캐시 결과, 수동 입력 폴백을 준비해야 한다.
- 소셜 로그인은 시연장에서 외부 provider 설정이나 네트워크 문제에 막힐 수 있으므로, 이메일 로그인 또는 데모 계정 fallback도 필요하다.

### 안전/개인정보/의료 표현 관점에서 조심해야 할 것

- “부족”, “과다”, “권장”, “예측”이라는 표현은 사용자가 진단이나 처방으로 오해하지 않도록 “참고”, “관리”, “상담 권장” 중심으로 다듬어야 한다.
- 이미지, OCR 텍스트, 건강 데이터, 동의 기반 학습 데이터는 어떤 조건에서 저장되고 삭제되는지 명확해야 한다.
- 모바일 앱에 API key나 provider secret이 들어가면 안 되고, 토큰 저장은 secure storage 기준을 유지해야 한다.

## 9. 왜 이렇게 진행하는가

오늘 작업 순서는 `PROJECT_GUIDE.md`의 MVP 목표를 기준으로 보면 자연스럽다. Lemon Aid는 단순 영양제 정보 앱이 아니라, 사용자 개인의 건강 정보와 섭취 기록을 바탕으로 분석하고 설명하는 앱이다. 그래서 먼저 사용자를 식별하는 로그인 흐름이 필요하다. 누가 로그인했는지 알아야 그 사람의 만성질환, 복약, 영양제 기록, 건강 데이터와 분석 결과를 연결할 수 있다.

동시에 OCR/LLM/vision 파이프라인을 진행하는 이유는 5종 출력의 재료가 되는 구조화 데이터가 필요하기 때문이다. 사진을 찍어도 제품명, 성분, 함량을 읽어내지 못하면 개인화 Agent나 평가 Agent가 판단할 수 있는 입력이 부족하다.

CI와 Python 버전 정리는 기능처럼 보이지 않지만, 여러 브랜치에서 백엔드·모바일·AI가 동시에 움직이는 지금 단계에서는 필수다. 팀원이 각자 만든 코드를 main에 합칠 때 같은 기준으로 검사하지 않으면, 마지막 통합 단계에서 문제가 한꺼번에 터질 수 있다.

문서 정리와 daily 보고서 자동화는 팀 운영을 위한 기반이다. `PROJECT_GUIDE.md`가 말하는 방향은 넓고 복잡하기 때문에, 매일 어떤 브랜치가 어느 목표에 기여했는지 설명하지 않으면 팀원이 전체 흐름을 놓치기 쉽다.

### 인증 / 개인정보 / 안전 검토

- 소셜 로그인은 개인정보와 토큰 저장이 포함되므로 보안 검토가 필요하다.
- 영양제 분석, 건강 데이터, 동의 기반 학습은 Safety Rules 검토가 필요하다.
- 건강/영양 문구는 진단, 치료, 처방처럼 읽히지 않게 작성해야 한다.
- 이미지/OCR 원문과 건강 관련 데이터의 보존 기간, 삭제, 학습 사용 여부를 명확히 해야 한다.

## 12. 오늘의 팀 결정사항

1. 일일 작업 공유 문서는 `changmin-plan/docs/daily/`에 날짜별로 작성한다.
2. GitHub 정리 요청은 특정 로컬 브랜치가 아니라 `origin/*` 전체 브랜치 기준으로 해석한다.
3. 커밋 목록은 근거로만 사용하고, 문서는 팀원이 이해할 수 있는 앱 흐름 중심으로 작성한다.
4. 인증, 개인정보, 건강/영양/복약 관련 변경은 일반 기능보다 강한 검토 항목으로 표시한다.
5. 앞으로 daily 문서는 `main/PROJECT_GUIDE.md` 기준으로 현재 진행 단계와 다음 우선순위를 함께 설명한다.
