## 5. 오늘의 파트별 학습 정리

### Auth / Mobile

소셜 로그인은 모바일 앱이 외부 제공자에서 토큰을 받고, 백엔드가 그 토큰을 다시 검증한 뒤 우리 서비스용 토큰을 발급하는 구조다. 모바일에서 로그인 버튼이 생겼다고 끝나는 것이 아니라, provider 설정, 서버 검증, DB 사용자 연결, 토큰 저장, 로그아웃까지 하나의 흐름으로 맞아야 한다.

### Backend / DB

소셜 로그인은 기존 이메일/비밀번호 로그인과 데이터 조건이 다르다. 소셜 계정은 비밀번호가 없을 수 있고, 이메일도 항상 안정적으로 제공된다고 가정하기 어렵다. 그래서 DB nullable 정책과 계정 병합 정책을 함께 검토해야 한다.

### Tooling / CI

팀 프로젝트에서는 Python 버전, 줄바꿈, 테스트 임시 파일처럼 기능과 직접 관련 없어 보이는 요소도 협업 안정성에 영향을 준다. CI 기준을 한 번 정하면 문서와 로컬 개발 환경도 같은 기준으로 맞춰야 한다.

### AI / OCR / Nutrition

영양제 이미지 분석은 OCR, vision, LLM, 영양제 기준 데이터, 사용자 동의가 연결된 흐름이다. 모델이 정보를 추출하더라도 최종 문구는 안전하게 표현되어야 하고, 사용자의 건강 데이터가 어떻게 저장되고 학습에 쓰이는지 명확해야 한다.

### Docs / Project Management

문서는 파일 보관소가 아니라 팀의 작업 흐름을 설명하는 시스템이다. 커밋 목록을 모으는 자동화와, 팀원이 이해할 수 있게 설명하는 문서 작성 기준을 분리해야 매일 반복 가능한 공유 문서가 된다.

## 6. 오늘 변경된 공통 설계

### API 변경사항

| API | Method | 설명 | 상태 |
|---|---|---|---|
| `/auth/google` | POST | Google `id_token` 검증 후 Lemon Aid JWT 발급 | 구현됨, 검토 필요 |
| `/auth/kakao` | POST | Kakao access token 검증 후 Lemon Aid JWT 발급 | 구현됨, 검토 필요 |
| 영양제/OCR/예측 관련 API | 여러 항목 | OCR 텍스트 분석, 영양제 등록/매칭, 예측 보조 흐름 확장 | `yeong-tech`에서 대규모 진행, 상세 리뷰 필요 |

### DB 변경사항

| 테이블명 | 변경 내용 | 영향 받는 파트 |
|---|---|---|
| `users` | `google_id`, `kakao_id` 추가, `email`, `password_hash` nullable 변경 | Auth, Backend, Mobile |
| privacy/consent 관련 테이블 | 동의 감사와 학습 gate 관련 migration 갱신 | Safety, Privacy, AI |
| supplement/analysis 관련 테이블 | 영양제 분석과 이미지/OCR 파이프라인 관련 migration 갱신 | AI, Nutrition, Backend |

### 화면 / 기능 흐름 변경사항

```text
모바일 로그인 화면
  → Google/Kakao 선택
  → 외부 provider 토큰 수신
  → 백엔드 /auth/google 또는 /auth/kakao 호출
  → Lemon Aid access/refresh token 저장
  → TODO: 로그인 성공 후 홈 화면 이동

영양제 분석 흐름
  → 사용자가 영양제 이미지 또는 텍스트 제공
  → OCR/vision/LLM이 제품명과 성분 후보 추출
  → 백엔드가 기준 데이터와 매칭
  → 사용자에게 확인/수정 가능한 결과 제공
  → 동의된 범위 안에서만 학습 또는 보존
```

### 인증 / 개인정보 / 안전 검토

- 소셜 로그인은 개인정보와 토큰 저장이 포함되므로 보안 검토가 필요하다.
- 영양제 분석, 건강 데이터, 동의 기반 학습은 Safety Rules 검토가 필요하다.
- 건강/영양 문구는 진단, 치료, 처방처럼 읽히지 않게 작성해야 한다.
- 이미지/OCR 원문과 건강 관련 데이터의 보존 기간, 삭제, 학습 사용 여부를 명확히 해야 한다.

## 7. 오늘의 팀 결정사항

1. 일일 작업 공유 문서는 `changmin-plan/docs/daily/`에 날짜별로 작성한다.
2. GitHub 정리 요청은 특정 로컬 브랜치가 아니라 `origin/*` 전체 브랜치 기준으로 해석한다.
3. 커밋 목록은 근거로만 사용하고, 문서는 팀원이 이해할 수 있는 앱 흐름 중심으로 작성한다.
4. 인증, 개인정보, 건강/영양/복약 관련 변경은 일반 기능보다 강한 검토 항목으로 표시한다.
