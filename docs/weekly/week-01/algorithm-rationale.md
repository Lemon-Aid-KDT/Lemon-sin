# 알고리즘 수정 근거 정리

> 기준 자료: `26-algorithm-modification-paper-rationale.pdf`  
> 연결 기준: `main/PROJECT_GUIDE.md`의 알고리즘, AI 호출 흐름, 컴플라이언스 파트

## 한 줄 요약

이번 자료의 핵심은 알고리즘을 더 복잡하게 만드는 것이 아니라, **공식 근거가 있는 계산식**과 **프로젝트 운영 중 검증해야 할 규칙**을 분리해서 설명 가능성과 안전성을 높인 것입니다.

## 전체 내용 요약

PDF는 Lemon Aid의 건강관리 분석 흐름을 다음처럼 정리합니다.

1. BMI, BMR, HRmax, KDRIs/DRI처럼 논문이나 공식 기준이 있는 값은 **공식 기반 계산식**으로 둡니다.
2. 성별, 나이, BMI에 따른 걸음수 보정, 질환 가중치, 부족/낮음/과다 기준, 영양제 매칭 가중치처럼 팀이 정한 값은 **프로젝트 운영 규칙**으로 따로 둡니다.
3. OCR이나 LLM이 만든 영양제 분석 결과는 바로 확정하지 않고 **후보 결과**로 보여준 뒤 사용자가 확인하게 합니다.
4. 사용자에게 보이는 문구는 진단, 치료, 처방처럼 들리지 않게 하고, **참고 정보**, **주의 가능성**, **전문가 상담 권장** 형태로 제한합니다.
5. 나중에 같은 입력으로 같은 결과가 나왔는지 확인할 수 있도록 입력값, 결과값, 알고리즘 버전, KDRIs 출처 버전 같은 정보를 함께 남깁니다.

비전공자 관점에서 보면, 이 작업은 "AI가 건강 판단을 마음대로 하게 만드는 일"이 아닙니다. 오히려 계산기처럼 근거가 있는 부분은 고정하고, 아직 검증이 필요한 부분은 운영 규칙으로 표시해서 팀이 나중에 검토할 수 있게 만든 것입니다.

## 팀 역할별 관련 범위

### C. AI 엔지니어

AI 엔지니어 파트는 OCR, LLM 구조화, Pydantic 검증, 의료법 표현 검수와 직접 연결됩니다.

- OCR 텍스트를 LLM에 넘겨 정해진 JSON 구조로 바꿉니다.
- LLM 응답은 Pydantic 스키마로 다시 검증합니다.
- "복용하세요", "치료됩니다" 같은 표현이 나오지 않도록 프롬프트와 필터를 둡니다.
- AI는 판단을 확정하는 주체가 아니라, OCR 결과를 정리하고 설명을 돕는 역할입니다.

### D. 백엔드

백엔드 파트는 영양제 이미지 업로드, 분석 preview 생성, API 응답, 버전 저장과 연결됩니다.

- `/api/v1/supplements/analyze` 같은 API에서 이미지를 받습니다.
- 이미지 형식, 크기, 손상 여부를 먼저 검사합니다.
- 원본 이미지나 OCR 원문을 불필요하게 저장하지 않고, 필요한 해시와 preview 정보만 남깁니다.
- 알고리즘 버전과 입력·결과 snapshot을 남겨 재현 가능하게 만듭니다.

### E. 데이터·도메인

데이터·도메인 파트는 KDRIs, 식약처 DB, 영양제 매칭 기준, 검수 상태 관리와 연결됩니다.

- KDRIs/DRI 기준값의 출처와 버전을 관리합니다.
- 식약처 건강기능식품 원료 DB나 영양제 CSV DB와 매칭합니다.
- 부족, 낮음, 적정, 과다, 위험 기준이 진단처럼 보이지 않도록 표현을 검토합니다.
- 운영 규칙으로 둔 가중치와 threshold는 추후 데이터로 검증해야 합니다.

## 쉬운 설명

이 자료는 Lemon Aid의 분석 기능을 세 부분으로 나누어 설명합니다.

첫째, **근거 있는 공식**입니다. BMI, BMR, KDRIs처럼 이미 알려진 기준이 있는 계산은 그대로 사용합니다. 이 부분은 "교과서 공식"에 가깝습니다.

둘째, **우리 프로젝트의 운영 규칙**입니다. 예를 들어 8,000보를 기준으로 성별, 나이, BMI에 따라 조금씩 보정하거나, 만성질환이 있으면 활동점수에 가중치를 주는 방식입니다. 이런 값은 공식처럼 단정하지 않고, 운영 중 검증해야 하는 규칙으로 분리합니다.

셋째, **안전한 사용자 노출 방식**입니다. 영양제 사진을 찍으면 OCR과 LLM이 성분을 읽어오지만, 그 결과를 바로 저장하지 않습니다. 사용자가 미리보기에서 확인하고 승인해야 최종 기록이 됩니다. 건강 문구도 "질병입니다"가 아니라 "권장량보다 낮을 수 있습니다", "전문가와 상담을 권장합니다"처럼 표현합니다.

## 발표용 핵심 문장

- "저희는 알고리즘을 공식과 운영 규칙으로 분리해서, 어떤 값이 과학적 근거이고 어떤 값이 프로젝트 검증 대상인지 구분했습니다."
- "OCR과 LLM 결과는 자동 확정하지 않고, 사용자 확인을 거친 preview 데이터로 처리했습니다."
- "Lemon Aid는 진단이나 처방 서비스가 아니라, 공식 기준과 사용자 입력을 바탕으로 참고 정보를 제공하는 건강관리 보조 도구입니다."
- "알고리즘 버전과 출처 버전을 함께 저장해 나중에 같은 입력에 대해 같은 결과를 재현하고 설명할 수 있게 했습니다."

## 정리 결론

이번 알고리즘 수정의 방향은 기능을 과장하는 것이 아니라 **설명 가능성**, **재현 가능성**, **안전성**을 강화하는 것입니다. 공식 근거가 있는 계산은 명확히 사용하고, 아직 운영 검증이 필요한 가중치와 threshold는 별도 규칙으로 관리합니다. 또한 OCR/LLM 결과는 사용자 승인 전까지 최종 건강 데이터로 보지 않기 때문에, Lemon Aid의 MVP 안전선과도 잘 맞습니다.
