# 음식 비전 브리핑 정리

> 기준 자료: `음식_비전_브리핑_05.12_smallpdf.pdf` 및 사용자가 공유한 5장 슬라이드  
> 연결 기준: `main/PROJECT_GUIDE.md`의 음식·식단 분석, AI 이미지 인식, 데이터/API, 컴플라이언스 파트

## 한 줄 요약

이번 자료의 핵심은 음식 사진을 AI가 완벽하게 판정하는 기능을 바로 만드는 것이 아니라, **MVP에서 음식 후보를 만들고 공식 영양 데이터와 연결해 100g 기준 영양정보를 제공하는 파이프라인**을 먼저 안정적으로 만드는 것입니다.

## 전체 내용 요약

음식 비전 파트는 사용자가 음식 사진이나 텍스트 식단을 입력했을 때, 앱이 다음 흐름으로 분석 결과를 만들 수 있게 하는 역할입니다.

1. 사용자가 음식 사진 또는 텍스트 식단을 입력합니다.
2. YOLOv8 mock detector와 Google Vision mock hint adapter가 음식 후보와 보조 힌트를 만듭니다.
3. Fusion engine이 YOLO 후보를 중심으로 GCV label, object hint, OCR 텍스트를 보강해 최종 음식 후보를 정리합니다.
4. RDA matcher가 음식 후보를 농진청/RDA 계열 영양 데이터와 연결합니다.
5. 음식별 100g 기준 영양 프로필을 만들고, 사용자가 g 또는 인분을 입력하면 선택적으로 실제 섭취량 기준 영양소를 다시 계산합니다.
6. 결과 응답에는 음식 후보, 100g 영양정보, 사용자 검토 필요 여부가 포함됩니다.

비전공자 관점에서 보면, 이 작업은 "AI가 사진만 보고 몇 g 먹었는지 정확히 맞히는 기능"이 아닙니다. 오히려 부정확한 추정을 줄이기 위해 기본 계산은 100g 기준으로 단순화하고, 사용자가 양을 입력한 경우에만 섭취량 기준 계산을 하는 구조입니다.

## 슬라이드별 쉬운 설명

### 1. CTO 판단 요약

첫 번째 슬라이드는 MVP에서 무엇을 할지와 하지 않을지를 정리합니다.

음식 사진만 보고 섭취량까지 정확히 추정하는 것은 기술 리스크가 큽니다. 그래서 MVP 기본 경로에서는 이미지 기반 섭취량 추정을 제외하고, 식별된 음식의 100g 기준 영양 프로필을 제공하는 방향으로 정리합니다.

이 결정의 효과는 세 가지입니다.

- 불확실한 computer vision 추정값 때문에 영양 결과가 크게 왜곡되는 위험을 줄입니다.
- AI Hub 전체 다운로드나 별도 학습 없이 mock 기반으로 API, Flutter, nutrition 도메인을 병렬 개발할 수 있습니다.
- 사용자는 음식의 기본 영양 성격을 빠르게 파악하고, 상세량을 직접 입력한 경우에만 계산 결과를 확장해서 볼 수 있습니다.

### 2. 현재 아키텍처와 데이터 흐름

두 번째 슬라이드는 음식 분석이 어떤 순서로 흘러가는지 보여줍니다.

흐름은 `Input -> Detection -> Fusion -> RDA Match -> Optional Scale`입니다. 사용자가 사진이나 텍스트를 넣으면, YOLOv8 mock과 GCV hint mock이 후보를 만들고, Fusion engine이 이를 합쳐 최종 음식 후보를 정리합니다. 이후 RDA matcher가 `food_code`와 100g 영양 프로필을 붙입니다.

중요한 구조 결정은 GCV를 음식명 확정 엔진으로 쓰지 않는다는 점입니다. 최종 후보 생성은 YOLO를 primary로 두고, GCV label이나 OCR은 alias 보강과 사용자 검토 플래그 판단에만 활용합니다.

### 3. 단계별 진행 현황

세 번째 슬라이드는 지금까지 끝난 일과 다음 일을 구분합니다.

A1부터 A2.5까지는 완료 상태입니다. DTO, 예외, seed data, mock fixture, YOLO mock detector, Google Vision mock hint adapter, fusion engine, portion estimator, pipeline 1차 조립이 끝났다는 뜻입니다.

A3는 다음 단계입니다. 여기서는 음식 후보를 농진청/RDA 영양 데이터와 연결하고, 100g 기준 영양 프로필을 만들고, 텍스트 입력과 의료 표현 안전성을 보강합니다.

### 4. 리스크와 CTO 의사결정 포인트

네 번째 슬라이드는 앞으로 문제가 될 수 있는 지점을 정리합니다.

대표 리스크는 이미지 기반 섭취량 추정 부정확, AI Hub 전체 데이터 용량 과다, 한국 음식 alias와 표기 변형, 의료 표현 오해, 실 GCV/YOLO 전환 시 인터페이스 흔들림입니다.

완화책은 명확합니다. 기본값은 100g 프로필로 전환하고, 사용자가 입력량을 제공할 때만 계산합니다. AI Hub는 전체 데이터가 아니라 Beta v1에서 작은 subset으로 시작합니다. 음식 alias는 `food_aliases.json` 같은 제품 자산으로 관리하고, 매칭 실패 시 review flag를 유지합니다. mock과 real 구현은 같은 ABC/Protocol을 따르게 고정합니다.

### 5. A3 마무리와 백엔드 API 연결

다섯 번째 슬라이드는 바로 다음 작업 목록입니다.

A3에서는 음식 후보를 농진청/RDA 데이터와 연결하고, 100g 기준 영양 프로필을 생성합니다. 사용자가 g 또는 인분을 입력하면 영양소를 다시 계산하는 선택 경로도 만듭니다. 텍스트 식단 입력을 처리하고, 의료 표현 금지 테스트도 추가합니다.

백엔드 API는 이미지와 텍스트 식단 인식을 받을 수 있어야 합니다. 응답에는 음식 후보, 100g 영양 정보, 검토 필요 여부가 포함됩니다. 실제 GCV/YOLO는 아직 mock을 유지하되, 나중에 real 구현으로 바꿔도 상위 pipeline 계약이 깨지지 않게 합니다.

## 팀 역할별 관련 범위

### C. AI 엔지니어

AI 엔지니어 파트는 YOLO/GCV 후보 생성, mock adapter, fusion 규칙, 의료 표현 검수와 연결됩니다.

- YOLOv8 mock detector는 음식 bbox, class, confidence 후보를 만듭니다.
- Google Vision mock hint adapter는 label, object hint, OCR 원문을 테스트용으로 제공합니다.
- GCV는 최종 판단자가 아니라 보조 힌트로만 사용합니다.
- 의료 표현이 들어간 설명이 사용자에게 노출되지 않도록 검수 흐름을 둡니다.

### D. 백엔드

백엔드 파트는 식단 분석 API, pipeline 조립, 응답 스키마와 연결됩니다.

- `/api/v1/meals/analyze`는 음식 사진 기반 분석 요청을 받습니다.
- `/api/v1/meals/manual`은 텍스트 식단 입력을 받습니다.
- MealPipeline은 detector, adapter, fusion, portion, matcher 결과를 조립해 `RecognizedMeal` 형태의 응답을 만듭니다.
- 응답에는 후보 음식, 100g 기준 영양정보, review flag가 포함되어야 합니다.

### E. 데이터·도메인

데이터·도메인 파트는 RDA matcher, 농진청 영양 데이터, 음식 alias 관리와 연결됩니다.

- 음식명과 alias를 `food_code`에 안정적으로 매칭합니다.
- `FoodNutritionProfile`에는 canonical name, category, base amount, default serving, nutrients per 100g, highlights, cautions, needs user review를 담습니다.
- 매칭 실패나 애매한 음식명은 확정하지 않고 사용자 검토가 필요한 상태로 남깁니다.
- 공식 데이터 출처와 라이선스를 관리합니다.

## 쉬운 설명

이 자료는 Lemon Aid의 음식 분석 기능을 세 부분으로 나누어 설명합니다.

첫째, **음식 후보 만들기**입니다. 사진이나 텍스트에서 "이 음식이 무엇일 가능성이 높은지" 후보를 만듭니다. 지금은 실제 모델이 아니라 mock으로 구현해 전체 흐름을 먼저 검증합니다.

둘째, **공식 영양 데이터 연결**입니다. 음식 후보를 농진청이나 식약처 계열 데이터와 연결해 100g 기준 영양성분을 가져옵니다. 이 부분이 사용자가 실제로 이해할 수 있는 영양정보의 기반입니다.

셋째, **안전한 계산과 표현**입니다. 사진만 보고 섭취량을 단정하지 않고, 기본값은 100g 기준으로 보여줍니다. 사용자가 직접 g이나 인분을 입력했을 때만 계산을 확장합니다. 건강 문구도 진단이나 치료처럼 보이지 않게 정보 제공형으로 제한합니다.

## 정리 결론

이번 음식 비전 브리핑의 방향은 복잡한 AI 모델 학습보다 **MVP에서 설명 가능하고 검증 가능한 음식 분석 흐름**을 만드는 것입니다. 음식 후보 생성은 mock으로 시작하되, 데이터 흐름과 API 계약은 실제 서비스 구조와 같게 맞춥니다. 영양 계산은 100g 기준을 기본으로 두고, 사용자가 양을 입력한 경우에만 재계산합니다. 이 방식은 기술 리스크를 줄이면서도 Lemon Aid의 음식·식단 분석 기능을 guide의 MVP 흐름과 일관되게 연결합니다.
