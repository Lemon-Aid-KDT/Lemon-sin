# 데이터베이스 보고서 정리

> 기준 자료: `lemon_aid_database_report.pdf`  
> 연결 기준: `main/PROJECT_GUIDE.md`의 백엔드, 데이터베이스, 데이터 모델, Agent, 보안 파트  
> 참고: PDF 원문에는 `DATABASE_GUIDE.md`도 기준 문서로 언급되지만, 현재 워크스페이스에서는 별도 파일을 확인하지 못했습니다.

## 한 줄 요약

이번 자료의 핵심은 Lemon Aid의 DB를 단순 저장소가 아니라, **로그인부터 프로필·식단·영양제·병원성 데이터·Agent 분석 결과까지 user_id 기준으로 연결하는 서비스의 기억 공간**으로 설계한다는 것입니다.

## 전체 내용 요약

PDF는 Lemon Aid의 데이터베이스와 데이터 호출 흐름을 다음처럼 정리합니다.

1. Flutter 앱은 DB에 직접 접근하지 않고, 모든 요청은 FastAPI 백엔드를 거칩니다.
2. FastAPI는 JWT로 사용자를 확인하고, 필요한 데이터만 PostgreSQL에서 조회합니다.
3. PostgreSQL은 사용자 정보, 프로필, 음식·영양제 기록, 분석 결과, Agent Memory를 장기 저장합니다.
4. Redis는 OCR 결과, 권장량 기준값, API 호출 제한처럼 빠르게 재사용할 데이터를 임시 저장합니다.
5. LLM Agent는 DB 전체를 직접 보지 않고, 백엔드 오케스트레이터가 정리해 준 최소 데이터만 받습니다.
6. 분석 결과는 자유 텍스트로 끝나지 않고, 정해진 스키마에 맞춰 PostgreSQL에 다시 저장됩니다.

비전공자 관점에서 보면, DB는 "앱의 기억장치"입니다. Lemon Aid에서는 이 기억장치가 단순히 로그인 정보를 저장하는 수준을 넘어, 사용자의 건강관리 흐름과 AI 개인화 분석을 이어 주는 중심 역할을 합니다.

## PDF 섹션별 쉬운 설명

### 1. 보고 목적과 설계 관점

첫 번째 내용은 DB가 왜 중요한지 설명합니다.

Lemon Aid는 단순 음식 분석 앱이 아니라, 사용자의 병원성 데이터, 복약 정보, 건강 데이터, 음식과 영양제 기록을 함께 다루는 앱입니다. 그래서 DB는 단순 저장소가 아니라, 사용자 데이터와 AI Agent 분석 흐름을 연결하는 중심 구조입니다.

쉽게 말하면 DB는 "사용자별 건강관리 기록을 안전하게 모아 두고, 다음 분석에서 다시 참고할 수 있게 하는 공간"입니다.

### 2. PostgreSQL 선정 근거

PDF는 PostgreSQL을 선택한 이유를 세 가지로 설명합니다.

첫째, `JSONB`입니다. 영양제마다 성분 개수가 다르고, 음식 분석 결과도 매번 구조가 다를 수 있습니다. JSONB를 쓰면 이런 유동적인 데이터를 한 테이블 안에서 유연하게 저장할 수 있습니다.

둘째, `TimescaleDB`입니다. 걸음수, 체중, 심박수처럼 시간 순서로 계속 쌓이는 데이터는 일반 테이블보다 시계열 처리에 맞는 구조가 필요합니다. PostgreSQL은 TimescaleDB 확장으로 이 흐름을 처리할 수 있습니다.

셋째, `RLS`입니다. 건강 정보는 민감하기 때문에 사용자는 자기 데이터만 볼 수 있어야 합니다. Row Level Security는 DB 차원에서 `user_id` 기준 접근 제한을 걸 수 있는 기능입니다.

### 3. 데이터베이스 연결 구성

전체 구조는 `Flutter App -> FastAPI Backend -> PostgreSQL / Redis -> 분석 알고리즘 / LLM Agent`입니다.

앱은 로그인, 사진 촬영, 프로필 입력, 챗봇 요청을 담당합니다. FastAPI 백엔드는 사용자가 누구인지 확인하고, 요청 형식을 검증하고, DB에 저장하거나 Agent를 호출합니다. PostgreSQL은 오래 보관해야 하는 정보를 저장하고, Redis는 반복 조회나 비용이 큰 결과를 임시로 저장합니다.

중요한 점은 Flutter가 DB에 직접 접근하지 않는다는 것입니다. 모든 접근은 FastAPI를 거쳐야 인증과 검증을 일관되게 적용할 수 있습니다.

### 4. 백엔드 파일 구조와 요청 처리 흐름

백엔드 구조는 요청이 들어온 뒤 어떤 파일을 지나 DB에 저장되는지 기준으로 이해하면 됩니다.

`main.py`는 FastAPI 앱 시작점이고, `config.py`는 환경변수를 읽습니다. `db/session.py`는 PostgreSQL 연결 통로입니다. `models/`는 DB 테이블 구조, `schemas/`는 요청과 응답 형식, `api/`는 실제 API 처리 위치입니다.

흐름은 `App 요청 -> api 라우터 -> schemas 검증 -> models 기준 -> db/session.py -> PostgreSQL 저장`입니다.

### 5. 로그인 및 프로필 저장 흐름

로그인은 이메일과 비밀번호를 받아 처리하지만, 비밀번호 원문은 저장하지 않습니다. 해시로 검증하고, 성공하면 JWT 토큰을 발급합니다. 이후 요청은 Bearer Token으로 사용자를 확인합니다.

프로필은 온보딩에서 입력한 나이, 성별, 키, 몸무게, 만성질환, 복약 정보를 저장합니다. 이 정보는 `profiles` 테이블에 들어가고, 개인화 Agent가 분석 기준으로 사용합니다.

쉽게 말하면 `users`는 로그인 계정 정보이고, `profiles`는 건강 분석에 필요한 사용자 기본 정보입니다.

### 6. LLM Agent 데이터 호출 구조

이 섹션의 핵심은 LLM Agent가 DB에 직접 접근하지 않는다는 점입니다.

LLM이 모든 사용자 데이터를 직접 조회하는 구조는 개인정보 관점에서 위험합니다. Lemon Aid에서는 FastAPI 오케스트레이터가 JWT로 사용자를 확인한 뒤, 프로필, 검사값, 복약 정보, Agent Memory 등 필요한 정보만 조회합니다. 그리고 Pydantic 형식으로 정리해 Agent에게 전달합니다.

즉 Agent는 "DB 관리자"가 아니라 "백엔드가 전달한 최소 정보로 분석하는 분석 보조자"입니다.

### 7. 사용자 이용 시 데이터 저장 흐름

영양제 사진 분석에서는 사진이 FastAPI로 전송되고, 백엔드는 이미지 해시로 Redis 캐시를 확인합니다. 캐시가 있으면 빠르게 반환하고, 없으면 OCR과 LLM 구조화로 제품명, 성분명, 함량, 단위를 추출합니다. 사용자가 수정·승인한 결과만 PostgreSQL에 저장합니다.

음식 기록 분석에서는 음식명과 섭취량을 추정한 뒤 식품영양 DB와 매칭합니다. 계산된 영양소는 개인화 기준과 결합되고, 평가 Agent가 부족·과다 분석과 식단 점수를 만듭니다.

Agent Memory는 모든 원본 데이터를 저장하지 않습니다. 다음 분석에 필요한 최근 검사값 요약, 만성질환 태그, 복약 정보, 최근 주의사항, 식단관리 점수 같은 핵심 요약만 저장합니다.

### 8. 주요 테이블과 보안 방향

전체 테이블은 `users`를 중심으로 `user_id` 기준으로 연결됩니다.

주요 테이블은 다음 흐름으로 보면 쉽습니다. `users`와 `profiles`는 사용자와 건강 기본 정보, `foods`와 `supplements`는 음식·영양제 마스터 데이터, `meals`와 `user_supplements`는 사용자의 실제 기록, `analysis_results`와 `daily_scores`는 AI 분석 결과와 점수, `agent_memory`와 `agent_runs`는 Agent 요약 기억과 호출 로그입니다.

보안 방향은 비밀번호 원문 저장 금지, JWT 기반 인증, user_id 기준 접근 제한, 민감정보 암호화 검토, 감사 로그, LLM 데이터 전달 최소화입니다.

PDF에서는 일부 설명에서 `diagnoses`라는 이름이 나올 수 있다고 주의하지만, `PROJECT_GUIDE.md` 기준으로는 AI 결과 테이블명을 `analysis_results`로 통일하는 것이 맞습니다.

### 9. 현재 구현 및 향후 확장 계획

현재는 PostgreSQL, Redis, Docker Compose 개발 환경과 users, profiles, refresh_tokens 같은 기본 테이블, 회원가입·로그인·JWT 인증, 프로필 API를 우선 잡는 흐름입니다.

이후 음식·영양제 마스터 테이블, OCR/LLM 분석 결과 저장 구조, Agent Memory, Agent Runs 로그, HealthKit 기반 시계열 데이터 저장 구조로 확장합니다.

### 10. CTO 설명용 핵심 문장과 예상 질문

마지막 섹션은 발표에서 설명할 포인트입니다.

핵심은 PostgreSQL을 쓰는 이유를 JSONB, TimescaleDB, RLS로 설명하고, Redis는 OCR/LLM 비용을 줄이는 캐시로 설명하는 것입니다. 또한 LLM이 개인정보를 모두 보는 것이 아니라, 백엔드가 필요한 정보만 골라 전달한다고 설명해야 합니다.

## 팀 역할별 관련 범위

### D. 백엔드

백엔드 파트는 이 보고서와 가장 직접적으로 연결됩니다.

- FastAPI API 라우터, 인증, JWT, refresh token 흐름을 구현합니다.
- `db/session.py`, `models/`, `schemas/`, `api/` 구조를 유지합니다.
- PostgreSQL과 Redis 연결, Docker Compose 개발 환경을 관리합니다.
- user_id 기준 접근 제한, RLS, 암호화, 감사 로그를 보안 요구사항으로 관리합니다.

### C. AI 엔지니어

AI 엔지니어 파트는 Agent 호출 구조와 연결됩니다.

- Agent가 DB에 직접 접근하지 않도록 오케스트레이터 입력 스키마를 따릅니다.
- OCR/LLM 결과는 Pydantic 스키마로 구조화합니다.
- Agent 결과는 자유 텍스트가 아니라 저장 가능한 구조로 반환해야 합니다.
- Agent Memory에는 전체 원본이 아니라 다음 분석에 필요한 요약만 남깁니다.

### E. 데이터·도메인

데이터·도메인 파트는 DB에 들어갈 기준 데이터와 민감정보 처리 기준을 관리합니다.

- KDRIs, 식약처, 농진청 데이터의 출처와 버전을 관리합니다.
- 음식·영양제 마스터 데이터와 alias 매칭 기준을 관리합니다.
- 병원성 데이터, 검사값, 복약 정보의 저장 범위와 동의 필요성을 검토합니다.
- 건강 데이터가 사용자별로 안전하게 분리되는지 컴플라이언스 관점에서 확인합니다.

## 쉬운 설명

이 자료는 Lemon Aid의 데이터 흐름을 세 부분으로 나누어 설명합니다.

둘째, **AI 분석에 필요한 데이터 전달**입니다. LLM Agent가 DB를 직접 뒤지는 것이 아니라, FastAPI가 필요한 정보만 꺼내 정리해서 Agent에게 전달합니다.

셋째, **결과 저장과 재활용**입니다. 분석 결과는 PostgreSQL에 저장되고, 다음 분석에 필요한 요약은 Agent Memory에 남습니다. Redis는 같은 사진이나 반복 조회되는 기준값을 빠르게 재사용하기 위한 캐시입니다.

| 민감정보는 어떻게 보호하나요? | user_id 기준 접근 제한, RLS, 암호화 검토, 감사 로그, LLM 전달 데이터 최소화가 기본 방향입니다. |

## 정리 결론

이번 데이터베이스 보고서의 방향은 Lemon Aid의 핵심 흐름을 데이터 관점에서 안전하게 연결하는 것입니다. 사용자는 로그인하고 프로필을 저장하며, 음식·영양제와 병원성 데이터를 기록합니다. 백엔드는 이 데이터를 검증하고 필요한 만큼만 Agent에게 전달합니다. Agent 분석 결과는 다시 PostgreSQL에 저장되고, 다음 분석에 필요한 요약은 Agent Memory로 재활용됩니다.

따라서 DB 설계의 핵심은 **PostgreSQL에 사용자별 장기 데이터를 안전하게 저장하고, Redis로 반복 비용을 줄이며, LLM Agent에는 필요한 최소 데이터만 전달하는 구조**입니다.
