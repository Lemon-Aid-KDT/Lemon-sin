# UX 1주차 브리핑 정리

> 기준 자료: `Lemon_Aid_UX_1주차_브리핑 (1).pdf`  
> 주제: 1주차 화면 뼈대와 타 팀 결과를 사용자 화면으로 받아내는 방식

## 한 줄 요약

UX 파트의 1주차 작업은 화면을 예쁘게 만드는 일이 아니라, **7개 핵심 화면의 뼈대와 AI·DB·음식 비전 결과를 사용자 화면으로 받아낼 규칙**을 정한 것입니다.

## 전체 내용 요약

이번 UX 브리핑은 다음 주에 AI, 백엔드, 음식 비전 파트의 결과를 화면으로 받을 수 있도록 사용자 흐름과 화면 기준을 먼저 정리한 자료입니다.

핵심은 "분석 결과가 있다"에서 끝나지 않는다는 점입니다. 사용자가 앱을 켰을 때 어디로 가야 하는지 알고, AI 결과를 믿을 수 있고, 틀린 부분을 바로 고칠 수 있고, 하루가 끝난 뒤 다시 들어올 이유를 느껴야 기술 결과가 서비스 가치로 이어집니다.

UX 파트는 1주차에 다음 네 가지를 정리했습니다.

1. 7개 기본 화면의 뼈대
2. 사용자가 앱을 쓰는 실제 맥락
3. 사용자가 멈추는 4가지 지점
4. AI·DB·음식 비전 결과를 화면으로 받아내는 공통 약속

## 1주차 화면 뼈대

1주차에는 서비스 전체를 구성하는 7개 기본 화면의 위치를 먼저 정리했습니다.

| 코드 | 화면 | 역할 |
|------|------|------|
| S-01 | Splash | 앱을 처음 켰을 때 보이는 시작 화면 |
| S-02 | Login | 기존 사용자의 빠른 로그인 |
| S-03 | Signup | 신규 사용자의 가입 |
| S-04 | Verify | 이메일 확인 또는 인증 흐름 |
| S-05 | Consent | 개인정보·건강정보·권한 안내 |
| S-06 | Onboarding | 첫 사용법과 기본 정보 입력 |
| S-07 | Dashboard | 기록, 분석 결과, 다음 행동을 보여주는 홈 |

이 화면들은 최종 디자인이 아니라, 다음 주에 각 팀의 결과를 받을 자리를 미리 정해 둔 구조입니다.

## 사용자 기준

Lemon Aid의 사용자는 단순히 "건강 앱을 쓰는 사람"이 아니라, 식사 직후나 약통 옆에서 짧은 시간 안에 기록을 남겨야 하는 사람입니다.

| 기준 | UX 해석 |
|------|---------|
| 주 사용자 | 30~50대 만성질환 관심층, 60대 이상도 함께 고려 |
| 사용 시간 | 한 번 켤 때 30초~2분, 하루 3~5회 |
| 사용 상황 | 식탁 앞, 약통 옆, 운동 직후 |
| 화면 원칙 | 큰 글씨, 큰 버튼, 높은 색 대비, 카메라 빠른 접근 |

따라서 UX는 긴 설명보다 짧은 안내, 작은 버튼보다 손가락 크기의 조작 영역, 복잡한 메뉴보다 바로 실행할 수 있는 카메라와 결과 카드를 우선합니다.

## 사용자가 멈추는 4가지 지점

| 지점 | 사용자 생각 | UX가 해결할 일 |
|------|-------------|----------------|
| 1. 처음 켤 때 | "앱을 켰는데 뭐부터 하지?" | 안내를 3단계 안으로 줄이고 카메라까지 한 번에 도달하게 함 |
| 2. 결과를 의심할 때 | "AI 결과를 믿어도 되나?" | 확신도, 출처, 고치기 경로를 함께 보여줌 |
| 3. 고치고 싶을 때 | "틀린 것 같은데 어떻게 고치지?" | 다른 화면으로 보내지 않고 같은 카드 안에서 바로 수정 |
| 4. 하루 마무리 | "오늘 잘했나? 내일 뭐 해야 하지?" | 점수만 보여주지 않고 다음 할 일 한 줄을 함께 제공 |

특히 2번과 3번은 UX 혼자 해결할 수 없습니다. AI 팀이 결과를 어떤 형태로 내려주는지, DB 팀이 빈 상태와 사용자 데이터를 어떻게 구분해 주는지가 함께 맞아야 합니다.

## 화면을 정한 6가지 기준

| 기준 | 내용 |
|------|------|
| 한눈에 무엇이 중요한지 | 한 화면에서 가장 중요한 것 하나만 진하게 보여줌 |
| 손가락이 닿는 거리 | 자주 쓰는 일은 세 번 안에, 카메라는 한 번에 도달 |
| 결과를 믿게 만드는 표시 | AI 결과에는 출처, 확신도, 시간 중 최소 두 가지 표시 |
| 누구나 읽고 누르게 | 글자는 작지 않게, 버튼은 충분히 크게, 배경과 글자는 명확히 구분 |
| 같은 의미는 같은 모양 | 같은 행동은 같은 위치, 색, 아이콘으로 통일 |
| 막혔을 때 빠져나갈 길 | 오류, 데이터 없음, 로딩에는 다음 행동 한 줄 제공 |

이 기준은 개인 취향으로 화면을 정하지 않기 위한 공통 판단 기준입니다. 다음 주 화면 작업에서도 AI 결과 카드, 음식 결과 카드, 빈 화면이 모두 이 기준을 통과해야 합니다.

## AI·알고리즘 결과를 화면으로 받는 방식

AI 팀의 핵심 원칙은 **공식 기반 계산, 운영 규칙 분리, 사용자 승인 전 임시 결과**입니다. UX는 이 원칙이 사용자에게 어렵게 보이지 않도록 화면으로 풀어야 합니다.

| AI·알고리즘 원칙 | UX 반영 방식 |
|------------------|--------------|
| OCR·LLM 결과는 자동 확정하지 않고 임시 결과로 둔다 | 결과 카드 상단에 "이대로 맞나요?"를 보여주고, `맞아요` / `고칠게요` 버튼 제공 |
| 사용자가 확인한 뒤 저장한다 | 고치기 선택 시 같은 카드 안에서 인라인 수정 |
| 확신도와 출처를 함께 보낸다 | 카드 우상단에 확신도 % 배지, 좌하단에 "출처: 공식 영양 DB" 라벨 표시 |
| 확신도가 낮은 결과는 검토가 필요하다 | 확신도 60% 미만은 주황 톤으로 표시해 자연스럽게 검토 유도 |
| 진단·처방 표현을 금지한다 | 모든 결과 화면 하단에 "건강 참고용이며 진단·처방이 아닙니다" 고정 푸터 표시 |

사용자에게는 AI가 완벽하다고 말하는 것보다, 어디까지 확실하고 어디를 고칠 수 있는지 정직하게 보여주는 것이 신뢰를 만듭니다.

## 데이터베이스 결과를 화면으로 받는 방식

DB 팀의 핵심 원칙은 **앱은 DB에 직접 접근하지 않고, 백엔드가 user_id 기준으로 필요한 데이터만 조회한다**는 것입니다. UX는 이 구조를 사용자가 안심할 수 있는 화면으로 표현해야 합니다.

| DB 원칙 | UX 반영 방식 |
|---------|--------------|
| 사용자별 데이터는 user_id 기준으로 분리한다 | 홈 상단에 "OO님의 기록"과 최근 업데이트 시각 표시 |
| 사용자는 자신의 데이터를 확인하고 관리할 수 있어야 한다 | 설정에 "내 데이터 보기 / 내보내기 / 삭제" 진입점 상시 제공 |
| AI Agent는 DB 전체를 보지 않는다 | 챗봇 하단에 "이번 대화에서 참고한 정보 보기" 링크 제공 |
| 데이터 없음의 이유는 구분해야 한다 | 신규, 동기화 실패, 권한 없음 화면을 각각 다르게 제공 |

빈 화면은 모두 같은 문구로 처리하면 안 됩니다. 처음 온 사용자는 "기록을 시작해요", 동기화 실패는 "다시 시도", 권한 없음은 "설정 열기"처럼 다음 행동이 달라야 합니다.

## 음식 비전 결과를 화면으로 받는 방식

음식 비전 파트의 핵심 결정은 **사진만으로 섭취량을 단정하지 않고, 100g 기준 영양 프로필을 기본으로 제공한 뒤 사용자가 양을 입력하면 재계산한다**는 것입니다.

| 음식 비전 원칙 | UX 반영 방식 |
|----------------|--------------|
| 기본 표시는 100g 기준 영양 프로필 | 결과 카드 큰 글씨로 "100g 기준" 표시 |
| 사용자가 양을 입력하면 재계산 | "한 그릇 / 반 그릇 / g 직접" 빠른 선택 제공 |
| 음식 후보가 여러 개일 수 있다 | 후보 3개를 가로 카드로 보여주고 탭으로 즉시 선택 |
| 매칭 실패 시 검토 필요 플래그를 둔다 | "이 음식이 맞나요? 직접 입력"으로 자연스럽게 연결 |
| 의료 표현은 정보 제공형으로 제한한다 | AI 결과 카드와 같은 하단 푸터를 재사용 |

이 방식은 기술적 한계를 숨기는 것이 아니라, 사용자가 자연스럽게 보정할 수 있게 만드는 UX 대응입니다.

## 세 팀에 걸친 공통 약속

AI, DB, 음식 비전 결과는 서로 다른 팀에서 오지만 화면에서는 같은 모양으로 보여야 합니다.

### 결과 카드 약속

모든 결과 카드는 다음 4요소를 갖습니다.

| 요소 | 설명 |
|------|------|
| 결과 큰 글씨 | 사용자가 가장 먼저 봐야 할 핵심 결과 |
| 확신 정도 | % 또는 색상 배지 |
| 출처 / 시간 | 공식 DB, 최근 업데이트 시각 등 작은 라벨 |
| 다음 행동 | 저장, 고치기, 더 보기 같은 한 줄 행동 |

### 빈 화면 약속

| 상태 | 화면 문구 방향 |
|------|----------------|
| 처음 온 사용자 | "기록을 시작해요" |
| 동기화 실패 | "다시 시도" |
| 권한 없음 | "설정 열기" |

### 안내 문구 약속

- "포만감", "권장량보다 낮음"처럼 일상어를 사용합니다.
- 진단·처방처럼 보이는 표현은 쓰지 않습니다.
- 모든 결과 화면 하단에 같은 참고용 푸터를 둡니다.
- 공통 형태는 `design_tokens_v2.dart` 한 곳에서 정의해 화면마다 달라지지 않게 합니다.

## 2주차 UX 작업

2주차에는 1주차에 정한 규칙 위에서 실제 결과 카드를 구체화합니다.

UX 파트가 진행할 일:

- 홈 화면: 점수 + 다음 할 일 카드 본 디자인
- 카메라 결과: 같은 카드 안에서 바로 고치기
- AI 결과 카드와 음식 결과 카드 디자인 토큰화
- 빈 화면 3종 신규/동기화 실패/권한 없음 통일
- "이번 대화에서 참고한 정보" 화면 시안

다른 팀에서 받아야 할 것:

| 대상 | 받아야 할 것 |
|------|--------------|
| AI 팀 | 결과에 함께 보낼 확신도, 출처, 고칠 칸, 낮은 확신도 문구 |
| DB 팀 | 데이터 없음 이유 분리표, AI에 전달되는 항목 명세 |
| 음식 비전 팀 | 음식 후보 응답 모양, top-N, 확신도, 후보 alias |
| 발주처 | 60대 이상 사용자 5분 인터뷰 일정, 우선 다룰 질환 명단 |

## 비전공자용 쉬운 설명

UX 파트는 버튼 색이나 화면 배치만 정하는 역할이 아닙니다. 사용자가 앱을 쓰면서 멈추는 순간을 찾고, 그 순간에 필요한 정보를 AI, DB, 음식 비전 팀에 요청하는 역할입니다.

예를 들어 AI가 음식 결과를 보냈다면 UX는 단순히 음식명만 보여주지 않습니다. 결과가 얼마나 확실한지, 어떤 출처를 썼는지, 사용자가 틀렸다고 느낄 때 어디를 누르면 되는지까지 함께 보여줘야 합니다.

또 음식 비전이 100g 기준 영양 정보를 보낸다면 UX는 이것을 딱딱한 표로 끝내지 않고, 사용자가 "한 그릇", "반 그릇", "g 직접 입력" 중 하나를 골라 바로 수정할 수 있게 만들어야 합니다.

## 발표 핵심 메시지

- "UX의 1주차 작업은 최종 화면 디자인이 아니라, 다른 팀의 결과를 받을 화면 구조와 규칙을 정한 것입니다."
- "사용자는 AI가 맞았는지, 틀렸을 때 어떻게 고치는지 알 수 있어야 계속 사용합니다."
- "AI, DB, 음식 비전 결과는 서로 다른 팀에서 오지만 화면에서는 같은 카드 구조로 보여야 합니다."
- "확신도, 출처, 다음 행동은 사용자가 AI 결과를 믿고 수정하게 만드는 핵심 장치입니다."
- "2주차에는 이 규칙을 바탕으로 홈, 카메라 결과, AI 결과 카드, 음식 결과 카드를 구체화합니다."

## 정리 결론

이번 UX 브리핑은 Lemon Aid의 화면을 예쁘게 만드는 자료가 아니라, **AI·DB·음식 비전 결과를 사용자가 이해하고 수정하고 다시 쓰게 만드는 화면 규칙**을 정의한 자료입니다. 핵심은 7개 기본 화면, 4가지 멈춤 지점, 6가지 화면 기준, 그리고 결과 카드·빈 화면·안내 문구의 공통 약속입니다.
