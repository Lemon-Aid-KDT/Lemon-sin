## 1. 작성 목적·범위·읽는 법

### 1.1 작성 목적
본 노트는 `yeong-Lemon-Aid/docs/` 전체(17개 메인 문서 + 30개 dev-guides = 47개 파일)에 대해 ① 갭 분석 ② 다관점 검토 ③ 경쟁 매핑 세 가지 렌즈로 브레인스토밍한 결과를 한 파일에 모아 둔다. **원본 문서는 수정하지 않으며**, 본 노트의 권장사항이 채택된 항목만 별도 PR로 원본에 반영한다.

### 1.2 범위
- 메인 문서: `docs/Nutrition-docs/01-project-overview.md` ~ `docs/Nutrition-docs/17-image-collection-consent-plan.md`
- Dev-guides: `docs/Nutrition-docs/dev-guides/00-setup-environment.md` ~ `docs/Nutrition-docs/dev-guides/29-final-deliverables-index.md`
- 본 노트(`docs/Nutrition-docs/18-*`)는 분석 산출물이며 컴플라이언스·기술 결정 문서가 아님.

### 1.3 읽는 법
- 모든 문서는 §3~§8 클러스터에 번호 순으로 정리됨.
- 우선순위 ★~★★★는 다음 기준:
  - ★ — 일정·안전성에 직접 영향 없음, 향후 폴리시 단계
  - ★★ — Phase 2 후반~Phase 3에 보강 권장
  - ★★★ — Phase 0~1에 즉시 손봐야 발주처 리뷰·CI·면책 리스크가 줄어드는 항목
- 액션 라벨: `[즉시]` Phase 0~1, `[Phase 2]` Phase 2 안, `[Phase 3+]` Phase 3 이후·발주처 리뷰 게이트 통과 시.
- 의료기기법·약사법·개인정보 관련 제안은 반드시 [docs/10](./10-compliance-checklist.md), [docs/15](./15-regulated-feature-feasibility-and-compliance-plan.md), [docs/17](./17-image-collection-consent-plan.md)와 교차 검증 후 반영한다.

## 3. 클러스터 A — 비전·전략(docs/01~04)

### 3.1 docs/01 — 프로젝트 개요 (Project Overview)

원본 분량: 214줄 | 갱신 우선순위: ★★★

#### 핵심 요약(3줄)
- 영양제 OCR + 식단·활동 통합으로 5종 출력(부족 영양소·권고량·체중 예측·운동·목적별 분석) 제공
- 회사 정의 산출식 3개(v1~v4·KDRIs·7-step)을 Python으로 구현, Flutter 모바일 프로토타입 개발
- 10주 학생 프로젝트로 4개 갭 영역(부족 영양소·목적별·식단·영양제)을 직접 정의·구현 필요

#### Lens 1. Gap
- [Gap-1] 5개 출력 중 "부족 영양소 추천"과 "목적별 분석(눈/간/피로)" 알고리즘이 **회사 가이드에 미정의** — 학생 팀이 의료 가이드라인 기반 독자 설계 필요 (§5 갭 분석 명시되나 구체 기준 없음)
- [Gap-2] "식단 입력 처리" 파이프라인 구체 설계 부재 — 텍스트 vs 이미지 인식 우선순위·정확도 목표 미지정 (§3 명세는 "텍스트 또는 이미지"만 명시)
- [Gap-3] 영양제 분석 OCR+LLM 파싱 세부 스펙 부재 — 성분명·용량·단위 추출 정확도 기준은 있으나(§9 성공 기준 85%) 테스트셋 정의·에러 처리 규칙 미상세화
- [Gap-4] **CLAUDE.md Rule 1 위반 위험** — "부족 영양소 추천" 함수명이 의도치 않게 의료 판단 표현으로 이름붙을 가능성. §3 Output ①은 명사만 있고 함수 네이밍 지침 없음.

#### Lens 2. Perspective
- 단순성(학생 10주): Phase 0~4 총 10주 계획이 타이트함 — Phase 3 "식단 + AI 고도화"(2주)에서 이미지 인식까지 요구되는데, OCR+LLM 파싱(Phase 2)에서 발생 가능한 정확도 이슈가 Phase 3을 압박할 리스크. 스코핑 재검토 필요 (§8 일정은 선형 배열되어 병렬화 여지 있음)
- 안전성(만성질환자·의료법): Rule 1 위반 패턴이 "부족 영양소 추천" 같은 core 기능 네이밍에서 쉽게 발생 — docstring·UI 텍스트·LLM 프롬프트 전부 검수 필요하나 §10 컴플라이언스 체크리스트 링크만 있고 본 문서에 구체 지침 부재
- 확장성(post-MVP·B2C): Year 1~3 로드맵이 03-project-intent.md에만 있고 본 개요 문서(§8)는 학생 프로젝트 10주만 표시 — 회사의 Year 2 정식 출시 시 데이터 형식·API 호환성 전제 조건이 지금 구현에 반영되는지 불명확

#### Lens 3. Competitive
- vs 필라이즈(국내): 필라이즈는 "영양제 조합 분석 + 검진 기록 통합"에 특화. 본 프로젝트는 "v4 만성질환 가중 + 의료기관 연계(향후)" 강점이나, 현 단계는 만성질환 정보 입력 의존 — LDB 연계 없이 필라이즈 수준의 사용자 신뢰도 획득 어려움 (의료기관 연계는 Year 2 목표)
- vs CalZen(해외): CalZen은 "사진→칼로리 3초" 극단적 마찰 최소화 표준화. 본 프로젝트 §9 성공 기준에 "모바일 앱 iOS+Android 양쪽 동작"만 있고 UI/UX 속도 기준 없음 — Phase 3 식단 이미지 인식에서 CalZen 수준의 응답성 달성 목표 부재 (추정)
- vs 마이약(국내): 마이약은 "처방약 스캔 + 영양제 상호작용"에 특화. 본 프로젝트는 영양제만 자동 스캔이고 처방약은 §6 Out of Scope → 마이약과의 협력·연계 경로가 기획에 없음

#### 추천 액션(상위 3개)
1. [즉시] **§5 갭 분석 재정의**: "부족 영양소 추천" 알고리즘을 구체화하는 별도 문서를 작성 — 한국영양학회 기준·임상 사례 3~5개 포함, Rule 1 함수명 지침 명시
2. [Phase 1 착수 전] **§8 일정 재검토**: Phase 2 OCR 완성도에 따른 Phase 3 이미지 인식 백플래닝 — 만약 OCR 정확도 85% 미달 시 선택지(글로벌 API 확대 vs 사진 입력 대신 텍스트 우선) 사전 결정
3. [Phase 4 전] **컴플라이언스 통합**: §10 링크가 아닌 본 문서에 "금지 표현·함수명·프롬프트 가이드" 최소 2절 추가 — 의료기관 발표 시 의료법 위반 가능성 사전 차단

### 3.2 docs/02 — 배경 및 문제 정의 (Background & Problem Statement)

원본 분량: 208줄 | 갱신 우선순위: ★★★

#### 핵심 요약(3줄)
- 770만 청구의신 사용자를 건강의신으로 피벗·필라이즈 75만 MAU 시장 선점 상황에서 차별화 필요
- 사용자 페인 포인트(영양제 정보 비대칭·중복 섭취·운동-체중 인과 불투명·만성질환자 맞춤 부재)를 해결할 가설 4개 제시
- RQ1~RQ4 (OCR 정확도·산출식 구현 가능성·사용성·법규 준수) 검증이 최종 성공 기준

#### Lens 1. Gap
- [Gap-1] §3.1 "페인 포인트" 테이블이 정성적 동기만 있고 정량 지표 부재 — "무엇을 얼마나 먹는지 파악 어려움"의 구체 문제(예: 평균 등록 시간·검색 실패율·사용자 이탈율) 미측정. 본 프로젝트의 SUCCESS가 이들 지표를 개선할지 불명확
- [Gap-2] §4 가설 4개 중 가설 3(모바일 환경 5종 통합)의 "SUS ≥70점" 목표는 있으나, 가설 1(OCR 정확도 85%)과 가설 4(LDB 연계 가능성)는 검증 방법이 "RQ1 리포트"·"LDB 인터페이스 설계"로만 명시 — 실패 시나리오 대응 정의 없음
- [Gap-3] §2.3 IPO와 프로젝트 관계가 "신성장 동력의 한 축" 수준으로만 표현 — 회사가 구체적으로 이 프로젝트 결과물을 Year 2 정식 출시에 어떻게 반영할 기대치(데이터 형식·API·보안 수준) 미정의
- [Gap-4] **CLAUDE.md Rule 2(Adapter 패턴) 사전 배치 부재** — §4 가설 1에서 "OCR+LLM 조합"이라 표현되나, 실제 구현 시 Google Vision·CLOVA·외부 LLM 선택·교체 전략이 없음 — 어댑터 설계 지침 누락

#### Lens 2. Perspective
- 단순성(학생): 4개 RQ가 모두 임상적·기술적 복잡도 높음 — RQ2(산출식 100% 구현)·RQ3(SUS 70점)·RQ4(법규 0건)은 모두 "완전함"을 요구하는데, 10주 학생 프로젝트에서 이들을 동시 달성 어려움. 우선순위 재정렬(예: RQ1 100% → RQ2 80% → RQ3·4 PoC) 검토 필요
- 안전성(의료법): 가설 4에서 "LDB를 통해 실제 진료·처방 데이터 연결 가능"이라 적혀 있으나, 건강정보 고속도로·개인정보보호법·의료법상 동의·암호화 요건이 §5에 학습 차원으로만 언급 — 정식 연계 시 규제 리스크(DTx 오인·데이터 유출)가 명시되지 않음
- 확장성(post-MVP): §5.2 "회사 관점" 항목이 "외부 시각 검증·갭 영역 보완·인재 발굴"로만 정의 — Year 2 정식 출시 시 학생 프로젝트 코드의 기술 부채(리팩토링·확장 난이도) 전제가 없음

#### Lens 3. Competitive
- vs 필라이즈: 필라이즈의 약점 중 "사진 스캔 자동화 약함·영양제 등록 방식이 검색 중심"이라 분석(가설 1). 본 프로젝트가 정면으로 이 약점을 타격할 수 있으나, OCR 정확도 85% 달성 불확실성이 높음 → 달성 실패 시 필라이즈와의 기능 경쟁에서 밀릴 가능성
- vs 마이약: "약-영양제 상호작용 검토"가 마이약의 강점이자 본 프로젝트 페르소나 B의 핵심 니즈. 문제는 본 프로젝트에서는 "LDB 연계 시(향후)"라고 명시 — Year 1에는 마이약과 기능 겹침 없음 (차별화 지연)
- vs 글로벌: CalZen·Nutrigenomix 같은 글로벌 강자가 한국 시장 진입 가능성 미언급 — 현재 기획은 국내 3개(필라·마이약·홈스픽)만 분석

#### 추천 액션(상위 3개)
1. [즉시] **RQ1~4 우선순위·실패 경로 정의**: RQ1(OCR 85%)을 "Tier 1 필수"로 격상, RQ2(산출식)를 "85% 동작" 완화, RQ3·4는 "PoC 수준" 명시 — Phase 1 종료 시 어느 RQ가 미달되면 Phase 2 일정 재조정 규칙 문서화
2. [Phase 1 착수 전] **Adapter 패턴 선행 설계**: OCR(Google Vision vs CLOVA vs 자체 모델) 검토 문서 작성 → backend/CLAUDE.md에 "외부 API 변경 시 구현 영향도" 가이드 추가
3. [Phase 2~3] **의료법 리스크 매핑**: LDB 연계 미포함 현 단계의 법규 준수 범위(약-영양제 상호작용은 "식약처 기능성" 범위 내)와 Year 2 정식 출시 시 추가 요구사항(ISMS-P·동의 UI) 분리 문서화

### 3.3 docs/03 — 기획 의도 (Project Intent & Differentiation Strategy)

원본 분량: 280줄 | 갱신 우선순위: ★★★

#### 핵심 요약(3줄)
- 필라이즈 카테고리 1위(75만) 추격이 아닌 "만성질환자·의료데이터" 빈자리 공략으로 차별화
- 페르소나 B(김건강 52세 고혈압·당뇨 전단계)·A(박직장 38세 예방) 이원화로 Year 1 만성질환 1순위·Year 2 예방 직장인 확장 전략
- 5가지 무기(LDB·v4 가중·770만 베이스·공식 데이터·5종 통합) + 3년 로드맵(Year 1 PoC·Year 2 정식·Year 3 DTx 검토) 제시

#### Lens 1. Gap
- [Gap-1] §2 "5가지 무기" 중 무기 ①(LDB) 현실화 경로 부재 — "향후 가능"이라 표현되나 Year 1 학생 프로젝트에서 LDB 인터페이스 설계까지 포함할지 명시 안 됨. 무기 ①이 최강 차별화인데 검증 일정 미정
- [Gap-2] 페르소나 B·A의 "성공의 모습"(§3.1·3.2)이 정성적 기대만 있고, 정량 KPI(1개월 사용률·재사용률·NPS) 없음 — 이 모습이 정말 달성 가능한지 베타 테스트 기준 불명확
- [Gap-3] §6 리스크 매핑 8개 중 R2(의료법 위반)·R5(OCR 정확도)·R7(개인정보 유출)이 "높음~매우 높음"인데, 완화 방안이 모두 "대책 일반론"만 제시 — 예: R5 "다중 OCR 엔진"이라 했으나 Google Vision + CLOVA 통합 구현 스펙이 없음 (비용·대기시간·정확도 trade-off 미분석)
