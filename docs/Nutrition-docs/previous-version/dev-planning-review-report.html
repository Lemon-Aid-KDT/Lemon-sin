1. 보고서 목적

이 문서는 Lemon Aid의 개발자 전달용 최종 기획서를 만들기 전, 현재 guide 문서에 정리된 기능 흐름과 기술 구현안을 팀이 검토하고 확정하기 위한 내부 보고서다.

제품 방향과 MVP 목표는 현재 확정된 것으로 본다. 다만 프론트엔드, 백엔드, 데이터 모델, 이미지 처리, AI Agent, 병원 데이터 연동 기반, 안전/컴플라이언스 구현 방식은 AI와 함께 초안화된 내용이므로 팀 검토와 확정이 필요하다.

이 문서는 개발자에게 바로 전달하는 최종 명세서가 아니라, 최종 명세서로 전환하기 전 팀이 결정해야 할 질문을 정리하는 문서다.

4. 이미지 처리 범위 검토

Lemon Aid의 핵심 입력은 사진이다. 따라서 이미지 처리는 단순 업로드 기능이 아니라, 분석 정확도, 개인정보, 저장 정책, 실패 대응, 시연 안정성에 직접 영향을 준다.

현재 초안에는 영양제 라벨·음식 사진 촬영, OCR, 이미지 저장, SHA-256 중복 감지, OCR 실패 시 재촬영 또는 수동 입력 흐름이 포함되어 있다. 다만 팀이 최종 확정해야 할 항목이 남아 있다.

  • 영양제 라벨 이미지는 MVP 핵심이므로 우선 처리 대상으로 보는 것이 자연스럽다.
  • 음식 사진은 서비스 차별화에는 중요하지만, 실제 이미지 인식까지 구현하면 난이도가 크게 올라갈 수 있다.
  • 처방전, 약 봉투, 검사표 이미지는 향후 가치가 크지만 개인정보와 의료정보 리스크가 높으므로 MVP에서 직접 처리할지 신중히 결정해야 한다.
  • 시연 안정성을 위해 실제 촬영 흐름과 샘플 이미지 흐름을 함께 준비할 필요가 있다.

팀 검토 포인트

  • 영양제 분석은 "OCR 텍스트 추출"보다 "표준 데이터와의 매칭"이 핵심이다.
  • 데이터 출처가 불명확하면 결과 신뢰도와 컴플라이언스 모두 약해진다.
  • 단위 변환과 성분 표준화는 건강 관련 결과에 직접 영향을 주므로, MVP에서도 최소 기준이 필요하다.

6. 향후 병원 데이터 연동 기반 검토

본 MVP는 실제 병원 데이터 연동을 수행하지 않는다. 다만 Lemon Aid의 장기 방향이 병원성 데이터 기반 개인화 Agent이므로, 향후 병원 데이터가 들어올 수 있는 구조와 시연용 대체 데이터는 검토해야 한다.

이 섹션의 목적은 실제 연동을 구현하자는 것이 아니라, 개발자용 최종 기획서에서 병원 데이터가 들어올 자리를 어떻게 표현할지 정하는 것이다.

팀 검토 포인트

  • 실제 병원 데이터를 가져오지 않으면서도, 향후 연동 가능성을 보여주는 구조가 필요하다.
  • 시연용 데이터는 실제 환자 데이터처럼 보이게 하면 안 된다.
  • 검사값, 질환 태그, 복약 정보는 민감정보이므로 저장 여부와 동의 흐름을 먼저 확정해야 한다.
  • "병원 기록을 기억하는 Agent"라는 메시지는 강점이지만, MVP에서는 "향후 연동을 가정한 참조 구조"로 표현하는 것이 안전하다.

7. AI Agent 및 기술 구현 질문

현재 초안은 4개 Agent 구조와 OCR, LLM Tool Use, Pydantic 검증, 의료 표현 필터를 포함한다. 이 구조는 서비스 방향을 잘 보여주지만 MVP 구현 난이도가 높을 수 있으므로 팀 확정이 필요하다.

  • 건강 관련 결과는 LLM 단독 판단보다 규칙, 출처, 검증 흐름을 함께 보여주는 것이 안전하다.
  • 기술적으로 멋진 구조보다 8주 MVP에서 안정적으로 시연 가능한 구조가 우선이다.
