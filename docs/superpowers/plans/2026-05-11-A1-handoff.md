## 코드 규칙 (반드시 준수)

- Python 3.13
- Pydantic v2 + `ConfigDict(frozen=True)` 권장 (DTO는 불변)
- `Field(..., ge=..., le=..., description=...)` 적극 사용
- **Python 파일 구조: 모듈 docstring → 빈 줄 → `from __future__ import annotations` → 빈 줄 → 다른 import**
  - 예:
    ```python
    """식단 인식 DTO."""

    from __future__ import annotations

    from typing import Literal

    from pydantic import BaseModel, ConfigDict, Field
    ```
- 모든 모듈: 1줄 docstring + 필요 시 Reference 주석
- 모든 함수·클래스: Google-style docstring + 타입 힌트
- `print()` 금지 (테스트에서도). 디버그 출력은 pytest의 capsys로 처리
- 의료 표현 금지: "진단", "처방", "치료", "보장" 단어를 식별자/문서/문자열 어디에도 쓰지 말 것
- ruff lint convention: `pydocstyle.convention = "google"` (pyproject.toml에 이미 설정됨)

### DTO 작성 가이드 (`backend/src/meal/base.py`)

dev-guide 16의 §1. base.py 명세를 따르되 다음 필드는 정확히 포함:

- `RecognizedMealItem`:
  - `name_ko: str` (Field로 min_length=1)
  - `food_code: str | None` (RDA 매칭 결과, 없으면 None)
  - `estimated_grams: float` (Field로 gt=0)
  - `estimated_amount: str` (예: "1공기")
  - `confidence: float` (Field로 ge=0, le=1)
  - `portion_confidence: float` (Field로 ge=0, le=1)
  - `needs_user_review: bool` (기본 False)
  - `sources: list[str]` (예: `["yolo_v8"]` 또는 `["yolo_v8", "google_vision"]`)
  - `alternatives: list[MealDetection]` (기본 빈 리스트)
- `RecognizedMeal`:
  - `meal_type: str` (Field pattern=`^(breakfast|lunch|dinner|snack)$`)
  - `items: list[RecognizedMealItem]`
  - `engine: str`
  - `raw_input: str` (기본 빈 문자열)

### 예외 (`backend/src/meal/exceptions.py`)

```python
"""식단 인식 예외 계층."""

from __future__ import annotations


class MealRecognitionError(Exception):
    """식단 인식 실패의 베이스 예외."""


class MealApiError(MealRecognitionError):
    """외부 API(YOLO/GCV) 호출 실패."""


class MealParseError(MealRecognitionError):
    """응답/입력 구조 파싱 실패."""
```

> 위 예시는 §"코드 규칙"의 파일 구조(docstring → 빈 줄 → `from __future__` → 빈 줄 → ...)를 그대로 따른다.

## 커밋 정책 (★ A1 한정 예외)

**평소 규칙**: 클로드는 git 상태 변경 명령을 실행하지 않고 사용자가 직접 커밋한다.

**A1 한정 예외**: 사용자가 명시적으로 허용 — 각 하위 작업이 검증되면 클로드가 직접 **로컬 커밋**까지 진행한다.

- 5개 커밋은 **권장 단위**이며, 단계 경계가 명확하면 그 단위로 커밋한다. 단계 안에서 추가 분할이 자연스러우면 더 작게 쪼개도 무방.
- 각 단계 검증(black/ruff/mypy/pytest) 통과 후 즉시 로컬 커밋.
- **push / merge / 원격 동기화는 절대 진행하지 않는다.**

### 허용·금지

- ✅ `git add ...` — A1 작업물에 한해 허용
- ✅ `git commit -m "..."` — 검증 통과 후 허용
- ✅ `git status`, `git log`, `git diff` — 자유롭게 허용
- ❌ `git push` — 절대 금지
- ❌ `git merge`, PR 생성 — 절대 금지
- ❌ A1 범위 밖 파일에 대한 커밋 — 절대 금지
- ❌ 다른 브랜치로 체크아웃, 브랜치 생성 — 절대 금지

### 권장 5 커밋 (각 단계별 검증 후 즉시 커밋)

각 커밋 메시지는 한국어, Conventional Commits 형식(`feat:`, `test:`, `chore:`).

1. **`feat(meal): DTO 4종과 예외 계층 추가`**

   - 검증: 파일 존재 + 기본 파싱 가능 확인

   - 검증: `git status`에서 `models/`, `ml/outputs/`, `*.pt` 등이 untracked로 표시되지 않음
