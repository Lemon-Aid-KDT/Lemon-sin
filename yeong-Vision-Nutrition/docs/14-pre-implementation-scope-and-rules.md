# 상세 구현 전 범위 및 규칙 설정안

## 1. 검토 대상

- 원본 파일: `dev-planning-review-report.html`
- 문서 성격: 개발자용 기획서를 작성하기 전에 팀이 먼저 결정해야 하는 기능 범위, 이미지 처리 범위, AI Agent 구조, 병원 데이터 연동 가능성, 개인정보 및 헬스케어 안전 규칙을 정리한 내부 검토 보고서
- 이번 반영 결과: 만성질환 환자를 1순위 사용자로 두고, 일반 사용자를 2순위 B2C 사용자로 확장할 수 있도록 제품 범위와 설정값을 고정함

## 2. 구현 전 반드시 고정해야 하는 범위

### 2.1 서비스 포지셔닝

Lemon Aid는 진단, 처방, 치료 서비스가 아니라 영양, 복약, 식단, 활동량 관리를 돕는 건강 관리 보조 서비스로 제한한다.

따라서 모든 화면, API 응답, AI 답변에서는 다음 표현을 피해야 한다.

- 진단합니다
- 처방합니다
- 치료됩니다
- 완치됩니다
- 질환입니다
- 약을 끊으세요
- 복용량을 바꾸세요
- 의사 대신 판단합니다

사용자에게 제공하는 답변은 “참고 정보”, “주의가 필요할 수 있음”, “전문가 상담 권장” 수준으로 제한한다.

### 2.2 사용자 우선순위

1순위 사용자는 만성질환 환자다.

- 대상: 고혈압, 당뇨, 고지혈증 등 지속적인 식단, 영양제, 복약 관리가 필요한 사용자
- 구현 기준: 안전성, 설명 가능성, 출처 표시, 사용자 확인 절차를 우선 적용
- 주요 기능: 복약 정보와 영양제 성분 주의, 질환별 과다 또는 부족 가능 영양 요소, 활동량과 체중 변화 참고 리포트

2순위 사용자는 일반 B2C 예방 관리 사용자다.

- 대상: 질환 진단 여부와 무관하게 영양, 식단, 체중, 활동 습관을 관리하려는 사용자
- 구현 기준: 진입 장벽이 낮은 온보딩, 건강 목표 기반 피드백, 일반적인 영양 균형 안내
- 주요 기능: 영양제 성분 확인, 식단 입력 기반 영양 균형, 활동량과 체중 변화 참고

기능 우선순위가 충돌하면 만성질환 환자의 안전성과 설명 가능성을 일반 사용자 편의성보다 먼저 적용한다.

### 2.3 MVP 포함 범위

이번 상세 구현의 기본 범위는 다음으로 제한한다.

- 영양제 라벨 이미지 OCR 분석
- 영양제 성분 매칭
- 텍스트 또는 수동 입력 기반 식단 분석
- 영양 균형 요약
- 5가지 핵심 결과 대시보드
- 체중 변화 예측 참고
- 활동 추천
- Ollama 기반 로컬 LLM 요약
- 규칙 기반 건강 안전 검증
- AI 분석 결과 저장 전 사용자 확인
- 건강 관련 주장에 대한 출처 표시
- 금지 의학 표현 필터

### 2.4 규제 게이트가 필요한 기능 범위

다음 기능은 구현 구조만 남기거나 데모 수준으로 제한한다.

- 음식 사진 분석: MVP에서는 데모 또는 선택 기능으로 제한
- 병원 데이터 연동: mock profile, mock FHIR, 사용자 수동 업로드, 공식 API 연동 후보 순서로 단계화
- LDB 또는 FHIR 기반 연동: 향후 확장을 위한 참고 구조와 adapter 인터페이스를 먼저 작성
- 처방전 이미지 분석: 처방 생성이나 변경이 아니라 OCR 추출, 사용자 확인, 복약 알림 설정으로 제한
- 검사표 이미지 분석: 진단이 아니라 검사명, 수치, 단위, 참고범위, 추세 표시로 제한
- 복용량 변경 안내: 직접 변경 지시가 아니라 위험 알림, 변경 요청 차단, 의료인 또는 약사 상담 연결로 제한

다음 기능은 MVP에서 제외한다.

- 진단 또는 치료 추천
- 약 복용량 변경 결정 또는 지시
- 건강 결과 보장형 예측
- 식별 가능한 건강정보를 외부 클라우드 LLM으로 전송하는 방식

즉, 팀이 구현하고 싶은 병원 데이터, 처방전, 검사표, 복용량 관련 기능은 완전 제외가 아니라 “규제 게이트가 있는 단계형 기능”으로 재분류한다. 다만 공개 B2C 버전에서는 의료 행위로 해석될 수 있는 결정 기능을 제공하지 않는다.

## 3. 이미지 처리 규칙

1차 이미지 처리 대상은 영양제 라벨이다.

- 입력 경로: 카메라, 갤러리, 샘플 이미지
- 최대 파일 크기: 10MB
- 최대 이미지 크기: 2048px
- 자동 회전 보정: 적용
- 흐림 감지: 적용
- OCR 이후 사용자 수정 및 확인: 필수

원본 이미지는 기본적으로 분석 후 삭제한다.

원본 이미지를 저장하려면 별도 동의가 필요하다. 처방전, 검사표, 병원 문서는 제한 입력 대상으로 분류한다.

- 처방전: 약품명, 용법, 용량, 기간, 처방일자 OCR 추출까지만 허용
- 검사표: 검사명, 수치, 단위, 참고범위, 검사일자 OCR 추출까지만 허용
- 병원 문서: 사용자가 직접 업로드한 요약 또는 공식 API로 받은 데이터만 허용
- 공통 규칙: 원본 이미지는 기본 저장하지 않고, OCR 결과는 사용자가 직접 확인한 뒤 저장
- 금지 규칙: 처방 생성, 복용량 변경, 약 중단, 질환 진단, 치료 추천은 제공하지 않음

## 4. AI 및 로컬 LLM 적용 규칙

LLM은 최종 의학 판단자가 아니라 설명 생성과 구조화 보조 역할로만 사용한다.

- 기본 실행 환경: Ollama 로컬 LLM
- 기본 실행 장비: MacBook Pro M4 Pro 24GB
- 우선 검토 모델군: Qwen3.5, Gemma 4
- 향후 확장 모델군: Qwen3.6, DeepSeek V4 Pro
- 모델 태그는 실제 구현 전 `ollama list`와 `ollama run`으로 로컬 실행 가능 여부를 확인해야 한다.

AI Agent는 처음부터 완전히 분리된 독립 시스템으로 만들기보다, MVP에서는 모듈화된 파이프라인으로 구현한다.

- 입력 분석 Agent: OCR, 입력값 정규화, 영양제 성분 추출
- 개인화 Agent: 사용자 프로필, 만성질환, 복약 정보 기반 규칙 적용
- 안전 평가 Agent: 금지 표현, 과잉 의학 표현, 위험 문구 검증
- 설명 Agent: 검증된 결과를 비전공자도 이해 가능한 문장으로 변환

영양 기준, 주의 성분, 위험 알림은 LLM 단독 판단이 아니라 출처 기반 데이터와 규칙 기반 검증으로 먼저 계산해야 한다.

## 5. 개인정보 및 안전 규칙

다음 정보는 민감 건강정보로 취급한다.

- 만성질환 정보
- 복약 정보
- 건강검진 또는 검사 요약 정보
- 활동량 및 체중 데이터
- AI 분석 결과 저장 데이터

MVP 공개 버전에서는 실제 병원 원본 데이터 직접 연동을 하지 않는다. 대신 mock FHIR, 사용자 수동 업로드, 공식 API 연동 후보를 분리하여 구현한다.

처방전과 검사표는 민감 건강정보로 취급한다. 분석에 필요한 최소 필드만 저장하고, 사용자가 요청하면 프로필, 입력값, OCR 결과, AI 분석 결과를 삭제할 수 있어야 한다.

응급 증상, 급격한 건강 악화, 심각한 부작용이 의심되는 경우에는 서비스 안내보다 119 또는 의료기관 상담을 우선 안내한다.

## 6. 구현 설정 파일

상세 구현에서 참조할 설정값은 다음 파일에 생성했다.

- `config/service-segmentation.settings.json`

이 설정 파일은 다음 내용을 포함한다.

- B2C 서비스 전략
- 만성질환 환자 우선 모드
- 일반 사용자 예방 관리 모드
- 사용자 세그먼트 라우팅 규칙
- MVP 포함, 제한, 제외 기능
- 이미지 입력 및 저장 정책
- Ollama 로컬 LLM 적용 정책
- AI Agent 역할 정의
- 의료 표현 제한 규칙
- 규제 게이트 기능별 구현 범위
- 개인정보 최소 수집 원칙
- 기능 플래그
- 상세 구현 전 점검 항목

## 7. 구현 시작 전 체크리스트

- 서비스 문구가 진단, 처방, 치료로 오해되지 않는지 확인한다.
- 만성질환 환자 모드와 일반 사용자 모드의 온보딩 필드를 분리한다.
- 민감 건강정보 수집 동의 항목을 세분화한다.
- 영양 기준과 주의 성분 판단은 출처 기반 데이터와 규칙으로 먼저 계산한다.
- LLM 답변은 안전 필터와 사용자 확인을 통과한 뒤 저장한다.
- 영양제 라벨 이미지를 1차 이미지 처리 대상으로 제한한다.
- 처방전, 검사표, 병원 문서는 별도 동의, OCR 추출, 사용자 확인, 원본 삭제 흐름을 거친다.
- 실제 병원 데이터 연동은 mock FHIR, 사용자 수동 업로드, 공식 API 연동 후보 순서로 단계화한다.
- 복용량 변경 요청은 직접 안내하지 않고 위험 알림과 전문가 상담 연결로 처리한다.
- Ollama 모델 태그는 로컬 실행 가능 여부를 확인한 뒤 환경 변수에 반영한다.
- 만성질환 환자 안전성과 설명 가능성을 일반 사용자 편의성보다 우선한다.
