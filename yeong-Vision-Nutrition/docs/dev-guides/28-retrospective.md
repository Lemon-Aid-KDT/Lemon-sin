## 📐 회고 진행 방법

### KPT 프레임워크

```
[Keep]   잘했던 것 — 계속 유지하고 싶은 행동
[Problem] 문제였던 것 — 개선이 필요한 부분
[Try]    시도해 볼 것 — 다음에 시도할 새 방법
```

### 진행 순서 (3시간 회고 미팅)

```
[10분] 도입 — 안전한 분위기 조성
[40분] Keep — 각자 포스트잇으로 작성 후 공유
[40분] Problem — 솔직하게, 사람 비난 X
[40분] Try — 구체적인 액션 아이템
[20분] 우선순위 정리 — Top 3 액션
[15분] 마무리 + 다음 단계
```

```markdown
- 시험 기간에 진행 늦어짐
- 일부 가이드 검증이 충분하지 않음
- → 액션: 다음 팀은 방학 활용 권장

### ❌ 모바일 디자인 일관성 미흡
- 화면별 색상·간격 약간 차이
- → 액션: 디자인 시스템 (Storybook 등) 도입

### ❌ 베타 테스터 모집 부족

- 페르소나 B 만성질환자 인터뷰 1회만
- → 액션: Phase 5에서 본격 사용자 리서치
```

```markdown
# 학습 사항 Top 10

## 1. 기획이 절반이다
- W1 4개 문서로 합의를 강하게 한 덕에 후반 흔들림 X
- 페르소나 B 차별화 메시지가 일관되어 발표에서 강력함
- **교훈**: 코딩 시작 전 충분한 합의

## 2. 컴플라이언스는 처음부터 코드에
- 의료법 표현 가이드를 자동 테스트로 강제
- "진단", "처방" 단어 0건 — 자동 검증
- **교훈**: 안전 요건은 사후 검증이 아닌 코드 레벨 제약

## 3. Adapter 패턴은 실용적이다
- 처음엔 "오버엔지니어링" 의심
- 실제로 OCR·LLM 백업 자동 폴백이 작동할 때 가치 입증
- **교훈**: 외부 의존성은 격리하라

## 4. Claude Code는 페어 프로그래머
- CLAUDE.md 컨벤션 → 일관된 코드
- dev-guides → AI가 명세 따라 코드 작성
- **교훈**: AI는 "도구" 가 아닌 "팀원"

## 5. 4-Tier 테스트의 가치
- 단위만으로는 통합 버그 발견 X
- E2E + 성능 테스트로 SLA 보장
- **교훈**: 테스트 피라미드는 실제 효과 있다

## 6. Hall 모델은 수식보다 실용
- 처음 논문 봤을 때 압도됐지만, 실제로는 일별 루프
- 단순화 버전도 7-step보다 정확
- **교훈**: 수식의 외관에 압도되지 말고 본질 보기

## 7. 사용자는 LLM이 100% 정확하다고 기대 X
- 식단 인식 결과를 사용자가 수정할 수 있는 UI 제공
- 오히려 신뢰도 ↑
- **교훈**: AI 한계를 정직히 인정하고 보완 UI 제공

## 8. 인수인계는 미리 시작
- W10 마지막에 몰아서 하지 말고 W8부터 점진
- 알려진 문제 솔직하게 — 발주처 신뢰
- **교훈**: 인계 자체를 프로젝트의 일부로

## 9. 의료 도메인은 무겁다
- "이 영양제가 좋아요" 한 줄 광고가 의료법 위반
- 식약처 인정 표시 외 효능 주장 금지
- **교훈**: 도메인 지식은 코드보다 중요

## 10. 시연은 준비가 90%
- 리허설 3회 + 백업 영상 + Q&A 30개
- 자신감은 준비에서 나옴
- **교훈**: 처음 시연하는 듯한 발표는 실패의 신호
```

```markdown
# 다음 팀에게 주는 조언

## 🎯 핵심 5가지

### 1. 발주처와 W1에 충분히 합의하라
"우리가 만들어야 할 것" 을 한 문장으로 정리할 수 없다면
코딩 시작 X. 차별화 메시지가 흔들리면 모든 게 흔들림.

### 2. CLAUDE.md를 먼저 만들어라
코딩 시작 전 컨벤션 + dev-guides 작성에 W1-W2 투자.
나중에 이게 없으면 코드 일관성 망함.

### 3. 의료 도메인의 무게를 가볍게 보지 말라
"진단", "처방" 단어 하나도 의료법 위반 가능.
컴플라이언스는 사후 검증이 아닌 코드 레벨 자동화.

### 4. 외부 API에 lock-in 되지 말라
Adapter 패턴 + 백업 폴백.
Claude API 가격이 10배 오르면? 다른 LLM으로 바꿀 수 있어야.

### 5. 인수인계도 프로젝트의 일부
마지막 1주에 몰아서 하지 말고, 처음부터 문서화.
"발주처 신규 개발자가 30분 내 빌드 가능한가" 가 검증 기준.

## 🛠 실용 팁

### 도구 선택
- **백엔드**: FastAPI + Pydantic v2 (Type-safe + 빠름)
- **모바일**: Flutter + Riverpod (코드 양 ↓)
- **테스트**: pytest + mocktail
- **외부 API**: Claude (한국어 강함) + OpenAI 백업

### 시간 관리
- W1-W2: 기획·합의 (코딩 X)
- W3-W4: 기본 알고리즘 + 환경
- W5-W7: 핵심 기능 (OCR + LLM + 모바일)
- W8-W9: 고급 기능 (Hall 모델 + 5종 출력 + 피드백)
- W10: 시연·인수인계

### 협업 도구
- GitHub Projects (칸반)
- Slack 또는 Discord (Discord가 학생에게 더 친숙)
- Figma (디자인 협업)
- Notion (문서)

### 안 됐을 때
- 솔직하게 발주처에 보고 — 늦을 수밖에 없는 이유
- 우선순위 조정 — Phase 4 일부 항목 다음 팀에 인계
- 약속한 모든 걸 못 끝내도 OK — 만든 건 완벽하게
```

## 💡 구현 팁

### 회고는 솔직해야

```
✅ "기획 부족했고 W3에 코드 다시 짰다"
✅ "팀 내 갈등이 있었지만 W6에 풀었다"

❌ "다 잘했다" (학습 X)
❌ 사람 비난 (블레임리스)
```

### 회고 자료는 기록·공유

- 발주처에 공유 → "학생들의 진정한 학습" 인식
- 학교에 공유 → 다음 팀의 자료
- 개인 포트폴리오 → 면접·이력서

### "성공" 의 정의

```
✅ 5종 출력 모두 동작
✅ 의료법 표현 가이드 위반 0건
✅ 발주처 만족도 (시연 후 피드백)
✅ 학생 팀 개인 성장
✅ 다음 팀이 이어받을 수 있는 인계 자료

→ 위 5가지 모두 만족하면 성공한 프로젝트
```
