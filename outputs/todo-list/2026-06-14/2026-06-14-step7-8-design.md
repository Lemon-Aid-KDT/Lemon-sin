## 2. 핵심 설계 결정 — learning enqueue post-commit background

### 문제
`maybe_store_learning_image_object(session=request_session)`가 mid-오케스트레이션에 request 세션을 commit → participate(lemon_app)서 GUC 드롭 → 후속 owner write(pipeline_metadata) WITH CHECK 거부. learning/pipeline.py는 DO-NOT-TOUCH(수정 불가). 게다가 learning store는 analysis 행이 commit된 후라야 FK 유효 — participate에선 의존성-exit까지 commit 안 됨.

### 권장 설계 (구현 시 최종 확정)
**Option A (권장) — 라우트 레벨 FastAPI BackgroundTasks**:
- `analyze_supplement_image`(서비스)는 **owner-data만** persist_scope로 write하고 learning store/annotation enqueue를 **호출하지 않음**. 대신 post-commit에 필요한 입력(user, analysis record id, image_bytes, image_metadata, learning_consents, ocr_result, settings, learning_object_store)을 결과에 담아 반환(또는 deferred 콜러블).
- **라우트**가 ① 오케스트레이터 호출 → ② 감사(record_sensitive_audit_event, ambient) → ③ `BackgroundTasks.add_task(...)`로 learning store+annotation enqueue 등록(**fresh session**: 자체 sessionmaker로 새 세션 열어 maybe_store_learning_image_object→annotation; learning/pipeline은 자기 commit 유지) → ④ 응답 반환(의존성-exit서 request tx commit).
- BackgroundTasks는 응답 후 실행 → analysis 행이 이미 durable → learning store FK 유효 + request tx와 독립.
- 장점: learning/pipeline.py 미접촉, GUC 드롭 없음, 비차단(응답 지연 회피). 단점: BackgroundTasks 신규 도입(테스트 패턴 필요), 실패 시 learning만 유실(기존도 best-effort).

**Option B — 서비스 내 별도 sessionmaker(즉시, 비-background)**: 오케스트레이터가 learning store용 새 세션을 직접 열어 호출. 문제: analysis 행이 아직 미commit(participate)이라 FK 깨짐 → **불가**. (post-commit이 본질.)

→ **Option A 채택**. 구현 시 BackgroundTasks가 의존성-exit commit 이후 실행됨을 통합 테스트로 확정(또는 라우트가 명시적으로 fresh-session post-commit 실행).

## 3. persist_scope 전환 계획 (owner-data write)
- supplement_intake.py:364 `async with session.begin()` → `async with persist_scope(session)` (refresh는 scope-exit flush 의존, meal 패턴). ⚠️ 이건 Step6 ③에서 이연된 것 — 여기서 오케스트레이터와 원자적으로.
- supplement_image_analysis.py: `_store_supplement_facts_guidance`(603)·`_store_pipeline_metadata`(713)·`_store_preview_warnings`(1742) 본문의 `session.add/dirty + commit + refresh` → `async with persist_scope(session): ... ` (commit 제거, refresh는 scope 뒤 또는 flush 후). 각 함수가 단일 record 갱신이라 meal confirm 패턴 적용.
- `_enqueue_supplement_section_annotation_task`(756)는 **post-commit으로 이동**(persist_scope 아님) — 서비스에서 제거하고 라우트 background로.
