# Ollama to A100 Remote GPU Integration Plan

Date: 2026-05-19
Target project: Lemon-Aid / Lemon Healthcare
Target path: `/Users/yeong/99_me/00_github/03_lemon_healthcare/yeong-Lemon-Aid`

## 1. 목적

VS Code 원격 터널 또는 Remote SSH로 접근 가능한 교육기관 A100 서버에서 Ollama 모델을 실행하고, Lemon-Aid 백엔드의 기존 Ollama 기반 LLM 파이프라인에 안전하게 연결한다.

핵심 목표는 다음과 같다.

- 로컬 Mac의 추론 한계를 A100 GPU로 보완한다.
- 기존 Pydantic structured output 검증과 의료/개인정보 guardrail을 유지한다.
- 원격 추론 서버 사용 여부를 코드와 운영 정책에서 명시적으로 구분한다.
- 실사용자 데이터 적용 전에는 비식별 fixture와 smoke test로만 검증한다.

## 2. 공식 문서 확인 근거

이 계획은 아래 공식 문서를 기준으로 작성한다.

- Ollama Linux install: https://docs.ollama.com/linux
- Ollama API base URL: https://docs.ollama.com/api/introduction
- Ollama Chat API: https://docs.ollama.com/api/chat
- Ollama structured outputs: https://docs.ollama.com/capabilities/structured-outputs
- Ollama FAQ (`OLLAMA_HOST`, `OLLAMA_NO_CLOUD`): https://docs.ollama.com/faq
- Ollama GPU support: https://docs.ollama.com/gpu
- Ollama Docker GPU setup: https://docs.ollama.com/docker
- NVIDIA Container Toolkit: https://docs.nvidia.com/datacenter/cloud-native/container-toolkit/1.18.2/install-guide.html
- VS Code Remote SSH port forwarding: https://code.visualstudio.com/docs/remote/ssh

확인된 공식 내용:

- Ollama API는 기본적으로 `http://localhost:11434/api`에서 제공된다.
- Chat API는 `POST /api/chat`를 사용하며, `format` 필드는 `json` 또는 JSON Schema object를 받을 수 있다.
- Structured output은 Pydantic `model_json_schema()`를 `format`에 전달하고 응답을 다시 `model_validate_json()`으로 검증하는 방식이 권장된다.
- Ollama는 기본적으로 `127.0.0.1:11434`에 바인딩하며, 네트워크 노출은 `OLLAMA_HOST` 환경변수로 변경한다.
- Ollama local-only 모드는 `OLLAMA_NO_CLOUD=1`로 cloud 기능을 비활성화할 수 있다.
- Ollama 공식 GPU support 표에서 A100은 NVIDIA compute capability 8.0 계열로 포함된다.
- Docker로 NVIDIA GPU를 쓰려면 NVIDIA Container Toolkit 설치와 Docker runtime 설정이 필요하다.
- VS Code Remote SSH는 포트 포워딩 또는 SSH config의 `LocalForward`로 원격 포트를 로컬 포트에 연결할 수 있다.

## 3. 현재 프로젝트 상태 요약

현재 Lemon-Aid 백엔드는 이미 Ollama 기반 LLM 구조를 갖고 있다.

- `backend/Nutrition-backend/src/config.py`
  - `LLM_PROVIDER=ollama`
  - `OLLAMA_BASE_URL=http://127.0.0.1:11434`
  - `OLLAMA_MODEL=qwen3.5:9b`
  - `ALLOW_EXTERNAL_LLM=false`
- `backend/Nutrition-backend/src/llm/ollama.py`
  - `ALLOW_EXTERNAL_LLM=false`일 때 `localhost`, `127.0.0.1`, `::1`만 허용한다.
  - `POST /api/chat` payload에 `format=<Pydantic JSON Schema>`를 넣는다.
  - 응답의 `message.content`를 다시 Pydantic으로 검증한다.
- `backend/Nutrition-backend/src/services/supplement_parser.py`
  - OCR text를 정규화하고 HMAC hash만 저장한다.
  - raw OCR text와 raw model response를 저장하지 않는 preview-first 구조다.
- `backend/Nutrition-backend/src/services/readiness.py`
  - 현재 `/ready`는 Ollama API를 직접 호출하지 않고 설정 중심으로만 상태를 노출한다.

따라서 A100 적용은 LLM 호출부를 새로 만드는 작업이 아니라, `OLLAMA_BASE_URL` 연결 정책과 원격 runtime guardrail을 명확히 하는 작업이다.

## 4. 권장 아키텍처

1차 개발 검증은 SSH/VS Code port forwarding 방식을 사용한다.

```text
Lemon backend
  -> http://127.0.0.1:11435
  -> SSH or VS Code LocalForward
  -> A100 server 127.0.0.1:11434
  -> Ollama model
```

이 구조의 장점:

- A100 서버의 Ollama를 공용 네트워크에 직접 열지 않는다.
- 백엔드 입장에서는 loopback endpoint로 접근하므로 기존 local-only guardrail과 충돌이 작다.
- 실험 중 인증/HTTPS gateway를 만들지 않아도 안전한 smoke test가 가능하다.

주의할 점:

- 코드상으로는 `127.0.0.1`처럼 보여도 실제 inference는 원격 A100에서 수행된다.
- 따라서 실사용자 데이터 적용 전에는 `remote-private LLM`이라는 별도 정책과 audit marker를 반드시 추가해야 한다.
- `ALLOW_EXTERNAL_LLM=true`로 우회하면 외부 상용 LLM 허용과 의미가 섞이므로 권장하지 않는다.

## 5. 단계별 구현 플랜

### Phase 0. 기관 정책 및 데이터 경계 확인

목표: A100 서버가 민감 건강정보 처리에 적합한지 먼저 확인한다.

체크리스트:

- 교육기관 A100 서버에서 사용자 OCR text, 건강정보, 식단/영양제 데이터 처리가 허용되는지 확인한다.
- 서버가 공동 계정인지, 개인 계정인지, 작업 디렉터리와 모델 cache가 다른 사용자에게 노출될 수 있는지 확인한다.
- 초기 검증 범위는 비식별 fixture, 샘플 OCR text, dummy request로 제한한다.
- 처방전/검사표 OCR intake는 1차 적용 대상에서 제외한다.

명시할 정책:

- raw image와 raw OCR text를 원격 서버에 저장하지 않는다.
- Ollama prompt와 response를 서버 로그에 남기지 않는다.
- 실사용자 데이터는 remote-private LLM gate가 구현된 뒤에만 사용한다.

### Phase 1. A100 서버에서 Ollama runtime 준비

원격 A100 터미널에서 확인할 항목:

```bash
nvidia-smi
```

Ollama 설치:

```bash
curl -fsSL https://ollama.com/install.sh | sh
```

Ollama local-only 실행:

```bash
export OLLAMA_NO_CLOUD=1
ollama serve
```

systemd service를 사용할 수 있다면 `OLLAMA_NO_CLOUD=1`을 service override에 넣는다.

모델 설치:

```bash
ollama pull <model-tag>
ollama list
```

현재 문서의 기본값은 `qwen3.5:9b`이지만, 실제 A100 서버에서 `ollama pull`과 `/api/tags`가 성공하기 전까지 모델 태그를 확정하면 안 된다.

서버 내부 smoke test:

```bash
curl http://127.0.0.1:11434/api/tags
curl http://127.0.0.1:11434/api/chat -d '{
  "model": "<model-tag>",
  "messages": [{"role": "user", "content": "Return JSON only: {\"ok\": true}"}],
  "stream": false,
  "format": "json",
  "options": {"temperature": 0}
}'
ollama ps
```

검증 기준:

- `/api/tags`가 모델 목록을 반환한다.
- `/api/chat`가 JSON 형태 응답을 반환한다.
- `ollama ps`에서 모델이 GPU에 올라갔는지 확인한다.

### Phase 2. VS Code 또는 SSH port forwarding 연결

권장 로컬 포트:

- 원격 Ollama: `127.0.0.1:11434`
- 로컬 tunnel endpoint: `127.0.0.1:11435`

SSH config 예시:

```sshconfig
Host a100-ollama
    HostName <a100-host>
    User <user>
    LocalForward 127.0.0.1:11435 127.0.0.1:11434
```

또는 VS Code Remote SSH의 Ports panel에서 원격 `11434`를 로컬 `11435`로 forward한다.

로컬 Mac에서 확인:

```bash
curl http://127.0.0.1:11435/api/tags
```

성공 후 Lemon-Aid backend `.env` 개발 설정:

```env
LLM_PROVIDER=ollama
OLLAMA_BASE_URL=http://127.0.0.1:11435
OLLAMA_MODEL=<model-tag>
OLLAMA_TIMEOUT_SEC=90
OLLAMA_TEMPERATURE=0
ALLOW_EXTERNAL_LLM=false
```

이 단계에서는 반드시 비식별 테스트 데이터만 사용한다.

### Phase 3. 프로젝트 설정 정책 확장

목표: 원격 A100을 단순 external LLM으로 처리하지 않고, private GPU runtime으로 구분한다.

추가할 설정 제안:

```env
LLM_RUNTIME_LOCATION=local_loopback
ALLOW_REMOTE_PRIVATE_LLM=false
REMOTE_PRIVATE_LLM_HOST_ALLOWLIST=[]
REMOTE_PRIVATE_LLM_REQUIRE_TUNNEL=true
```

허용 값 제안:

- `LLM_RUNTIME_LOCATION=local_loopback`
  - 기존 기본값.
  - `localhost`, `127.0.0.1`, `::1`만 허용.
- `LLM_RUNTIME_LOCATION=ssh_tunnel_remote_gpu`
  - 개발/검증 전용.
  - endpoint는 local loopback이어도 audit상 원격 GPU로 기록.
  - 실사용자 데이터는 기본 차단.
- `LLM_RUNTIME_LOCATION=private_gpu_gateway`
  - 향후 운영 후보.
  - HTTPS, auth, host allowlist, audit, rate limit이 모두 필요.

`ALLOW_EXTERNAL_LLM=true`는 외부 상용 API 허용과 혼동되므로 remote-private GPU 허용 용도로 쓰지 않는다.

### Phase 4. 코드 변경 범위

주요 변경 파일:

- `backend/Nutrition-backend/src/config.py`
  - `llm_runtime_location`
  - `allow_remote_private_llm`
  - `remote_private_llm_host_allowlist`
  - production/staging validation 추가
- `backend/Nutrition-backend/src/llm/ollama.py`
  - `_validate_local_ollama_settings()`를 `validate_ollama_runtime_policy()`로 확장
  - local loopback, SSH tunnel remote GPU, private gateway 정책 분리
- `backend/Nutrition-backend/src/services/readiness.py`
  - `runtime_location`, `remote_private_llm_allowed`, `live_probe=not_run` 같은 sanitized details 추가
  - raw URL, prompt, response, OCR text는 노출 금지
- `backend/Nutrition-backend/tests/unit/test_config.py`
  - 정책별 설정 validation test 추가
- `backend/Nutrition-backend/tests/unit/llm/test_ollama_parser.py`
  - remote-private policy 허용/차단 test 추가
- `backend/.env.example`
  - A100 SSH tunnel 개발 예시 추가

중요 구현 원칙:

- prompt, OCR 원문, raw model response를 로그에 남기지 않는다.
- `format=<schema>`와 `model_validate_json()` 검증은 유지한다.
- LLM이 `nutrient_code`, diagnosis, treatment, dosage-change guidance를 만들 수 없게 한다.
- timeout과 rate limit을 명시한다.
- 실패 시 수동 입력 fallback 또는 기존 deterministic preview로 돌아간다.

### Phase 5. 실험용 smoke checker 추가

권장 스크립트:

```text
backend/scripts/probe_ollama_runtime.py
```

기능:

- `/api/tags` 확인
- configured model 존재 여부 확인
- `POST /api/chat` structured output smoke test
- 응답 JSON schema 검증
- latency 측정
- raw prompt/output 저장 금지

출력 예시:

```text
ollama_base_url=http://127.0.0.1:11435
runtime_location=ssh_tunnel_remote_gpu
model=<model-tag>
tags_status=200
model_present=true
structured_output_valid=true
latency_ms=...
gpu_processor_observed=manual_check_required
```

성능 숫자는 실제 측정 전까지 문서에 단정해서 쓰지 않는다.

### Phase 6. 비식별 fixture 평가

평가 대상:

- supplement OCR parser
- supplement recommendation explanation
- vision assist는 별도 gate 승인 전까지 제외

평가 지표:

- schema validation pass rate
- hallucinated field count
- forbidden term detection count
- empty/null handling correctness
- p50/p95 latency
- timeout/fallback 발생률

금지 사항:

- fake accuracy metric 생성 금지
- dummy data를 실제 결과처럼 표기 금지
- 모델별 우열을 측정 없이 단정 금지

## 7. 적용 우선순위

1. A100에서 `nvidia-smi`, Ollama install, `/api/tags`, `/api/chat` smoke test 성공.
2. SSH/VS Code port forwarding으로 로컬 `127.0.0.1:11435` 연결.
3. 비식별 fixture로 현재 parser flow smoke test.
4. remote-private LLM 설정과 validation test 추가.
5. readiness와 probe script로 팀원이 같은 절차를 반복할 수 있게 정리.
6. benchmark 결과가 쌓인 뒤 모델 태그와 timeout/concurrency 정책 확정.

## 8. 권장 커밋 메시지

```text
feat(llm): add remote-private Ollama runtime policy

Why:
An A100-hosted Ollama runtime can improve local inference capacity, but the app
must distinguish SSH-tunneled private GPU inference from true local loopback and
from external LLM providers before sensitive OCR text is sent.

The policy keeps structured output validation, privacy-preserving OCR handling,
and production fail-closed behavior intact while allowing controlled development
smoke tests against a remote private GPU.
```

## 9. 최종 판단

원격 A100 Ollama 적용은 기술적으로 가능하며, 현재 프로젝트의 Ollama structured-output 구조와 잘 맞는다. 하지만 바로 `OLLAMA_BASE_URL`만 원격 주소로 바꾸는 방식은 개인정보/의료 guardrail을 흐리게 만든다.

따라서 1차는 SSH tunnel 기반 개발 검증으로 진행하고, 2차에서 `remote-private LLM` 정책을 코드에 명시한 뒤, 3차에서 private gateway와 운영 보안 요건을 검토하는 순서가 가장 안전하다.
