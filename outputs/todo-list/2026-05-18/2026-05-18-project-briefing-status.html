Chapter 4.

사용자는 어떻게 경험하나

1 사용자가 영양제 라벨을 찍거나 이전에 찍은 이미지를 올립니다.
2 사진의 숨은 개인정보와 비정상 이미지를 먼저 점검합니다.
3 YOLO가 브랜드, 성분표, 섭취 방법, 주의사항 영역을 표시합니다.
4 OCR이 각 영역의 글자를 읽고, AI와 규칙이 표처럼 정리합니다.
5 사용자가 내용을 확인해야 최종 저장됩니다. 자동 확정하지 않습니다.

Chapter 4.

전체 파이프라인 설계

ML / 백엔드
영양제: PaddleOCR 중심 글자 인식, YOLO bbox 영역 보조, LabelLayout 카테고리 정리
분석 API: Ollama image-to-text, text-to-text 결과를 FastAPI로 전달
목표: 브랜드, 성분, 종류, 섭취 방법, 주의사항을 검수 가능한 구조로 변환
프론트엔드 / 기획
UX / UI: 카메라 촬영, 갤러리 업로드, 분석 중 화면, 결과 확인 화면 구현
기획: 사용자 시나리오, 안전 문구, 상담 안내, 저장 전 확인 흐름 설계
핵심: 사용자가 AI 결과를 그대로 믿지 않고 직접 확인하도록 만드는 화면
안전 / DB
개인정보: EXIF/GPS 제거, 이미지 크기 제한, 동의 기록, 감사 로그 관리
DB 연동: 사용자 정보, 섭취 중인 항목, 알레르기·주의 조건과 비교
경계: 진단·처방이 아니라 과다·중복·위험 가능성 경고와 약사·의사 상담 안내

Chapter 4.

사용 기술 스택

개인정보를 지키면서 빠르게 검증하기 위해
로컬 처리와 단계별 검수 구조를 선택했습니다

로컬 우선 처리

PaddleOCR
라벨 글자를 먼저 읽습니다.

YOLO
읽어야 할 영역을 bbox로 잡습니다.

Ollama
로컬 LLM으로 요약·경고 문구를 만듭니다.

FastAPI·DB
앱, 사용자 정보, 저장 흐름을 연결합니다.

▶ OCR ROI LLM API

외부 서비스만 쓰는 방식

Google Vision·CLOVA
품질 비교와 보조 수단으로 유용합니다.

사진이 외부로 전송될 수 있어
명시 동의가 필요합니다.

건강 데이터에서는
보안 기준과 설명 책임이 더 중요합니다.

따라서 기본값은 로컬,
외부 OCR은 선택 경로로 둡니다.
