# 🔍 DrawingLLM — AI 기반 산업 도면 검색 및 분류 시스템

> **Engineering Drawing Retrieval & Classification System powered by Open-Source LLM**

---

## 1. 문제 정의 (Problem Statement)

### 1.1 현황 분석

자동차 산업을 비롯한 제조업에서는 하나의 완성차에 **약 2만~3만 개의 부품**이 사용되며, 각 부품에는 최소 1개 이상의 기술 도면이 존재한다. 대형 OEM(완성차 제조사)과 Tier 1~3 협력사를 포함하면 하나의 차종에 관련된 도면 수는 **수십만 건 이상**에 달한다.

| 구분 | 추정 규모 |
|------|-----------|
| 완성차 1개 차종 부품 수 | 약 20,000 ~ 30,000개 |
| 차종당 관련 도면 수 | 약 50,000 ~ 100,000건 |
| OEM 전체 관리 도면 수 | 수백만 건 이상 |
| 자회사/협력사 포함 시 | 기하급수적 증가 |

### 1.2 핵심 문제 (Pain Points)

#### 🔴 문제 1: 도면 검색의 비효율성
- 엔지니어가 필요한 도면을 찾기 위해 **평균 30분~2시간** 소요
- 부품번호, 도면번호 체계가 회사/부서마다 상이하여 통합 검색 불가
- 키워드 기반 검색으로는 도면의 시각적 내용(형상, 구조)을 검색할 수 없음

#### 🔴 문제 2: 도면 분류 및 관리의 한계
- 신규 입사자나 타 부서 엔지니어는 도면 분류 체계 이해에 **수개월** 소요
- 도면 메타데이터(부품명, 재질, 공차 등)가 누락되거나 비표준화된 경우 다수
- 수작업 분류로 인한 오분류 및 중복 도면 발생

#### 🔴 문제 3: 지식 단절
- 숙련 엔지니어 퇴직 시 도면 관련 암묵적 지식(어떤 도면이 어디에 쓰이는지) 소실
- 자회사 간 도면 공유 시 컨텍스트 부재로 의사소통 비효율 발생

### 1.3 기대 효과

| 지표 | AS-IS (현재) | TO-BE (도입 후) | 개선율 |
|------|-------------|----------------|--------|
| 도면 검색 소요 시간 | 30분 ~ 2시간 | 1분 이내 | **~95% 단축** |
| 도면 분류 정확도 | 수작업 의존 | 자동 분류 + 검증 | **일관성 확보** |
| 신규 인력 적응 기간 | 3~6개월 | 즉시 활용 가능 | **대폭 단축** |
| 도면 중복 등록률 | 관리 불가 | 유사 도면 자동 탐지 | **중복 최소화** |

## 2. 솔루션 개요 (Solution Overview)

### 2.1 시스템 컨셉

**DrawingLLM**은 오픈소스 멀티모달 LLM과 벡터 검색을 결합하여, 기술 도면을 자동으로 분석·분류·검색할 수 있는 AI 시스템이다.

```
┌─────────────────────────────────────────────────────────┐
│                    사용자 인터페이스                       │
│  ┌─────────────┐  ┌──────────────┐  ┌────────────────┐  │
│  │ 도면 업로드  │  │ 자연어 검색   │  │ 유사 도면 탐색  │  │
│  └──────┬──────┘  └──────┬───────┘  └───────┬────────┘  │
├─────────┼────────────────┼──────────────────┼───────────┤
│         ▼                ▼                  ▼           │
│  ┌─────────────────────────────────────────────────┐    │
│  │              파이프라인 엔진                      │    │
│  │  ┌────────┐  ┌────────┐  ┌────────┐  ┌───────┐ │    │
│  │  │  OCR   │→ │Embedding│→ │VectorDB│→ │  LLM  │ │    │
│  │  └────────┘  └────────┘  └────────┘  └───────┘ │    │
│  └─────────────────────────────────────────────────┘    │
├─────────────────────────────────────────────────────────┤
│                    데이터 저장소                          │
│  ┌──────────┐  ┌───────────┐  ┌───────────────────┐    │
│  │ 도면 원본 │  │ 벡터 인덱스 │  │ 메타데이터 DB     │    │
│  └──────────┘  └───────────┘  └───────────────────┘    │
└─────────────────────────────────────────────────────────┘
```

### 2.2 핵심 기술 접근법

**멀티모달 RAG + 다중 VLM 파이프라인 아키텍처**

1. **이미지 임베딩** — OpenAI CLIP(ViT-L/14, Fine-tuned)으로 DXF→PNG 도면을 768차원 벡터로 변환. 68,647건 등록 완료
2. **텍스트 임베딩** — PaddleOCR + SentenceTransformer(multilingual-e5-small)로 도면 내 텍스트를 벡터화. 한/영/일 다국어 지원
3. **다중 VLM 역할 분담** — 단일 모델의 속도/정확도/한국어 한계를 극복하기 위해 4개 VLM을 역할별 배치:
   - glm-ocr: 고속 텍스트 추출 (<1s)
   - translategemma: 배치 분류/요약 (2~9s)
   - qwen3.5/qwen3.5: 상세 분석 및 Q&A (60~90s)
4. **하이브리드 벡터 검색** — ChromaDB에 이미지/텍스트 3채널 컬렉션 구성, 가중치 튜닝으로 최적 비율 도출 (image=0.15, text=0.85)
5. **YOLOv8 이미지 분류** — YOLOv8-cls v2 81클래스 학습으로 도면 카테고리 자동 분류 (정확도 93.87%)
6. **YOLOv8 객체 탐지** — YOLOv8-det 학습으로 표제란, 부품표, 치수 영역 자동 탐지 (mAP50=0.552)
7. **LLM 컨텍스트 주입** — YOLO/OCR 추출 데이터를 AnalysisContext로 구조화하여 LLM 프롬프트에 자동 주입, HallucinationDetector로 환각 검증

## 4. 기술 스택 (Tech Stack)

### 4.1 핵심 컴포넌트

```yaml
# LLM / AI — 실제 구축 환경 (2026-03-06 기준)
멀티모달 VLM (Ollama 로컬):
  메인 분석:
    - qwen3.5:9b (6.1GB) — 도면 상세 분석, Q&A, 가장 정확한 설명 생성
    - qwen3.5:9b (6.6GB) — 보조 분석, 한국어 설명 품질 우수
  경량 고속:
    - translategemma (3.3GB, Gemma3 기반) — 배치 분류/요약, 2~9초/task
    - translategemma:12b (8.1GB) — 정밀 분류 (대형 버전)
  OCR 전용:
    - glm-ocr (2.2GB) — 도면 내 텍스트 고속 추출 (<1초)
  사용 불가 확인:
    - glm-4.7-flash: Vision 미지원 (이미지 입력 시 TIMEOUT)
  선택 이유:
    - 단일 VLM의 한계(속도/정확도/한국어 품질) 극복을 위해 역할별 분담
    - 모두 Ollama 로컬 실행, API 비용 없음
    - M4 Pro 24GB에서 3~9GB 모델 교대 로드 가능

이미지 임베딩:
  - OpenAI CLIP (ViT-L/14) — 도면 이미지 → 768차원 벡터
  - 용도: ChromaDB 이미지 컬렉션 (61,473건 등록 완료)

텍스트 임베딩:
  - intfloat/multilingual-e5-small — 384차원 벡터
  - 용도: OCR 텍스트 + 검색 쿼리 → 벡터 변환
  - 한/영/일 다국어 지원, "query: " / "passage: " 접두사 기반

OCR:
  - PaddleOCR 3.4.0 (한/영/일 지원)
  - 부품번호, 재질, 치수 패턴 자동 추출 (정규식 기반)
  - OCR 텍스트 살균 (셸 인젝션/경로 탐색/null byte 제거)

도면 분류:
  - YOLOv8-cls v2 (자체 학습) — 81 카테고리 이미지 분류
  - **정확도 93.87%** (Top-1), Top-5 98.04%
  - SHA256 모델 무결성 검증, 디바이스 자동 선택 (MPS/CUDA/CPU)

객체 탐지:
  - YOLOv8-det (자체 학습) — 표제란, 부품표, 치수 영역 탐지
  - **mAP50=0.552**, IoU 임계값 0.5
  - 탐지 결과를 AnalysisContext에 구조화하여 LLM 주입

LLM 컨텍스트 주입 (Phase 4):
  - AnalysisContext: YOLO-cls/det + OCR 결과를 LLM 프롬프트에 자동 주입
  - HallucinationDetector: LLM 응답 vs OCR/YOLO 사실 4필드 대조 검증
  - 텍스트 전용 모드: 충분한 컨텍스트 시 이미지 인코딩 스킵 (60-90s → <20s)

# 데이터 저장
벡터 데이터베이스:
  - ChromaDB 1.5.1 (68,647건 이미지 + 텍스트 임베딩)
  - DB 크기: 441MB
  - 하이브리드 검색: image_weight=0.15, text_weight=0.85 (CLIP Fine-tuning 후 가중치 조정)

파일 저장소:
  - 로컬 파일시스템 (MiSUMi DXF 61,451 + 베어링 DWG 463)

# 프론트엔드
프로토타입 UI: Streamlit 1.53.0

# 런타임
Python: 3.13.11 (conda ml 환경)
하드웨어: Apple M4 Pro 24GB, MPS GPU 가속
```

## 7. 성능 평가 계획 (Evaluation Plan)

### 7.1 검색 성능 지표

| 지표 | 설명 | 목표 | 현재 측정값 (2026-03-06) | 상태 |
|------|------|------|--------------------------|------|
| Recall@5 | 상위 5건 내 정답 포함 비율 | ≥ 0.80 | 0.372 | ❌ |
| Recall@10 | 상위 10건 내 정답 포함 비율 | ≥ 0.90 | — | ❌ |
| CLIP 카테고리 R@5 (t2i) | 텍스트→이미지 카테고리 매칭 | ≥ 0.80 | **0.947** | ✅ |
| CLIP 카테고리 R@5 (i2t) | 이미지→텍스트 카테고리 매칭 | ≥ 0.60 | **0.732** | ✅ |
| MRR | 첫 정답의 역순위 평균 | ≥ 0.70 | 0.263 | ❌ |
| 검색 응답 시간 | 쿼리 ~ 결과 반환 소요 시간 | ≤ 3초 | 0.104초 | ✅ |

> **검색 성능 분석:** 현재 낮은 검색 성능의 근본 원인은 DXF→PNG 변환 도면에서 OCR로 추출한 텍스트가 **부품명이 아닌 치수/기호**(예: "M5", "φ20", "PSFCG20-82-F8-P8-M5") 위주이기 때문. "shaft를 찾아줘"라는 쿼리와 "M5, 82, F8" 같은 OCR 텍스트 사이에는 의미적 연결이 없음.
>
> **개선 전략:** glm-ocr로 정확한 텍스트 추출 + translategemma/qwen3.5로 도면 설명(부품명, 용도) 텍스트를 생성하여 텍스트 임베딩 품질을 향상시키면 검색 성능 개선 기대.

**가중치 튜닝 결과 (Grid Search, 142쿼리):**

최적 가중치: `image_weight=0.15, text_weight=0.85` (CLIP Fine-tuning 후 이미지 임베딩 활성화)

CLIP Fine-tuning 전에는 text_weight=1.0이 최적이었으나, 도면 데이터로 Fine-tuning 후 이미지 임베딩의 카테고리 수준 검색 기여도가 크게 개선됨 (카테고리 R@5 t2i: 94.7%).

## 8. 포트폴리오 차별화 전략

### 8.1 기술적 차별점

1. **실제 산업 데이터 대규모 처리** — MiSUMi 54,185 + 베어링 383 + 자동차/특허/Kaggle 등, 총 68,647건 벡터 DB 등록. 프로토타입이 아닌 실전 규모
2. **다중 VLM 역할 분담 아키텍처** — 단일 모델 한계를 5개 모델(qwen3.5, qwen3.5, translategemma, glm-ocr) 비교·분석 후 역할별 최적 배치
3. **오픈소스 완전 로컬 구동** — Ollama + ChromaDB, API 비용 없이 M4 Pro 24GB에서 전체 파이프라인 구동
4. **엔드투엔드 파이프라인** — DXF/DWG 원본 → PNG 변환 → CLIP+OCR 임베딩 → ChromaDB 등록 → YOLO 분류 → VLM 분석까지 전 과정 구현
5. **정량적 성능 측정** — Grid Search 가중치 튜닝, 142쿼리 Ground Truth, mAP/Recall/MRR 등 객관적 지표로 평가

### 8.2 발표/면접 시 강조 포인트

- **문제 발견 능력**: 도면 검색의 실제 비효율성(30분~2시간)을 AI로 1분 이내로 단축
- **기술 선택 근거**: VLM 5개 모델 실제 비교 후 역할별 분담 결정, CLIP 이미지 vs 텍스트 가중치 튜닝으로 최적 비율 도출
- **정량적 성과**: YOLOv8-cls v2 정확도 93.87%(81카테고리), CLIP Fine-tuning 카테고리 R@5=94.7%, 검색 응답 0.104초, 68,647건 벡터 DB, 412 테스트
- **문제 해결 과정**: MPS 메모리 누수 → Epoch Runner 고안, ChromaDB 손상 → sqlite3 .recover 복구, VLM 환각 → HallucinationDetector 구현, pip 26 빌드 실패 → setuptools 핀닝
- **보안 의식**: SSRF 방어, 프롬프트 인젝션(18 패턴), 모델 무결성(SHA256), 레이트 리미팅, OCR 살균, 비루트 Docker
- **확장 가능성**: VLM 기반 메타데이터 자동 생성으로 검색 품질 개선, Docker 패키징으로 배포 가능
