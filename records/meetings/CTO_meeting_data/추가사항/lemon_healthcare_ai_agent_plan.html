1. 기획 한 줄 정의

병원성 데이터와 건강 데이터를 이해하는 AI Agent가 만성질환자의 음식, 영양제, 복약, 식단관리 수준을 개인화해주는 건강관리 MVP를 기획한다.

우리는 건강의신과 직접 연동하지 않는다. 대신 건강의신에 적용 가능한 독립형 참조 앱을 만들어, 레몬헬스케어가 향후 실제 서비스에 반영할 수 있는 방향을 제안한다.

2. 우리가 기획하는 것

우리가 기획하는 것은 단순 음식 분석 앱이나 영양제 정보 앱이 아니다.

핵심은 만성질환자의 병원성 데이터, 복약 정보, 음식·영양제 섭취 기록, 건강 데이터를 AI Agent가 함께 이해하고, 사용자에게 맞는 식단관리 판단과 실천 피드백을 제공하는 서비스다.

내 병원 기록과 생활 데이터를 기억하는 만성질환자용 AI 영양·복약 관리 Agent

이 서비스는 사용자가 음식이나 영양제를 촬영하면 AI가 분석하고, 그 결과를 사용자의 질환 정보, 검사값, 복약 정보, 건강 데이터와 함께 해석한다. 이후 Agent는 부족 영양소, 과다 섭취 가능성, 주의가 필요한 성분, 식단 조절 수준과 영양제 관리 상태를 설명한다.

3. 레몬헬스케어 목표 해석

레몬헬스케어의 목표는 개인 헬스 시계열 데이터를 활용한 생애 전 주기 개인화 헬스케어 서비스다.

PPT에서 제시된 목표는 음식, 식단, 활동, 병원 진료 정보, 실시간 바이탈, 유전자 등 다양한 개인 데이터를 시계열로 쌓고, 이를 기반으로 개인화된 건강관리 서비스를 제공하는 것이다.

우리 프로젝트는 이 목표 전체를 2개월 안에 구현하는 것이 아니다. 대신 음식·영양제 분석과 만성질환자 개인화 관리를 중심으로, 레몬헬스케어가 목표로 하는 개인화 헬스케어 방향을 작게 실증하는 MVP다.

4. 과제 해석

레몬헬스케어가 제시한 과제는 크게 영양제 분석과 음식·식단 분석으로 나뉜다.

과제 #1. 영양제

복용 중인 영양제를 기반으로 개인별 섭취 영양소, 권장 섭취 영양소 기준, 부족 영양소 추천, 영양소 섭취량 권고, 목적별 특화 분석을 제공한다.

과제 #2. 음식

음식과 식단을 기반으로 섭취 영양소를 분석하고, 활동 정보와 결합해 체중 변화 예측, 활동 권고, 부족 영양소 추천으로 확장한다.

우리는 이를 단순 AI 모델 개발 과제로 보지 않는다. 이 과제는 레몬헬스케어의 병원 데이터 기반 개인화 헬스케어 목표에 도달하기 위한 Agent형 서비스 실증 과제로 해석한다.

따라서 음식과 영양제 분석은 각각 독립된 출발점이고, 최종 방향은 두 섭취 데이터를 병원성 데이터와 생활 데이터에 연결한 만성질환자 관리 서비스다.

5. 문제 정의

만성질환자는 병원 검사, 복약, 식단, 영양제, 활동 데이터를 따로 관리한다.

  • 내가 먹는 음식과 영양제가 내 질환 상태에 맞는지 알기 어렵다.
  • 복용 중인 약과 영양제를 함께 고려하기 어렵다.
  • 병원 검사 수치와 일상 생활 데이터를 연결해서 관리하기 어렵다.
  • 건강관리 앱은 많지만 병원 기록을 이해하는 개인화 서비스는 부족하다.
  • 분석 결과가 실제 식단관리 피드백, 알림, 꾸준한 기록 동기로 이어지지 않는다.

7. MVP 핵심 흐름

사용자가 기본 프로필과 만성질환 정보를 확인한다.
Kaggle 기반 검사값, 질환, 복약 데이터가 사용자 상태로 구성된다.
사용자가 음식 또는 영양제를 촬영한다.
AI가 음식명, 섭취량, 영양소, 영양제 성분을 분석한다.
사용자가 인식 결과를 수정한다.
Agent가 병원성 데이터, 복약 정보, 건강 데이터를 함께 참고한다.
Agent가 부족 영양소, 과다 가능성, 주의 가능성을 판단한다.
챗봇이 결과를 쉬운 말로 설명한다.
사용자가 자연어로 알림이나 진료 일정 등록을 요청한다.
Agent가 날짜, 시간, 반복 조건을 확인한 뒤 복약 알림, 식단 기록 알림, 진료 일정 캘린더 등록을 실행한다.
음식·영양제 사진 기록 참여 상태에 따라 응모권이 자동으로 누적된다. 식단관리 점수는 응모권 지급 조건이 아니라 개선 피드백에 사용한다.

영양제 분석 보완 흐름

사용자가 복용 중인 영양제 제품명, 라벨, 성분표를 촬영한다.
AI가 제품명, 성분명, 함량, 1회 섭취량, 권장 섭취 횟수를 OCR로 분석한다.
사용자가 실제 섭취량, 섭취 빈도, 복용 시간을 수정·확인한다.
Agent가 영양제 성분을 영양소 단위로 환산하고 음식 섭취 영양소와 합산한다.
Agent가 개인별 권장 기준, 질환 정보, 검사값, 복약 정보를 함께 참고한다.
Agent가 부족 영양소, 과다 섭취 가능성, 중복 성분, 주의가 필요한 성분을 설명한다.
Agent가 눈건강, 간기능, 피로회복 같은 목적별 관리 방향을 제시하되, 특정 제품 구매 추천은 하지 않는다.

8. Agent 범위

2개월 MVP에서는 Agent 범위를 4개로 제한한다.

8.1 음식·영양제 분석 Agent

음식 사진과 영양제 사진을 분석하고, 사용자의 수정 입력을 반영한다. 이 Agent의 책임은 이미지/OCR 인식과 영양소 산출까지이며, 질환 기준 해석과 점수화는 식단관리 평가 Agent가 담당한다.

음식 분석 역할
  • 음식명 추정
  • 섭취량 입력 보정
  • 영양소 계산
  • 식단관리 평가에 필요한 영양소 데이터 산출
영양제 분석 역할
  • 영양제 제품명/OCR 분석
  • 성분표 분석
  • 1회 섭취량과 섭취 빈도 기반 1일 섭취량 환산
  • 음식 섭취 영양소와 영양제 섭취 영양소 합산
  • 사용자 수정 입력 반영

8.2 병원데이터 개인화 Agent

질환 태그, 검사값, 복약 정보를 참고해 분석 결과를 개인화할 기준을 제공한다.

  • 만성질환 정보 확인
  • 검사 수치 참고
  • 복약 정보 참고
  • 개인별 관리 기준 생성
  • 식단관리 평가 Agent가 사용할 질환별 주의 기준 제공

8.3 챗봇 설명·실행 Agent

사용자와 대화하는 입구 역할을 한다. 의료 판단자가 아니라, 분석 결과를 설명하고 사용자의 명시적 요청을 알림·캘린더 같은 실행 기능으로 연결한다.

  • 분석 결과를 쉬운 말로 설명
  • 부족·과다·주의 이유 설명
  • 식단 기록 실행
  • 사용자 요청 기반 복약 알림 등록
  • 사용자 요청 기반 식단 기록 알림 등록
  • 사용자 요청 기반 진료 일정 캘린더 등록

8.4 식단관리 평가 Agent

음식·영양제 사진 촬영 자체는 출석처럼 채워지는 기록 기능으로 처리한다. 이 Agent는 촬영 여부를 판단하거나 이벤트를 운영하지 않고, 분석 Agent가 산출한 영양 데이터를 질환 기준으로 해석해 사용자가 식단을 얼마나 잘 조절했는지와 영양제를 얼마나 잘 관리했는지 피드백한다.

식단관리 점수는 응모권 지급 조건이 아니라 사용자 피드백 지표로 사용한다. 점수가 낮은 날에도 사용자가 음식·영양제 사진 기록을 완료했다면 응모권은 받을 수 있고, Agent는 낮은 점수의 원인과 다음 식사에서 개선할 방향을 설명한다.

  • 질환별 주의 영양소 기준과 식단 기록 비교
  • 끼니별/하루별 식단관리 점수 산정
  • 부족·과다·주의 성분 설명
  • 좋은 선택과 개선이 필요한 음식 설명
  • 영양제 섭취 빈도와 중복·과다 가능성 평가
  • 눈건강, 간기능, 피로회복 등 목적별 영양제 관리 방향 설명
  • 점수가 낮은 끼니의 원인과 개선 방향 설명

9. 건강의신 적용 가능 방향과 제외 범위

이 문서는 건강의신의 기존 이벤트나 미션 구조를 복제하는 기획서가 아니다. 음식·영양제 사진 기록, 식단관리 피드백, 응모권 참여 구조가 건강의신에 적용될 수 있는 독립형 참조 모델을 제안한다.

가져올 것
  • 건강 데이터 기반 관리
  • 사용자가 매일 음식·영양제 사진 기록을 채우는 참여 UX
  • 꾸준한 기록을 유도하는 응모권 참여 구조
  • 만성질환자도 쉽게 사용할 수 있는 UX 방향
제외할 것
  • 건강의신 기존 이벤트
  • 건강의신 기존 미션 구조
  • 건강의신 실제 API 연동
  • 보험청구 기능
  • 포인트 정책
  • 기존 앱 전체 기능 복제

응모권 이벤트 구조

  • 하루 음식·영양제 사진 기록 완료 시 응모권 1개를 지급한다.
  • 1주일 연속 기록 완료 시 추가 응모권 1개를 지급한다.
  • 1개월 전체 기록 완료 시 추가 응모권 3개를 지급한다.
  • 예시 보상은 1등 1명 2박 3일 숙박권, 2등 3명 국내 호텔 숙박권, 3등 10명 건강식품 5만원권으로 둔다.
  • 응모권은 Agent가 생성하거나 점수로 제한하는 과제가 아니라 사진 기록 참여 여부를 바탕으로 자동 누적되는 참여 장치다.
  • 식단관리 점수는 응모권 지급 조건이 아니라 사용자가 자신의 식단을 이해하고 개선하기 위한 피드백 지표로 사용한다.

10. 데이터 활용 방향

실제 병원 연동은 불가능하므로 MVP에서는 Kaggle 데이터와 사용자 입력을 조합한다.

사용 데이터
  • 만성질환/검사값 데이터
  • 복약 데이터
  • 식단/영양 데이터
  • 사용자 기본 정보
  • 음식 이미지
  • 영양제 이미지
  • HealthKit/Health Connect 기반 걸음수, 체중, 심박 데이터
  • 혈압은 가능하면 연동하고, 불가능하면 수동 입력으로 보완한다.
  • 영양제 제품명, 라벨 이미지, 성분표, 1회 섭취량, 섭취 빈도, 복용 시간, 목적 태그

Kaggle 데이터는 실제 병원 데이터가 아니다. 다만 레몬헬스케어가 병원 연동 데이터를 활용할 수 있다는 강점을 고려해, 실제 서비스 적용 시 병원 데이터가 들어갈 자리를 시연용으로 대체하는 역할을 한다.

영양제 데이터는 음식 섭취 영양소와 분리해서 보관하되, 개인별 판단 시에는 두 데이터를 합산해 총 섭취량을 계산한다. 목적 태그는 눈건강, 간기능, 피로회복처럼 사용자가 영양제를 먹는 이유를 설명하는 보조 정보로 사용한다.

11. Agent 요약 기억

Agent는 전체 원본 데이터를 모두 저장하는 방식이 아니라, 개인화에 필요한 핵심 정보만 요약 기억한다.

  • 최근 검사값 요약
  • 만성질환 태그
  • 복약 정보 요약
  • 음식/영양제 기록 요약
  • 건강 데이터 요약
  • 식단관리 점수
  • 영양제 섭취 달성 이력
  • 응모권 누적 이력
  • 최근 주의사항

이 구조를 통해 사용자는 매번 처음부터 설명하지 않아도 되고, Agent는 사용자의 상태 변화를 반영해 더 개인화된 관리 판단을 제공할 수 있다.

12. 건강 데이터 연동

우선 연동 대상
  • 걸음수
  • 체중
  • 심박
가능하면 연동할 대상
  • 혈압

iOS는 HealthKit, Android는 Health Connect 연동을 고려한다. 혈압은 모든 기기에서 자동 수집되는 데이터가 아니므로, 연동이 불가능한 경우 수동 입력으로 보완한다.

건강 데이터는 진단 목적이 아니라 Agent의 식단관리 평가와 생활 상태 해석을 보조하는 참고 데이터로 사용한다.

13. 챗봇 기능 범위

챗봇은 설명형과 실행형을 함께 제공한다.

포함 기능
  • 분석 결과 설명
  • 부족/과다/주의 이유 설명
  • 식단 기록
  • 사용자 요청 기반 복약 알림 등록
  • 사용자 요청 기반 식단 기록 알림 등록
  • 사용자 요청 기반 진료 일정 캘린더 등록
제외 기능
  • 의료 진단
  • 처방
  • 복용 가능 여부 단정
  • 약물-영양제 상호작용 확정 판단
  • 응급 판단
  • 자유 의료 상담

사용 예시

  • "매일 아침 8시에 혈압약 알림 맞춰줘"라고 요청하면 복약 알림을 등록한다.
  • "다음 진료가 6월 3일 오후 2시야. 캘린더에 넣어줘"라고 요청하면 진료 일정을 등록한다.
  • "이 영양제 계속 먹어도 돼?"라고 질문하면 복용 가능 여부를 단정하지 않고, 성분 중복·과다 가능성과 전문가 상담 필요성을 설명한다.
  • 챗봇 대화 중 정기 진료나 전문가 상담을 안내하더라도, 실제 일정 등록은 사용자가 요청하거나 확인했을 때만 실행한다.

14. 법적·의료적 안전선

이 서비스는 진단·치료 서비스가 아니라 웰니스 기반 건강관리 보조 서비스로 정의한다.

표현 기준

  • "복용해도 됩니다"라고 단정하지 않는다.
  • "주의가 필요할 수 있습니다"라고 표현한다.
  • "전문가 상담을 권장합니다"라고 안내한다.
  • 질병 악화나 치료 효과를 단정하지 않는다.
  • AI 판단은 참고용이며 최종 판단은 의료진과 상의하도록 한다.
  • 복약 알림 등록은 사용자의 기존 복약 정보를 기반으로 한 일정 관리 기능이며, 복용 가능 여부 판단이나 처방 변경 기능이 아니다.
  • 진료 일정 캘린더 등록은 일정 관리 기능이며, 의료 진단이나 치료 결정을 대신하지 않는다.
  • 영양제 분석은 성분 중복, 과다 가능성, 부족 가능성을 안내하는 관리 보조 기능이며, 특정 제품 구매 추천이나 복용 가능 여부 확정 판단을 제공하지 않는다.

17. 최종 메시지

우리 프로젝트는 음식과 영양제 분석을 출발점으로 한다.

하지만 최종적으로는 만성질환자의 병원성 데이터, 복약 정보, 식단, 건강 데이터를 연결해 개인화된 식단관리 판단과 영양제 관리 피드백을 제공하는 AI Agent 서비스를 제안한다.

이는 레몬헬스케어가 목표로 하는 생애 전 주기 개인화 헬스케어로 가기 위한 작은 MVP이자, 건강의신에 적용 가능한 서비스 참조 모델이다.
